Pherix · Slice 7.5 decisions

Closing the two open calls from Slice 7.

Slice 7 shipped dry-run as "agent_txn that rolls back at the end instead of committing" — and explicitly deferred two questions because answering them would have broken the "nearly free" framing. Slice 7.5 is those two questions. They are independent but ship together because each one is a narrow follow-up — combined, they still sit comfortably inside one slice. Five decisions below, one open call, both Slice-7 deferrals retired.

The premise — two open calls, both narrow

Slice 7's decision aid was honest: "per-adapter state diff is where 'nearly free' stops being honest", and "concurrency-aware dry-run is real complexity that deserves its own slice." Both got deferred. Slice 7.5 is the moment to take them seriously — but they remain narrow follow-ups, not new architectural ground.

Operators asking real questions of dry-run today: "Which rows would have been inserted? Which files would have been written? Is another agent's in-flight transaction going to invalidate this preview by the time I act on it?" Slice 7's DryRunResult already answers "what tool calls would have happened" and "what would policy have said." 7.5 answers the structural state question and the concurrency question.

Strand A — state diff

"Which rows? Which files?"

Each adapter that can compute a meaningful before/after diff opts in. SQL queries the savepoint plus the live state and returns row-level deltas. Filesystem walks the copy-on-write tree and returns added/modified/deleted files. HTTP opts out — its diff is the would_have_fired list Slice 7 already exposes.

Strand B — concurrency awareness

"Would this conflict with anyone in flight?"

Dry-run registers with JournalRegistry as a soft claim — visible to other peers' isolation diffs but never blocking a real transaction's commit. At end of the dry-run, run check_conflicts against the live registry and surface results in result.isolation_conflicts.

D1 — One slice covering both, or two sub-slices?

The question

"The two open calls are independent — state diff doesn't touch isolation, isolation-awareness doesn't touch adapter protocols. Do we ship them as one Slice 7.5, or as 7.5a (state diff) + 7.5b (concurrency)?"

Option A — split: 7.5a then 7.5b

Two decision aids, two worktrees, two merges, two doc sweeps.

Cleaner intellectual scope per worker; smaller diffs to review.

Doubles the slice-ceremony overhead (aid + TASK + review + sweep) for two work-streams that each fit comfortably in 2–3 SP.

Two follow-up doc sweeps to keep the suite consistent. The "split for cleanness" instinct competes with the operator's "I might have over-vibed" signal — more slice transitions, not fewer, is the wrong direction.

Option B — one Slice 7.5 covering both

One decision aid (this one), one worktree, one PR, one sweep.

Strand A and Strand B are independent at the code level — they touch different files (adapters/*.py + new dry_run.state_diff field versus isolation.py soft-claim + dry_run.py registry hook). The worker can build them in either order or in parallel within the same worktree. The PR reviewer sees two well-scoped diff regions rather than two PRs.

Lower ceremony overhead — appropriate for a 5–6 SP follow-up.

One sweep, one merge commit, one retirement of this decision aid.

Worker has to keep two distinct work streams straight in one PR. Mitigation: TASK.md scopes them as two strands explicitly.

Locked — Option B

Ship as one Slice 7.5. The decision aid covers both strands separately (D2/D3 for state diff, D4/D5 for concurrency); the worker builds them in one PR; one post-merge sweep retires both deferrals.

D2 — State diff: where does the diff() method live?

The question

"To compute 'which rows would have been inserted,' an adapter has to know how. SQL knows; FS knows; HTTP fundamentally can't. Where does the method declaring 'I can do this' live in the protocol hierarchy?"

Option A — mandatory on ResourceAdapter base

Every adapter must implement diff(snapshot, current) → dict.

HTTPAdapter has no structural state. Forcing it to implement a method that returns "empty diff, see would_have_fired" makes the protocol lie about what every adapter actually does.

Future adapters that wrap fundamentally non-snapshotable resources (webhooks, message queues) would have the same forced-empty problem.

Option B — new StateDiffable sub-protocol

Adapters opt in by implementing StateDiffable, mirroring the TransactionalResourceAdapter pattern Slice 2 established.

class StateDiffable(Protocol):
    def diff(self, snapshot: SnapshotHandle, current: Any) -> dict:
        """Return adapter-specific before/after state diff."""

# SQLiteAdapter and FilesystemAdapter implement StateDiffable.
# HTTPAdapter does NOT — its diff is the would_have_fired list.

Runtime dispatch via isinstance(adapter, StateDiffable) at the dry-run finalise hook. Adapters that opt in get their diff into result.state_diff; adapters that don't are silent. Matches Slice 2's discipline exactly.

Honest about which adapters can structurally diff.

Reuses an established pattern (Slice 2's sub-protocol approach for transactional adapters).

Extensible — a future adapter that can diff implements the protocol; one that can't, doesn't.

Locked — Option B

New StateDiffable sub-protocol in adapters/base.py; SQLiteAdapter and FilesystemAdapter implement it; HTTPAdapter deliberately does not. Runtime dispatches via isinstance at dry-run finalise time.

D3 — State diff: what shape does the payload have?

The question

"SQL would naturally return rows; filesystem would naturally return file paths. Do we force them into a common shape, or let each adapter return its own dict?"

Option A — fully standardised StateDiff dataclass

One canonical shape: {added: [...], modified: [...], deleted: [...]} with stringified entries.

Operator-facing API is one shape — they learn once.

Stringifying SQL rows and FS paths into a common "entry" type loses structure the operator actually wants. A SQL row inserted with columns (id=42, name='Alice', tier='basic') becomes "users:42" and the operator has to query the journal again to recover the args.

Forces adapter authors to invent a canonical form for any future resource type. Higher friction on each new adapter.

Option B — standardised wrapper + adapter-specific payload

A StateDiff namespace containing per-resource entries; each entry's payload is adapter-specific.

result.state_diff = {
    "sql": {
        "rows_added": [{"table": "users", "row": {"id": 42, "name": "Alice"}}, ...],
        "rows_modified": [...],
        "rows_deleted": [...],
    },
    "fs": {
        "files_added": ["new.txt", ...],
        "files_modified": ["existing.txt", ...],
        "files_deleted": [...],
    },
}

The outer shape is standardised (a dict keyed by resource name, mirroring adapters); the inner shape is whatever the adapter knows is useful. Operators write result.state_diff["sql"]["rows_added"] against the adapter they care about.

Preserves the structure the operator actually wants.

Each adapter author defines the shape that's natural for their resource — no Procrustean common form.

Composes with would_have_fired (HTTP's "diff") — the operator already knows which result field carries which resource type.

Operator learns a new shape per adapter type. Mitigation: the slice ships docstrings on each diff() method describing the shape.

Option C — flat list of standardised "change" records

[Change(resource, key, kind, before, after), ...]

Single-axis iteration; easy to count or filter.

Loses the per-resource grouping operators naturally want ("show me all the SQL changes"). Mitigation by group-by is fine but adds a step.

"key" and "before"/"after" are vague for some resources — a file's "before" might be a content hash, a row's "before" a dict, an HTTP call's "before" undefined.

Locked — Option B

Standardised outer shape (per-resource dict keyed by resource name); adapter-specific inner shape (rows for SQL, paths for FS). Each adapter's diff() docstring documents its return shape. result.state_diff is a dict[str, dict].

D4 — Concurrency: how does dry-run register with JournalRegistry?

The question

"Slice 4 ships JournalRegistry as the in-process arbitration substrate. A real txn registers, commits or rolls back, deregisters. A dry-run never commits. What should the registry see?"

Option A — dry-run doesn't register at all

Status quo (Slice 7). dry_run.py currently calls ISOLATION_REGISTRY.register(ctx) as bookkeeping; this option removes that.

Simplest possible model — dry-runs are invisible to the rest of the system.

Dry-run can't answer "would this txn conflict with anyone in flight?" — the operator has no way to ask. Defeats Strand B entirely.

Option B — soft claim

Dry-run registers with a soft_claim=True flag. Visible to check_conflicts; invisible to Serialize waiters.

# Slice 4's JournalRegistry gains a soft-claim distinction:
ISOLATION_REGISTRY.register(ctx, soft_claim=True)

# Real txns running check_conflicts see the dry-run's writes:
#   "txn-abc would have written key K — your txn read K@v=5 — POTENTIAL conflict"
# Serialize waiters do NOT block on the dry-run:
#   wait_for_blockers() skips soft-claim entries

The two desirable properties hold simultaneously: a dry-run sees the live world (so check_conflicts at finalise sees real in-flight peers) and a real txn doesn't hang on a dry-run (because the dry-run never commits, so blocking on it is a deadlock waiting to happen).

Symmetric: dry-runs and real txns see each other for conflict purposes; only real txns gate each other for serialization purposes.

Registry contract widens by one boolean — small mechanical change.

Slice 7's existing ISOLATION_REGISTRY.register(ctx) call upgrades cleanly to register(ctx, soft_claim=True).

Option C — hard claim

Dry-run registers exactly like a real txn. Serialize waiters block on it.

A Serialize waiter blocks on a dry-run that will never commit — definitionally a deadlock-shaped pattern. Mitigation by timeout doesn't fix the semantics.

Locked — Option B

Soft-claim model. JournalRegistry.register gains a keyword-only soft_claim: bool = False parameter; dry_run.py passes True; wait_for_blockers skips soft-claim entries; check_conflicts includes them when diffing read-keys against live writes.

D5 — Conflict detection: when does it fire?

The question

"During a dry-run, when does the operator find out 'your txn would have conflicted'? Continuously as the body runs, or once at the end?"

Option A — continuous, mid-body

After each effect, run a partial check_conflicts; surface conflicts in real time.

Earliest possible signal.

Doesn't match how a real txn discovers conflicts — they fire at commit-time, not mid-body. Asymmetric semantics between dry-run and the txn it's simulating.

Per-effect check_conflicts is genuinely more expensive — O(n) conflict diffs per dry-run instead of one.

Option B — once at finalise

Run check_conflicts(txn.effects, adapters) once at the dry-run finalise hook, before rollback. Surface in result.isolation_conflicts.

Same shape as a real txn's commit-time conflict check (Slice 4). The dry-run reports "if you'd committed this txn right now, you'd have hit these conflicts". result.isolation_conflicts is a list[Conflict] — empty under a clean speculative run, populated when a live peer would have moved a key.

Symmetric semantics with real txns — the dry-run answers the question "would commit have succeeded?" using the same predicate check_conflicts.

Single conflict diff per dry-run; cheap.

Slots into the existing _dry_run_finalise hook before rollback().

Locked — Option B

One check_conflicts call at the dry-run finalise hook, before rollback. Results land in DryRunResult.isolation_conflicts as a list[Conflict]. result.is_clean on DryRunResult widens to AND in not isolation_conflicts alongside the existing all(v.allow for v in policy_verdicts).

Open — cross-process arbitration

The question

"Slice 4's filesystem-shared SQLite meta-connection arbitrates real txns across processes. Does the soft-claim registry need the same machinery for dry-runs?"

The honest tradeoff:

  • If yes: Slice 7.5 also widens the meta-connection's schema to track soft-claims, with the same multi-process bookkeeping Slice 4 ships. Real complexity — bumps the slice from 5–6 SP toward 8–9.
  • If no: Dry-run is in-process only. A second process running a real txn never sees the dry-run's soft-claim, so the dry-run won't detect cross-process conflicts. Acceptable for the v1 use case (one operator running a dry-run for their own decision-making), bad if dry-runs are themselves an agent-population primitive.

Recommendation — defer, document the boundary clearly

Slice 7.5 ships in-process soft-claim only. The StateDiffable + soft-claim registry contracts are designed so a future Slice 7.6 can extend them to cross-process without surgery — same pattern Slice 4 used (in-process first, filesystem-shared meta-connection second). Until 7.6 lands, dry_run's docstring explicitly notes the single-process boundary.

Closing — both Slice 7 deferrals retire here

Slice 7 was honest about what it couldn't deliver in one slice. Slice 7.5 is that honesty paid back. The StateDiffable sub-protocol gives dry-run the structural answer to "what would have happened to my data"; the soft-claim registry gives it the cross-agent answer to "would my commit have raced anyone." Both retire as deferrals on the Slice 7 detail card; Slice 8 inherits a complete dry-run capability.

The slice composes more than it builds. Strand A adds one sub-protocol and two adapter methods. Strand B adds one boolean to a registry method and one call site in _dry_run_finalise. The DryRunResult dataclass grows two fields (state_diff + isolation_conflicts); is_clean widens its conjunction. The Slice 1–7 engine stays exactly as it is.

The journal is the asset; every slice adds one more way of reading it. 7.5 reads it for structural state diff and for cross-agent conflict — both folds the journal already supported in principle, finally surfaced as operator-facing capabilities.
