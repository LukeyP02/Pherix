How it works

One engine, one idea.

Pherix turns every side-effecting tool call into an entry in a versioned journal, mediated by per-resource adapters. Commit, rollback, audit — they're all just traversals of that one journal. Here's the whole mental model on a page.

The journal

Every side-effecting tool call becomes an effect: an entry in an append-only, ordered, versioned log held by a transaction. Commit folds the journal forward, applying each effect; rollback folds it backward, restoring or compensating each one. Nothing extra is needed for audit — the journal is the audit log. You just read it.

The two lanes

An effect goes down one of two lanes, decided by whether its resource can be snapshotted. These are two different undo mechanisms, not one.

Reversible → snapshot / restore

Runs live. The adapter snapshots the resource before, and on rollback restores it — a DB SAVEPOINT, filesystem copy-on-write. The backend does the undo. This is state rollback: exact, and needs no knowledge of what the action meant.

Irreversible → stage, then gate

Cannot be snapshotted, so it's staged and only fires at commit. With a registered compensator it auto-commits and the compensator runs on rollback; without one it gates for explicit approval. A compensator is a semantic inverse: approximate, and needs to know what the action meant.

Adapters

An adapter is a triple — snapshot, apply, restore — over one class of resource. Tools declare which resource they touch; the runtime routes each effect to the right adapter. Backend drivers are optional and lazily imported, so the kernel itself is dependency-free.

Honest about what it can't undo: HTTP cannot be snapshotted, so the runtime forces it down the staged/gated lane rather than pretending it can be rolled back.

Policy

Allow/deny rules plus capability caps, enforced at commit — and again at stage-time, so a forbidden effect fails fast. Policy is a clean slate: the buyer writes their own rules against it. Because state can change between staging and commit, the same rule is evaluated twice, which closes the time-of-check-to-time-of-use gap.
