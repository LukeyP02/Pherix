What Pherix is, and why it holds.

Two authoritative artifacts: the system and the position. Everything we build is judged against the north star; everything we sell is in the position.

Where we are

The engine is built and the value is crystallised. Pherix is one idea — make being wrong cheap — factored into four orthogonal axes over a journal: interception, policy, adapters, compensators. Everything else (audit, isolation, replay, control plane, governed memory) is a consequence of those four, not a fifth thing. Phase 1, the engine, is done — reversible + irreversible lanes, MVCC isolation, replay, policy, dry-run, the MCP gateway, plus the engine-hardening pass that made the moat claims true: crash-consistent recovery, cross-process isolation, world-state policy, the longitudinal envelope. 433 tests green. Now: flesh each axis to its base — the common adapters + the compensator catalog — in two languages (Python done, TypeScript in progress); then the demos; then the first design partner. The architecture is a convergently generalising core: each axis we grow strengthens the shared substrate rather than bloating it.
