Roadmap

Where Pherix is going.

The engine is built. The work now is making it visible and adoptable — without compromising the data-untouched moat that keeps Pherix out of customer infrastructure.

Phase 1 — Engine

Done

The runtime substrate. Every capability — commit, rollback, replay, dry-run, isolation, audit — is one traversal of one journal. Both languages, fully offline tests.

  • Two lanes — reversible (snapshot/restore via the backend) and irreversible (stage + gate, or stage + compensate).
  • MVCC-style isolation between concurrent agents, with read/write-key conflict detection.
  • Deterministic replay — fold the journal forward against fresh state, assert identical results.
  • Policy — predicate evaluated at stage-time and commit-time (TOCTOU-safe).
  • Dry-run / speculation — fold forward against a snapshot, then discard.
  • MCP gateway — any-language agents hit the same engine, no per-language SDK required.
  • Crash-consistent recovery driven from the durable journal (core/recovery.py).
  • World-state policy via PolicyContext.read — predicates that read live state.
  • Longitudinal envelope (core/envelope.py) — cross-process / single-host isolation.
  • 16 Python adapters + 14 TypeScript at semantic + field-name parity.
  • Vetted compensator catalog at module parity in both languages: {identity, payments, provisioning, saas}.
  • 936 Python + 210 TypeScript tests, fully offline. Parity proven by tests/test_sdk_parity.py: 13 scenarios producing structurally identical journals in both engines.
Open follow-ups — surfaced by the parity suite, deliberately scoped out of the engine pass:
  • sql.py execute_isolated — returns a live sqlite3.Cursor; materialise via .fetchall() for readers (latent Python correctness bug).
  • TS SqliteAdapter.readsCommittedOnly() hardcoded false; on-disk cross-process isolation parity awaits the TS meta-connection path.

Phase 2 — Make it visible

Active

Build to the base — the things any buyer assumes work. Let the design partner bring the edge cases.

The laptop demo — six baseline stories

A single 5-minute script the buyer runs on their own machine. Each story lands a different buyer; together they prove this is a runtime, not a rollback library.

  • Rollback on failure — adapter restore. "Our agent broke prod last week."
  • Isolation under concurrency — N agents racing on the same DB, conflict caught. "50 agents step on each other."
  • Policy / capability enforcement — forbidden call blocked at stage time, before it happens. Compliance / security / platform teams.
  • Irreversible gating — money / email / external API paused for human approval. "More autonomy, last-mile trust."
  • Dry-run / speculation — show what the agent would do, without doing it. Reviewer / planning UIs.
  • Replay + audit — re-run yesterday's run on fresh state, confirm same result. Compliance / debugging / regression.

Five additional engine stories

Capabilities the engine already has, but no buyer-facing demo yet. Each is a real differentiator vs the obvious competitors.

  • Cross-language parity — Python service + TypeScript edge worker on one journal. Most competitors are single-language.
  • MCP gateway / any-language agents — Go / Rust / Java agents get the same guarantees without an SDK.
  • Crash recovery — kill the process mid-transaction, restart, fold the durable journal forward, end consistent. Most "transactional" libraries cheat here.
  • World-state policy — predicate evaluated against live external state at commit time. TOCTOU-safe by construction.
  • Journal-as-truth vs LLM transcript — the LLM claims it called X but the journal disagrees. The journal is ground truth. Security / audit angle.

Distribution levers

How the Saturday-afternoon user finds Pherix. They're the funnel; the enterprise sale follows them home.

  • One ecosystem integration — pick one to start: LangGraph / OpenAI Agents SDK / Anthropic Agent SDK / Mastra.
  • Migration guide — "naive tool calling → Pherix in 5 lines". The lowest-friction adoption path.

Operational proof points

The questions every buyer asks before deployment. Own the numbers before they're asked.

  • Latency / throughput overhead benchmark per tool call. Ship the number with the laptop demo.
  • Compensator catalog growth — Stripe, SendGrid, Twilio, AWS side-effects, … "We did the unsexy work; you don't have to."
  • Multi-tenant isolation for SaaS agent companies — one runtime, N tenants, no cross-leak.

Phase 3 — Paid hosted control plane

Pending · design-partner-pulled

OSS does single-host. The paid tier is the things that inherently need centralised infrastructure — you can't self-host them usefully even if you had the code, because the value is the aggregation. Not built ahead of demand; pulled in by the first design partner who needs them.

  • Cross-host arbiter — multi-region, multi-agent coordination at org scale.
  • Audit search across many agents — "what did agent #47 do to our database in Q3?"
  • Anonymised policy library — aggregating across tenants, so each customer benefits from every other customer's threat-discovery without ever seeing their data.
Constraint — metadata only. The control plane sees txn ids, effect shapes, policy verdicts, conflict witnesses. It never sees row values, file contents, or HTTP payloads. The data-untouched moat is preserved across both tiers — the boundary between OSS and paid is drawn on infrastructure, not on features.

Phase 4+ — Future-bets

Future

Signposted on the trajectory, not committed. Each one extends the substrate without forcing engine surgery — the convergent-generalisation test.

  • Agent memory as an adapter — governed and auditable memory mutations. Sleeper differentiator; the spec is already implied by the four axes.
  • Long-running / checkpointing for hours-long autonomous agents that want to commit periodically rather than one giant transaction.
  • Hosted control plane preview UI — even just a mocked screen of the multi-agent view, planting the upgrade seed before the plane ships.

Why this shape

The roadmap reflects four strategic choices that compound. They are the context that makes the phase ordering above make sense.

  • AI-native infrastructure, not an AI-native application. Pherix only exists because of the agent moment, but it sells shovels to the miners. The vertical-agent companies (Sierra, Decagon, Devin, Harvey, …) are customers, not competitors. The bigger that wave gets, the bigger Pherix's market.
  • Open-core, not freemium. OSS gets every capability that runs on a single host. Paid is hosted infrastructure that genuinely cannot be self-hosted usefully. The line is drawn on infrastructure, not features. Never gate a single-host capability behind paid.
  • Small teams are the funnel, not a leak. ~99% of users will never pay a cent. The ~1% that does funds the whole thing because they genuinely need the org-scale piece. The Saturday-afternoon user on a personal project is the same engineer who champions the enterprise sale 12 months later — same person, longer memory than you'd think.
  • Data-untouched moat. The laptop demo runs on the buyer's machine; the hosted plane sees only metadata. Hosting cost tracks revenue by construction — zero customers means zero infra to run; first paying customer funds the first piece of the control plane.
