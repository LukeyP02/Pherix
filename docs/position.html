Pherix · position & direction

Autonomy inside an enforced, reversible envelope.

A statement of what Pherix is, who it is for, and why it holds — written to be argued with, not admired. The engine is built; this fixes the direction.

Pherix lets you deploy an autonomous agent fast, because its actions are continuously bounded and reversible — so being wrong is cheap, and you no longer need a governance meeting to say yes.

We replace the governance meeting with a governance policy. Trust becomes mechanical, not procedural.

The stance: autonomy-first, not human-first

The whole promise of agents is collapsing weeks of work into hours. A governance design that puts a human in the loop on every action gives that speed straight back. So we reject it as the mechanism.

Governance is not a checkpoint the agent stops at. It is an invariant preserved along the whole trajectory — a bound that holds at every step, automatically, so the agent runs flat-out inside it and nobody watches.

The gate is the exception, not the product. If a demo leads with "a human approves the dangerous call," it is selling the ugly 10%. Pherix leads with the envelope; the gate is the footnote.

What we provide — see, stop, undo

A side-effecting tool call goes from a one-way door to a door you can walk back through.

Before Pherix

Effects fire instantly, irreversibly, unlogged, unbounded. A mistake is discovered after the fact and cleaned up by hand — if it can be. Deploying the agent is an unbounded bet.

After Pherix

Effects run inside a transaction: reversible ones journalled and undoable, irreversible ones staged and bounded by policy, all of it audited. A mistake is one rollback away. Deploying is a bet you can retract.

Every other governance tool lowers the cost of detection (it tells you, after, that you were wrong). Pherix lowers the cost of being wrong. Only the second one actually unblocks deployment.

How it works — one idea

A versioned effect journal, mediated by resource adapters. Every tool call becomes an entry in an append-only, ordered log. Every capability is a traversal of that log:

  • Commit — fold forward, apply each effect.
  • Rollback — fold backward, restore each snapshot (or run its compensator).
  • Govern — evaluate policy over the journal, at stage-time and again at commit-time (the world moved between them).
  • Audit — the journal is the record; just read it.

Adapters use real backend semantics: database savepoints, filesystem copy-on-write. The backend does the undo — we don't guess at it. And the adapter is honest when it cannot undo (an external API), forcing that effect down the staged / gated path rather than pretending.

The moat — honestly

For a library, the code is not the moat and never will be. The engine is clever, not exotic; it is copyable, and AI now makes copying cheap. Any defensibility thesis that rests on "they can't rebuild it" is dead. We concede that on purpose, and put the moat where it actually lives for infrastructure software — the same place Redis, Terraform, Sentry and Temporal keep theirs:

  • Neutrality. The buyer needs one control point across Cursor and Claude Code and the homegrown agent. A model vendor will never govern a competitor's agent — so a neutral, cross-vendor control plane is a position no vendor can occupy. (Why Okta exists though every product has a login.)
  • Data network effect — the honest version. Pherix sits on every agent action, so it is the natural place to learn what agents attempt and what proved dangerous. The naive claim — "customer #500 inherits #1–499's raw incidents" — does not survive contact with reality: a buyer acting on systems of record will never let their journal feed a corpus a competitor draws on. So the asset is one step removed — an opt-in, anonymised pattern library: abstracted "this shape of action proved dangerous" rules, never raw effects, shipped as better default policies. It still compounds with volume and no code copy reproduces it, but it lives behind a privacy boundary rather than on top of the raw log.
  • Trust & switching cost. Nobody forks the thing standing between their agent and prod and self-maintains it. Once an org's policies, compensators and audit integrations are written against Pherix, ripping it out is expensive. Trust — eventually certified (SOC 2 / HIPAA) — is earned, not cloned.
Open-core. Open-source the engine and common adapters — that is the wedge and the distribution, not the moat. The business is the layer above: the hosted control plane (policy management, audit search & retention, cross-agent coordination) and the cross-customer policy intelligence.
One clarification — policy intelligence is not governed memory. The roadmap's Phase-3 "governed memory" pillar is a per-customer product feature: an agent's own memory placed under the same transactional, policy-bound, auditable envelope as its other effects. The cross-customer pattern library above is a different thing — anonymised, aggregate, sitting behind a privacy boundary. They share the journal as substrate but are distinct: one is a feature you sell, the other is a moat you accrue. Both stay on the map.

Why metered there, flat here. The OSS runtime is unmetered — self-host and run flat-out free, so usage pricing never taxes the behaviour we want. The managed gateway meters throughput because it's an operated service you opt into and consume (like any gateway), kept smooth and capped so there's no bill-shock. The regulated buyer — who hates variable bills — lives on the flat tier, because the compliance surface (retention, replay, SSO, audit) genuinely isn't usage-shaped. Each tier meters the thing its buyer can reason about; the metered unit is the legible one (a transaction the customer already counts), not raw effects.
Design-partner phase (now). The first one or two partners get it hands-on and effectively free — what we're buying from them is the logo, the reference, and the incident data that seeds the default-policy library. Revenue is a Phase-2-exit concern; the partner is the asset.

The technical depth — where it isn't a weekend clone

The reversible engine itself — savepoints, copy-on-write, fold-forward/back — is the easy, copyable part, and we delegate its hardest work to the backend (SQLite/Postgres do the rollback; we don't reinvent it). The genuinely hard, defensible complexity lives elsewhere — not in novel distributed-systems theory (that's decades-solved), but in getting known-but-fiddly correctness right and verified, the class of problem where competent teams ship subtle bugs. Most of this is now built (the engine-hardening pass, merged); cross-host and hard cross-process budgets are deferred to the control plane.

  • Concurrent multi-agent isolation. MVCC-style read/write-set tracking and conflict resolution. In-process and cross-process (single-host) coordination are built and tested; cross-host arbitration is deferred to the control plane. Distributed-systems correctness is precisely what AI does not reliably get right — and verifying it is the expensive part.
  • Crash-consistent compensation. Exactly-once semantic inverses under partial failure: the compensator as a left-inverse (compensator ∘ tool ≈ identity). Built: mid-commit unwind and crash-consistent recovery-on-restart — resuming an interrupted unwind correctly, exactly-once, even when the process died mid-fold.
  • The longitudinal envelope. Bounds that span the agent's whole life, not one transaction — caps folded over the durable journal ("£X / Y rows / Z irreversible calls per day, across every run"). Built (single-host): per-transaction and durable cross-session caps over the journal; hard cross-process budget enforcement is deferred to the control plane. This is what turns a gating tool into continuous lifecycle governance.
