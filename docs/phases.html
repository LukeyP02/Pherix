The product

Pherix — the guardrails layer for AI agents. Any agent plugs in over MCP; every action it takes gets ACID guarantees, a provable audit trail, and governed memory.

The positioning: we don't make the agent correct — we make it safe to be wrong. The market consensus answer to agent hallucination is "verify the reasoning" (theorem-proving, formal verification), which only works where there's a formal spec. Pherix's answer — atomic, gated, audited effects — generalises to every workflow, and it's true today, not after a benchmark clears.

Phase 2 — Earn a design partner (guardrails MVP)

⟳ underway

From "passes 367 tests" to "one real user runs it against real resources and trusts it." The dogfood suite has landed; the work now is running it for real and finding the buyer.

Exit criterionA security / platform team pays for governed + audited agent actions. The exit is commercial, not "code shipped" — which makes this the hardest phase, and the one most likely to stall on go-to-market rather than engineering.

The MVP facet-set: control + recovery + audit

  • Live MCP-client interop ("test the fucker") — done. The dogfood suite shipped a real-subprocess parity test (gateway as a child process over real stdio) — the boundary gap Slice 8 flagged is closed. Next: point a real out-of-box CLI at it (the local-openclaw capstone).
  • Run the dogfoods for real — three real-agent dogfoods are runnable now (see dogfood.html → Start here); running them with a key + a spare machine is the first-user signal that picks the wedge.
  • Production-harden the deferred gaps — Slice 6.5 (world-state-aware policy reads), FS cross-process arbitration, Serialize cross-process enforcement, the meta-connection lifecycle, the accumulated nits.
  • Audit pillar v1 — query surface over the journal, compliance/provenance views, journal-as-MCP-resource. The trust evidence a security buyer needs to sign off.
  • One wedge bolt-on — GitAdapter (coding-agent shop) or CI/CD primitive (deploy shop). Picked by the dogfood vertical, not in the abstract.
  • Dogfood one vertical end-to-end against real infra.
  • Land a design partner — the highest-leverage move, and not an engineering task. One team running Pherix in anger, with a quote, is worth more than any feature.

The YC lens

The application bar is demo + pull + narrative — not a finished platform.

YC funds "something people want" plus a team that ships, not a complete product. That shrinks the pre-application scope dramatically: most of Phase 2 and all of Phase 3 become what the funding is for. You don't need the audit pillar finished or memory started to apply.

The highest-leverage pre-application move is landing a design partner, because YC's hardest question is "who's using it?" — that flips Phase 2's order so the design-partner hunt jumps ahead of the build. Even one team with a quote beats a finished feature.

The narrative is already fundable: "agent ACID" is a crisp, category-naming wedge; agent safety/governance is a hot space; the journal is a real moat; and the cancelled-coding-CLI story is a today-pain with a today-buyer.

Timeline (confirm exact dates — YC shifts them): next realistic batch ≈ Fall 2026 (apps ~mid-summer, ~8 weeks — demo + narrative, maybe no paying user) or Winter 2027 (~6 months — room to land a design partner and show traction). Tradeoff: narrative-now vs traction-later; YC increasingly rewards traction.

Defensibility — the honest read

The code is not the moat. The engine is assembly of known parts (savepoints, an append-only journal); a competent team could rebuild it in a week. Pretending the code is IP is the trap. Real defensibility lives in four places, none of which is the code:

  • "Could build" ≠ "will build / maintain / certify / support for five years." Buyers pay infra companies to not own commodity infrastructure. The week of engine is the demo; the product is the years of adapter breadth, edge cases, compliance, and support.
  • System-of-record / audit data-gravity [strongest]. Once a company's compliance process depends on Pherix's audit log as the authoritative record, you're infrastructure, not a swappable tool. Earned per-customer; doesn't exist yet.
  • Adapter breadth + battle-tested correctness compounds with usage — a copycat is always N customers behind.
  • Compliance / trust capital — slow, hard, exactly what buyers won't build themselves.

At YC stage you are not defensible, and that's fine — YC funds velocity into a hot wedge, not moats. "Built the engine in a week solo" is the best signal in the room, not a weakness. The real threat isn't a copycat — it's platform absorption (Anthropic / a cloud / a framework bundling transactional tool-calls for free). The defense: cross-model, cross-platform neutrality (govern anyone's agents against anyone's systems — Anthropic won't ship the tool that also polices OpenAI agents and your internal APIs) plus owning the compliance/audit surface they won't prioritise, and being the entrenched neutral standard before they care.

The compressed "why you" answer: "Not the code — that's a week's work and I'll say so. The moat is being the cross-platform system-of-record for what agents did: neutral across models, owning the compliance surface the platforms won't, embedded in customers' risk processes before anyone bundles it. Today I'm defensible on speed; the durable moat is earned with customers."

Owning the honesty on the code is what makes the rest credible.
