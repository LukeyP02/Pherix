Pherix · Slice 5 decisions

Re-doing yesterday's journal — and what that's even for.

The journal is already a complete record of everything an agent did. Slice 5 turns it from a transcript into something you can execute — re-fire the same effects against a starting point of your choice. The decisions below are about why you'd ever do that, how you'd judge whether it worked, and what shape the API takes.

The premise

Yesterday, an agent did 50 things — wrote files, charged a card, queried databases, opened a PR. Pherix has the journal of every single action: the tool name, the args, the resource, whether it succeeded, what it returned. The journal is durable now (pre-work P1 made on-disk the expectation), it carries isolation keys (pre-work P2), and the versions disambiguate self-bumps from cross-txn writes (pre-work P3). It is, at this point, a complete record.

So what? What can Pherix do with that journal beyond just reading it as a log?

The answer is the same one CLAUDE.md has been promising since Slice 1: every capability is a different traversal of the journal. Commit is a forward fold; rollback is a backward fold; isolation is a diff. Slice 5 is another forward fold — but instead of folding yesterday's effects against the live world, it folds them against a fresh world the operator provides. That's replay.

D1 — Why are we even replaying?

The question

"We have yesterday's journal. We're about to re-do every action against a fresh starting point. Why?"

Two distinct reasons an operator would ever ask Pherix to replay, each useful, each shipping with this slice:

Reason A — Prove Pherix is doing its job

"I want to convince myself (or a regulator) that Pherix is deterministic. Same journal in, same world out."

Replay walks yesterday's journal, re-fires the effects against fresh adapters, and asserts each effect's result matches yesterday's recorded result. If everything matches, the engine and the tools agreed across runs — deterministic by witness. If anything diverges, something is producing different output and Slice 5 surfaces exactly which effect went wrong.

Pherix calls this: mode="verify"

Reason B — The agent crashed; rebuild the world

"Halfway through a 50-step run the process died. The journal has the first 30 steps. To resume, we need to reconstruct what the world looked like at step 30."

Replay walks yesterday's journal against a freshly-created starting point (empty database, empty filesystem, etc.), re-applies every reversible effect, trusts the journal's recorded result for irreversible effects (we don't charge the card twice), and lands in a world identical to where the original txn was when it died.

Pherix calls this: mode="reconstruct"

The options weighed

Option A — ship verify only

Smallest scope. Recovery is Slice 5+ or never.

Smaller surface area; fewer corner cases at launch.

Recovery is the use case operators most viscerally want when an agent crashes mid-incident. Punting it means the dogfood DevOps prototype can't lean on Slice 5 for resumption.

Option B — ship reconstruct only

Verification falls out as a side effect of reconstruct.

If reconstruct fires every effect and we check each result, we get verification for free.

Verification under reconstruct re-fires non-deterministic tools (timestamps, fresh IDs) and reports them as divergences — false positives. Verification wants the option to use the recorded result precisely so non-determinism doesn't trigger spurious failures.

Option C — ship both, same engine, mode parameter selects

One forward-fold walker. The mode flag picks the per-effect dispatch.

Verify uses recorded results for non-deterministic tools and asserts match. Reconstruct re-fires every reversible against fresh state and skips-and-reuses irreversibles. Both share 80% of the implementation.

Genuinely cheaper than building one then retrofitting the other; the dispatch branch is one line per effect.

Both modes feed the dogfood verticals — verify validates engine correctness in CI, reconstruct recovers crashed agents.

Locked — Option C

Both modes ship together. Operator-facing API takes mode='verify' | 'reconstruct'; the engine is one walker with a per-effect dispatch that branches on the mode.

D2 — How do we know yesterday's result "matches" today's replay?

The question

"Yesterday the journal recorded that charge_card returned {'charge_id': 'ch_abc', 'amount': 100}. Today we replay; we get back some result. Are those two things the same? What does 'same' even mean?"

Three honest definitions of "same":

Option A — strictest: byte-for-byte equal

yesterday_bytes == today_bytes

Falls apart instantly. Anything with a timestamp, a generated ID, or even a Python dict whose key order isn't stable will fail. False positives on every realistic result.

Option B — pragmatic: convert both to JSON, compare strings

Run each result through strict_json_default (the existing serialiser), compare resulting strings.

Works for most realistic results: dicts, lists, primitives, dataclasses, bytes (base64), datetimes (ISO). The serialiser already handles whatever Pherix's other infrastructure handles, so there's no new normalisation work.

Reuses existing serialisation; covers 80% of tools without any per-tool work.

A tool whose result legitimately includes a fresh timestamp will fail verification under this comparator — but that's where option C plugs in.

Option C — flexible: per-tool custom comparator

@tool(comparator=fn) where fn(yesterday, today) -> bool.

Tools whose results vary across runs but are "equal" by some domain-specific rule (ignore-the-timestamp, compare-only-the-id, whatever) declare their own comparator. The decorator change is small; the registry already has the shape for it.

Escape hatch for the 20% of tools the default doesn't cover.

Adds API surface; one more thing for tool authors to think about.

Locked — Option B as default + Option C as escape hatch

JSON equality is the default; tools that need different semantics override via @tool(comparator=fn). Most tools won't need to think about it; the few that do have a clean place to do that thinking.

D3 — What about things we already did that we shouldn't re-do?

The question

"Yesterday the agent charged a credit card. Today's replay walks the same journal. Do we charge the card again?"

Obviously no. Double-charging is exactly the failure mode Pherix exists to prevent. So replay must know the charge already happened and not re-fire it. How does it know?

The journal already records the answer. Every effect carries a status (STAGED → APPLIED → COMPENSATED...) and a recorded result. An effect that fired yesterday has status APPLIED and a recorded result. Replay sees APPLIED + an irreversible adapter (supports_rollback() → False) and skips the actual fire, using the recorded result instead.

This is exactly what Effect.effect_id was built for from Slice 1, and what the Slice 3 review captured as "the idempotency test is a pin, not a scenario — strengthen when replay arrives". Slice 5 is the scenario. The pin retires; the property gets exercised at scale.

No real choice to make — confirming behaviour

Already-applied irreversibles are skip-and-reuse: replay sees APPLIED status, doesn't re-fire, uses the journal's recorded result for the effect's contribution to the replay. The Slice 3 idempotency-key infrastructure already carries this; Slice 5 is the first place it actually matters.

D4 — What about tools that return different things every time?

The question

"A tool returns datetime.now(). Yesterday it was 2026-05-18T14:00; today's replay it'll be 2026-05-19T15:00. Is that a problem?"

Depends on what replay is for. This is exactly the place where D1's two modes branch:

Under verify mode

Use yesterday's recorded result; don't actually re-run the tool.

Verification asks "is Pherix's bookkeeping correct", not "is the wall clock correct". The journal IS the witness: if the journal records "the tool returned T", verification uses T. No false positives on every datetime.now().

Under reconstruct mode

Re-fire the tool; accept whatever fresh result comes back.

Reconstruct wants a real working world, not a frozen snapshot. If the tool returns "now", reconstruct's "now" is today, not yesterday. That's correct for the use case — we're not verifying behaviour, we're materialising a new world from a recipe.

Locked — mode parameter from D1 carries the decision

Same parameter as D1; no extra knob. Verify uses recorded results for non-deterministic tools and never false-positives on datetime.now(); reconstruct re-fires and accepts the new output.

D5 — How does the operator ASK for a replay?

The question

"What does the operator type in their Python code to start a replay?"

Option A — extend run_txn with a replay parameter

run_txn(fn=None, replay_from=txn_id, …)

One entry point for everything Pherix runs. The function knows whether it's running fresh agent code (fn=...) or replaying a journal (replay_from=...).

Conceptually muddled — running an agent's code is a different operation from replaying a recorded journal. The function ends up with mutually-exclusive parameter sets pretending to be one API.

Option B — new top-level replay() function

pherix.replay(txn_id, adapters, *, source_audit, mode=...)

Replay is its own operation: it walks a recorded journal rather than executing fresh agent code. A separate entry point keeps both APIs clean.

Conceptually honest — replay isn't "running an agent", it's "executing a transcript".

Operator-facing signature is small and focused; nothing pretends to be both kinds of run.

Locked — Option B

New top-level replay() function in pherix.frontends.library. agent_txn and run_txn stay exactly as they are; replay sits alongside them, not inside them. Signature shape: replay(txn_id, adapters, *, source_audit, target_audit=None, mode='verify').

Open — Who provides the "fresh slate"?

The question

"Reconstruct mode needs to fold the journal against fresh adapters — an empty database, an empty directory. Where do those come from?"

Two reasonable answers:

Pherix tries to make them itself

Each adapter gets a fresh() method that produces an empty version.

SQLiteAdapter.fresh() creates a new in-memory database with the same schema. FilesystemAdapter.fresh() creates an empty temp directory. Tidy from Pherix's side; one less thing the operator has to do.

Fresh adapters need environment-specific input (DB connection strings, file paths, auth tokens) that Pherix has no business knowing or hardcoding. A fresh() method either makes wrong assumptions or leaks operator config into Pherix's scope.

The operator provides them

Operator opens fresh resources and hands them to replay() as the adapters parameter.

Pherix walks the journal against whatever the operator passed. The correctness of the "fresh state" is the operator's contract; Pherix's contract is just the walk.

Respects Pherix's scope boundary — environment knowledge stays with the operator.

Same shape as how adapters are already passed to agent_txn; no new mental model.

Recommendation — operator provides

This is the open call but the recommended answer is to keep adapter construction in the operator's hands. If during implementation a clean fresh() shape emerges that doesn't leak environment specifics, surface it as a Slice 5 follow-up rather than retrofitting it mid-build.

Closing — replay is just another fold

Three sentences of maths, the only ones in this document. Commit is a forward fold over the journal against the live world. Rollback is a backward fold over the journal against the live world. Replay is a forward fold over the journal against a fresh world the operator provides.

Three operations, one engine, one journal. The Slice 5 build is mostly wiring — the engine already knows how to fold; the replay() entry point just hands it a different substrate to fold against. Verify mode asserts each step matches the recorded result; reconstruct mode accepts each step's output as the new state. Idempotency keys from Slice 3 keep us from re-firing irreversibles. Read/write keys from Slice 4 (and the P2 schema bump) let us cross-check isolation behaviour replays cleanly.

Everything below cognition; nothing about the LLM. The journal is the witness; replay is what makes the witness executable.
