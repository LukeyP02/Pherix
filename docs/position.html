Pherix · position & direction

Autonomy inside an enforced, reversible envelope.

A statement of what Pherix is, who it is for, and why it holds — written to be argued with, not admired. The engine is built; this fixes the direction.

Pherix lets you deploy an autonomous agent fast, because its actions are continuously bounded and reversible — so being wrong is cheap, and you no longer need a governance meeting to say yes.

We replace the governance meeting with a governance policy. Trust becomes mechanical, not procedural.

The stance: autonomy-first, not human-first

The whole promise of agents is collapsing weeks of work into hours. A governance design that puts a human in the loop on every action gives that speed straight back. So we reject it as the mechanism.

Governance is not a checkpoint the agent stops at. It is an invariant preserved along the whole trajectory — a bound that holds at every step, automatically, so the agent runs flat-out inside it and nobody watches.

The gate is the exception, not the product. If a demo leads with "a human approves the dangerous call," it is selling the ugly 10%. Pherix leads with the envelope; the gate is the footnote.

Who we solve for

One claim — autonomy inside a reversible envelope — stress-tested at five interception points. Each is a NO someone is saying today that becomes a bounded YES.

Coding-CLI agentsCursor · Claude Code · Aider — buyer: platform / security lead

NO: "We blocked the agentic coding tool over guardrails concerns."

YES: shell / file / git calls run through Pherix — a bad refactor rolls back byte-clean, a destructive command gates, every action is on the audit trail. The banned tool becomes a provisioned one, fleet-wide.

Audit agentsagents acting on systems of record — buyer: compliance / ops

NO: "We won't let an agent mutate the ledger / records."

YES: every mutation is reversible and recorded; a wrong remediation is undone; the journal is the compliance evidence — what was attempted, allowed, denied, undone, with provenance.

Dev agentsmigrations · infra · internal automation — buyer: engineering lead

NO: "An agent running migrations or infra changes on prod is too risky to allow."

YES: the whole run is atomic — all-or-nothing at the workflow boundary. A failed step at #14 unwinds #1–13; no half-applied mess, no manual cleanup.

Regulated agentsfinance · health · legal — buyer: risk / compliance

NO: "Compliance will not approve an autonomous agent in this domain."

YES: a policy envelope enforced continuously (and re-checked at commit), an immutable audit trail, and reversibility — the three things an auditor asks for, as first-class primitives rather than bolted-on logging.

Local agentson a dev machine / air-gapped — buyer: developer / endpoint security

NO: "I won't run an agent loose on my machine."

YES: copy-on-write over the filesystem and savepoints over local DBs mean "let it rip" — if it makes a mess, roll the working tree back to byte-identical. Model-blind and offline-capable.

What we provide — see, stop, undo

A side-effecting tool call goes from a one-way door to a door you can walk back through.

Before Pherix

Effects fire instantly, irreversibly, unlogged, unbounded. A mistake is discovered after the fact and cleaned up by hand — if it can be. Deploying the agent is an unbounded bet.

After Pherix

Effects run inside a transaction: reversible ones journalled and undoable, irreversible ones staged and bounded by policy, all of it audited. A mistake is one rollback away. Deploying is a bet you can retract.

Every other governance tool lowers the cost of detection (it tells you, after, that you were wrong). Pherix lowers the cost of being wrong. Only the second one actually unblocks deployment.

How it works — one idea

A versioned effect journal, mediated by resource adapters. Every tool call becomes an entry in an append-only, ordered log. Every capability is a traversal of that log:

  • Commit — fold forward, apply each effect.
  • Rollback — fold backward, restore each snapshot (or run its compensator).
  • Govern — evaluate policy over the journal, at stage-time and again at commit-time (the world moved between them).
  • Audit — the journal is the record; just read it.

Adapters use real backend semantics: database savepoints, filesystem copy-on-write. The backend does the undo — we don't guess at it. And the adapter is honest when it cannot undo (an external API), forcing that effect down the staged / gated path rather than pretending.

The moat — honestly

For a library, the code is not the moat and never will be. The engine is clever, not exotic; it is copyable, and AI now makes copying cheap. Any defensibility thesis that rests on "they can't rebuild it" is dead. We concede that on purpose, and put the moat where it actually lives for infrastructure software — the same place Redis, Terraform, Sentry and Temporal keep theirs:

  • Neutrality. The buyer needs one control point across Cursor and Claude Code and the homegrown agent. A model vendor will never govern a competitor's agent — so a neutral, cross-vendor control plane is a position no vendor can occupy. (Why Okta exists though every product has a login.)
  • Data network effect. Pherix sits on every agent action — a choke point. The journal across all customers becomes the only corpus of what agents attempt and what proved dangerous. That trains the default policies; customer #500 inherits #1–499's incidents. It compounds with volume and no code copy gets the data.
  • Trust & switching cost. Nobody forks the thing standing between their agent and prod and self-maintains it. Once an org's policies, compensators and audit integrations are written against Pherix, ripping it out is expensive. Trust — eventually certified (SOC 2 / HIPAA) — is earned, not cloned.
Open-core. Open-source the engine and common adapters — that is the wedge and the distribution, not the moat. The business is the layer above: the hosted control plane (policy management, audit search & retention, cross-agent coordination) and the cross-customer policy intelligence.

The technical depth — where it isn't a weekend clone

Adapters are easy to generate now; that work no longer carries the weight. The genuinely hard, defensible complexity lives elsewhere, and it is where we invest:

  • Concurrent multi-agent isolation. MVCC-style read/write-set tracking and conflict resolution across processes and hosts. Distributed-systems correctness is precisely what AI does not reliably get right — and verifying it is the expensive part.
  • Crash-consistent compensation. Exactly-once semantic inverses under partial failure: the compensator as a left-inverse (compensator ∘ tool ≈ identity), correct even when the process dies mid-unwind.
  • The longitudinal envelope. Bounds that span the agent's whole life, not one transaction — caps folded over the durable journal ("£X / Y rows / Z irreversible calls per day, across every run"). This is what turns a gating tool into continuous lifecycle governance.

What's left to build

Almost none of it is more engine — it is the gap between "works" and "a buyer says yes." Three tiers, in order.

Tier 1 — unblock the first yes

  • Longitudinal envelope — cross-session caps folded over the durable journal ("£X / Y rows / Z irreversible calls per day, across every run"). The one feature that makes continuous lifecycle governance real instead of a slogan.
  • Audit query surface (the receipt) — "every denial / gate / rollback by this agent in this window." Today the journal is only fetchable by transaction id; compliance can't buy a log it can't query.
  • Privacy / redaction seam — secrets must not land in a plaintext, delete-never journal. The procurement blocker for any security buyer. Needs a decision first — redact at write, encrypt at rest, or set retention — and it trades against replay.

Tier 2 — the conversion proof (the sales asset)

  • Finish the coding-CLI spearhead end-to-end, plus the air-gapped adversarial capstone — the demo is the pitch.
  • Journal the compensator fire — closes the "when did the refund actually fire?" audit hole; strengthens the receipt above.

Tier 3 — the business (after a design partner, not before)

  • Hosted control plane — policy UI, audit search & retention, gate approvals in Slack / web, cross-agent coordination. The monetised layer over the OSS library.
  • Cross-customer policy data loop — the moat. Needs customers first, so it is last by definition.
Two of these need you, not code: the privacy decision (Tier 1) and which vertical is the spearhead (locks the Tier 2 proof). Critical path: Tier 1 → Tier 2 → sign a partner → Tier 3. Don't touch Tier 3 until someone has signed.
