Three decisions that lock the filesystem adapter.

Slice 2 plugs a second resource adapter into the engine built in Slice 1. Three design choices were debated and locked. This page shows each one with every alternative weighed, the pick, and — most importantly — why one option follows logically from the others, not by convention.

The three picks, in one breath

D1
Optional sub-protocol for transaction lifecycle.

Only adapters that can bracket a real transaction promise to. The type system mirrors the real taxonomy of resources.

D2
FsHandle: rooted + lazy.

Tools reach inside one directory; nowhere else. The adapter snapshots a file the first time it is touched — no upfront declaration.

D3
Mixed-resource transaction.

Acceptance is one agent_txn carrying SQL and FS, both rolled back together — with no special-case engine code.

A note on jargon — read this first if any terms feel slippery.

Protocol (Python's term): an interface contract. A class "is" a Protocol if it has the right method names with the right shapes — no inheritance needed. Think of it as a predicate on types: "this thing has snapshot, apply, restore." If it does, the runtime can use it.

Adapter: a thin object that knows how to make a journal entry executable and reversible against one class of real resource (SQL database, filesystem, HTTP endpoint, ...). The whole point of Pherix is that every resource speaks the same five-method language.

The journal: an ordered list of Effects — every side-effecting tool call. Commit is a fold forward (apply each); rollback is a fold backward (restore each). If you can see why commit and rollback are duals of the same operation, you have the whole engine.

Savepoint (SQL): a marker inside a database transaction that you can later ROLLBACK TO, undoing only the work since the marker. That is exactly what the SQL adapter uses for snapshot and restore.

D1

Where does adapter lifecycle live in the type system?

Some adapters can bracket a real transaction (begin / commit / rollback). Some can't. The protocol should say which.

In Slice 1, the SQL adapter has begin / commit / rollback methods — they map onto literal SQL BEGIN / COMMIT / ROLLBACK. The runtime calls them by name. But those methods are not declared on the ResourceAdapter protocol; the runtime probes for them with hasattr (Python's "does this object have an attribute called X?" — a duck-typing check, named for "if it walks like a duck"). The Slice 1 review flagged this as debt to pay before Slice 2 — because the FS adapter needs the same lifecycle, and Slice 3's HTTP adapter (irreversible) deliberately doesn't.

Why (b) wins

Pherix's own supports_rollback() principle says: be honest about what a resource can and cannot do. Forcing HTTP to claim a transaction lifecycle violates that principle. Hiding it in hasattr hides reality from the type checker — which is the one tool that catches typos before runtime. Only (b) puts the taxonomy where it belongs: in the type.

The price is one extra Protocol class and an isinstance swap in the runtime. The reward is that "this resource has a transaction lifecycle" becomes a first-class, type-checked fact instead of folklore in runtime.py.

The actual code shape — before vs after.

Before (Slice 1)

# adapters/base.py
class ResourceAdapter(Protocol):
    name: str
    def supports_rollback(self) -> bool: ...
    def snapshot(self, e: Effect) -> SnapshotHandle: ...
    def apply(self, e: Effect) -> object: ...
    def restore(self, h: SnapshotHandle) -> None: ...

# runtime.py — duck-typed lifecycle
if hasattr(adapter, "begin"):
    adapter.begin()

After (Slice 2 lock)

# adapters/base.py
class ResourceAdapter(Protocol): ...  # unchanged

class TransactionalResourceAdapter(ResourceAdapter, Protocol):
    def begin(self) -> None: ...
    def commit(self) -> None: ...
    def rollback(self) -> None: ...

# runtime.py — taxonomy is now a type-check
if isinstance(adapter, TransactionalResourceAdapter):
    adapter.begin()

The diff is small. The semantic gain is large: "transactional" is now a property a type can have or not have, not a fact that lives only in the runtime's probe. Future adapters (HTTP, Slice 3) deliberately do not declare the sub-protocol — which is precisely how Pherix says "this resource has no undo."

Aside An object can satisfy a Protocol structurally — Python doesn't need it to inherit. So existing SQLiteAdapter code that already has begin/commit/rollback automatically counts as a TransactionalResourceAdapter. No edits to its source needed; the type system catches up with reality for free.
D2

How do filesystem tools touch files inside a transaction?

SQL tools receive a conn as their first argument — the adapter injects it. The FS adapter will inject an FsHandle the same way. What shape should it have?

Two orthogonal sub-decisions: scope (where on disk can the handle reach?) and snapshot timing (when does the adapter take the pre-state backup?). Each is decided on its own; together they define the handle.

D2a — Scope of the handle: rooted vs unrestricted

Rejected

Unrestricted

The handle exposes raw os.path writes. A tool can fs.write("/etc/passwd", ...) and the adapter will do it.

Failure mode: the agent's blast radius is the whole filesystem. Roll-back undoes the data but does not undo "the agent learned where the system files live."

Picked

Rooted

FilesystemAdapter(root: Path). Every FsHandle path is resolved relative to root. .. escapes, absolute paths outside root, and symlink games all raise ValueError.

Why: an agent able to overwrite /etc/passwd is a non-starter. Policy can layer further restrictions on top of root later; you cannot retrofit a boundary the runtime never had.

D2b — Snapshot timing: eager vs lazy

The adapter must back up the pre-state of any file an effect will modify, so restore() has something to roll back to. The question is when that backup happens.

Rejected

Eager — declare paths up front

The tool declares all paths it will touch; the adapter copies them before the tool runs.

Cost: the tool must know its paths in advance — which is often the opposite of what tools do (they compute paths from inputs).

Picked

Lazy — backup on first touch

The handle backs up a path's pre-state the first time the tool writes/deletes it inside an effect. Reads do not trigger a backup. Subsequent writes to the same path skip the backup — pre-effect state is already captured.

Why: tools don't have to know their paths upfront. Same correctness; cleaner ergonomics. The mathematical invariant — "the first backup is the pre-effect state" — is preserved either way.

Why rooted + lazy

Rooted gives a hard boundary on agent reach — a precondition for trusting the runtime at all. Lazy lets tools stay transaction-unaware, which is Pherix's core promise: agents keep their existing shape, the runtime adapts to them, not the other way round.

The two pair cleanly: the handle is the only way to touch files (so rooting is enforceable), and it always knows when a path is first written (so laziness is implementable). One object, two invariants.

What the handle looks like, in code. 
# core/adapters/filesystem.py
class FsHandle:
    def __init__(self, root: Path, effect_backup_dir: Path):
        self._root = root.resolve()
        self._backup_dir = effect_backup_dir
        self._backed_up: set[str] = set()

    def write(self, rel_path: str, data: bytes) -> None:
        abs_path = self._resolve(rel_path)   # rejects .. and out-of-root
        if rel_path not in self._backed_up:
            self._backup(rel_path, abs_path) # lazy: only on first touch
            self._backed_up.add(rel_path)
        abs_path.write_bytes(data)

The _resolve step is the rooting invariant; the _backed_up set is the laziness invariant. Both are local to the handle — nothing in runtime.py changes.

D3

What does "Slice 2 done" mean?

"Filesystem works in isolation" is not enough. The acceptance bar must prove the adapter protocol is a real abstraction, not two SQL-shaped things in a trenchcoat.

The decisive test: one agent_txn() carrying both adapters — {"sql": SQLiteAdapter(conn), "fs": FilesystemAdapter(root)}. An "agent" calls SQL tools and FS tools intermixed. Each effect is routed to its adapter by name. The journal records both, in order. Rollback folds the journal backward through both adapters; commit folds forward through both. And there must be zero special-case code in the engine beyond the D1 lifecycle swap. If cross-resource routing needs branching logic, the protocol is the wrong shape — and that is the more interesting problem to surface.

Interactive — step the mixed-resource transaction

Why this is the bar

If the engine needs any special case to handle "SQL plus FS in one transaction," the protocol isn't actually abstracting over resources — it's two adapters with hidden coupling. The mixed-resource test is the load-bearing assertion that ResourceAdapter is the right shape.

Mathematically: the journal is a free sequence over the disjoint union of effect types. Commit and rollback are folds with respect to a per-adapter action. If routing-by-resource works uniformly, the folds compose. If it doesn't, the abstraction has a leak and Slice 3 (a third adapter) would compound it. Catch it here.

The full acceptance criteria, from TASK.md.
  • Round-trip: write a file via a tool, snapshot, modify, restore → original content. Verified.
  • File that didn't exist pre-snapshot, written inside an effect → restore deletes the file (no orphan).
  • File that did exist pre-snapshot, deleted inside an effect → restore recreates it with original contents.
  • Two effects writing the same path: the backward fold restores newest pre-state, then older pre-state, landing at the original.
  • Within one effect, multiple writes to the same path → only one backup taken (the first one — pre-effect state).
  • Path traversal: fs.write("../etc/passwd", ...) or absolute paths outside root → raises ValueError.
  • Reads do not trigger backups.
  • Backup tempdir created on begin() and removed after commit() / rollback() — no orphan dirs.
  • TransactionalResourceAdapter sub-protocol exists; SQLiteAdapter and FilesystemAdapter are instances; runtime.py uses isinstance for lifecycle dispatch.
  • Regression: a custom non-transactional adapter (only snapshot/apply/restore/supports_rollback) inside an agent_txn does not trigger begin/commit/rollback calls on itself.
  • The mixed-resource cross-cutting test. Rollback wipes both DB and files; commit persists both. No engine code changes beyond the D1 swap.
  • examples/slice2_demo.py — mixed-resource fake agent; mid-sequence rollback wipes both; audit journal shows the whole story across both adapters.
∴

Why these three, together, are the slice.

D1, D2, D3 aren't independent — they hang off the same insight.

Pherix's central claim is that every capability of the system — commit, rollback, replay, dry-run, isolation, audit — is a fold over a single journal of effects, mediated by adapters. Slice 2 is the first stress-test of that claim with a non-SQL resource. So:

D1 makes the protocol honest about which adapters can bracket a transaction — without that, the runtime keeps guessing via hasattr and the type system lies about Pherix's own taxonomy. D2 makes the new adapter fit the protocol cleanly — rooted means the boundary is enforceable, lazy means tools stay transaction-unaware, and both invariants live inside the handle so runtime.py doesn't need to know. D3 is the load-bearing experiment that says: if these two adapters compose under one fold with no special-case code, the abstraction is real.

That's the slice. Three picks, one claim, one engine.
