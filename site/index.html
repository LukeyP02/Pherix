The problem

An agent calls a tool and the effect just happens — a wiped table, a wrong payment, a clobbered file. No undo, no approval, no record.

Pherix sits at the tool-call layer and makes that mistake contained instead of catastrophic: reversible effects roll back, irreversible ones wait for approval, and the whole run is on the record.

What you get

01

Undo the reversible

Database and file writes roll back through the backend's own savepoints and copy-on-write. The backend does the undo — exact, not best-effort.

02

Gate the irreversible

A charge or an email can't be un-sent, so it doesn't fire until commit — and one with no safe undo waits for a human to approve it.

03

Audit everything

Every tool call is recorded — who did what, when, allowed or denied. The audit trail is a byproduct, always on.

Where it earns its place

Ops agent

An agent working on a production database. Let it act, and roll the whole step back the moment something looks wrong. Nothing sticks until you commit.

Payments agent

A charge waits behind the gate; a refund is its registered undo. A failed run unwinds cleanly — the money moves only when the whole step is sound.

Data agent

Bulk writes across a database, object storage, and search all run as one step. Commit them together, or roll them all back together.

How it fits

Pherix is a library. You pip install it, wrap your agent's tool-call layer, and keep your existing agent loop and model. It doesn't run agents or call models — it makes their actions behave like a database transaction. Pre-release and self-hosted: no console, no hosted service, no pricing.
