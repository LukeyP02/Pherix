Vision · Slices 1—4 shipped · 5 next (replay) · 6—8 ahead · then the trilogy

One engine, three faces.

Pherix grows by traversing the same sequence in a new direction each time. Forward fold. Backward fold. Fold against fresh state. Fold against a snapshot, then discard. Diff. Walk. Six verbs over one ordered object — the journal — yield every capability the system claims. Slices 1—8 ship the determinism engine. Beyond, two more pillars grow from the same substrate: a memory layer that indexes the journal, and an audit layer that queries it.

Section 01 · The unifying axiom

One engine, seven costumes.

Slice 1 built an ordered, append-only list of Effect records — the journal — plus a ResourceAdapter protocol (a triple (snapshot, apply, restore) wrapping a real backend). The claim of this document is simple: every later slice is a different traversal of that same list.

A journal is an ordered sequence e1, e2, …, en. Each ei carries an inverse ei-1 (via its adapter's restore). Once you have an associative composition and inverses, the operations you can do are the ones the structure permits — and that is exactly the list below.

Capability Journal operation What you assert at the end Slice
Commit forward fold — e1 ∘ e2 ∘ … ∘ en via adapter.apply real backend now holds the composed effect Slice 1
Rollback backward fold — en-1 ∘ … ∘ e1-1 via adapter.restore backend identical to its pre-txn state Slice 1
Replay forward fold against fresh state every ei.result equals the original (per the comparator) Slice 5 · shipped
Speculative dry-run forward fold against a snapshot, then discard diff = the hypothetical change set Slice 7
Isolation / conflict diff of this txn's read_keys / write_keys against the committed journal no overlap ⇒ commit; overlap ⇒ abort / retry / serialize Slice 4 · shipped
Audit / observability walk — just read the journal causal trace for free; it is already there Slice 1

Five hard problems — rollback semantics, isolation, partial-failure correctness, deterministic replay, what-if simulation — reduce to six verbs over one list. That is the wedge. The slices below are each one verb wearing a different costume.

Slices 1–5 are merged — the SQL adapter at ad4e9c0, the filesystem adapter at c926f97, the irreversible / HTTP lane at a13018b, isolation (read/write-set capture through the resource handle, commit-time conflict diff via per-adapter versioning, resolution policy Abort / Retry(N) / Serialize) at 0fab344, and replay (pherix.replay(txn_id, adapters, source_audit=...) folds the journal forward against fresh state, verify mode asserts each effect's recorded result equals the replayed result with a per-tool @tool(comparator=fn) escape hatch, reconstruct mode rebuilds the world; irreversible APPLIED effects are skip-and-reused) — 249 tests green. Slice 1 is covered in detail in slice-1.html; Slice 6 (real policy engine) is next on the critical path. This document is the trajectory: how the same engine extends to real policy, speculative execution, and a language-agnostic gateway — without rebuilding the engine — and then how two further pillars, memory and audit, grow from the same journal.

Section 04 · The slices in turn

Each slice, the same verb in a new costume.

For each slice: what it adds, what it unlocks for the operator running an agent under Pherix, and the operation it performs over the journal. The third line is the load-bearing one — it is what makes the slice cheap. Slices 2–4 are shipped (past tense). Slice 5 is next on the critical path. Slices 6–8 are still ahead.

Slice · 02 · shipped c926f97filesystem

The filesystem adapter

A structurally different resource — same protocol. The abstraction has earned its keep.

What it added

A FilesystemAdapter(root) that snapshots files via a per-transaction tempdir backup, applies tool writes through an FsHandle (rooted, with lazy per-path snapshotting and explicit path-traversal & symlink-escape defence), and restores by copying backups back into place (or deleting paths that did not exist before the effect). A new sub-protocol TransactionalResourceAdapter now formalises the begin / commit / rollback lifecycle that SQLite previously had only via duck-typing — the type system reflects the real taxonomy of resources, and a typo'd begin fails at type-check rather than silently skipping.

What it unlocked

The acceptance bar — cleared — was a mixed-resource transaction: one agent_txn writes to the database and the filesystem, you call rollback(), and both are wiped. The demo lives in examples/slice2_demo.py. The operator can stop thinking about Pherix as a SQL thing — it is a transaction over arbitrary resources, with the second resource as the existence proof.

How it traverses the journal

Same forward fold for commit, same backward fold for rollback — a new adapter under the hood, nothing else. Heterogeneous effects compose in journal order because apply and restore are independent per-effect. The journal does not know — or need to know — that two adjacent effects touch different kinds of resource. The protocol survived contact with a structurally different backend; that was the test.

Slice · 03 · shipped a13018birreversible

HTTP & the irreversible path

Honesty about what cannot be undone. Stage, gate, compensate.

What it adds

An HTTPAdapter whose supports_rollback() is False — honest, not pretend. The runtime routes such effects down the irreversible lane: instead of executing live, they become StagedResult placeholders. At commit(), staged effects fire in journal order. Each may have a registered compensator (an inverse tool that runs on rollback after the fact — e.g. issue a refund); without one, the effect gates and commit() blocks until approve_irreversible() is called by a human or higher-trust policy. Every staged effect carries an idempotency key = hash(txn_id + index + tool + args), so a re-fire after a partial crash hits the API at most once.

What it unlocks

The agent can plan with calls to "send the email", "charge the card", "post to Slack" without any of them happening yet. rollback() before commit means those calls simply never occurred. commit() with gating means a human can vet the planned irreversible set as a batch — the same review surface as a DB transaction, applied to the rest of the world.

How it traverses the journal

Forward fold delayed to commit(): staged entries are inert until then. The "backward fold" for an already-applied irreversible effect is not free — it is the registered compensator, which is itself a new effect appended to the journal. Inverses are still there, but as declared rather than structural.

Slice · 04 · shipped feat/slice-4isolation

Isolation between concurrent agents

A diff, not a lock. Abort · Retry(N) · Serialize, with the journal as arbiter.

What it added

Every Effect already carried read_keys ((resource, key, version) triples observed) and write_keys ((resource, key) pairs altered) — baked in from Slice 1 precisely so isolation was never a retrofit. Slice 4 turned those slots from inert metadata into live MVCC. Capture is automatic through the resource handle: FsHandle.read/write/delete records keys against the active Effect; SQL has explicit execute_isolated(conn, stmt, params, reads, writes) because parsing SQL is out of scope. Versioning is per-adapter (VersionedResourceAdapter sub-protocol): SQL via a monotonic counter side-table read through a dedicated meta-connection (so cross-process commits see each other's bumps), FS via sha256 content hash, HTTP not-conforming (irreversibles are isolated-by-construction via staging). The resolution policy is an operator-chosen callable f: Conflict → Action — Abort(), Retry(N) via run_txn(fn, ...), or in-process Serialize().

What it unlocked

Two agents acting on the same resources stop stepping on each other. The losing transaction is told why it lost — "you read row 42 at version 3, but version 4 was committed by txn B" — and the resolution policy gives it a path: redo, abandon, or wait its turn. No global mutex. No "just hope". Multi-agent in one Python process is the in-process JournalRegistry case; multi-process on one host is the filesystem-shared SQLite side-table case — both substrates ship.

How it traverses the journal

Set diff at commit-time. For every (resource, key, v_at_read) in this txn's read-set, ask adapter.read_version(key) now. If the tag has moved, flag a conflict. The resolution policy decides what to do with the list. Conflict detection is structurally a relation on the journal × current adapter state, not a procedure.
Slice · 05replay · shipped

Replay from the journal

Nearly free — the journal already exists. Shipped.

What it adds

A top-level pherix.replay(txn_id, adapters, source_audit=..., mode='verify'|'reconstruct') entrypoint: load the recorded journal from source_audit, fold it forward through the operator-supplied fresh adapters. Under mode='verify', each replayed result is compared against the original via the per-tool @tool(comparator=fn) (default canonical-JSON equality); divergences raise ReplayDivergence. Under mode='reconstruct', today's result becomes the new world's state. Irreversible APPLIED effects are skip-and-reused from the journal, never re-fired — Pherix cannot honestly snapshot an external HTTP call, so the journal is the witness for it.

What it unlocks

Determinism checks (verify). Disaster recovery (reconstruct: rebuild end-state from the journal on a fresh substrate). Debugging: rerun a problematic txn against fresh adapters with breakpoints. Compliance: prove that a sequence of agent actions reconstructs to a specific state. The journal becomes truth, not just a trace.

How it traverses the journal

Forward fold, second time around. Same direction as commit; the difference is the substrate — not the live backend, a fresh one. The new Transaction carries replayed_from = source_txn_id; the replay's own audit row links back. If the agent and adapters are deterministic the verify-fold lands on the same observables; if not, the comparator flags the offending effect.
Slice · 06policy

The real policy engine

Evaluated twice. State changes; permission must too.

What it adds

Replaces the Slice 1 allow-list with capability grants ("this agent may write to orders.*"), spend caps ("at most $5 across all charge calls this txn"), and content-aware rules ("never email anything containing password"). Evaluated at stage-time (fail fast, before any work) and again at commit-time (state may have changed in between — classic TOCTOU defence: time-of-check vs. time-of-use). A grant that was true when an effect was staged might be false by the time the whole transaction is ready to land.

What it unlocks

The operator can hand an agent a budget and a permission set and trust them. Tools and arguments are inspected; the policy decides per-effect and per-transaction. Capability enforcement becomes a first-class commit-time primitive — one of the three wedges that nobody else has.

How it traverses the journal

Walk + diff. Walk the journal to compute aggregates ("total spend so far", "domains contacted"); diff each effect's request against the policy's grant set. Same operations, new predicate — that is why it bolts onto Slice 1 without a rewrite.

Slice · 07dry-run

Speculative dry-run

Run it — then throw the world away.

What it adds

A dry_run() entrypoint: take a snapshot of the involved resources, fold the proposed journal forward against the copy, capture the resulting diff, throw the copy away. No live effect leaks. Irreversible effects are recorded as "would have called X with Y" without firing.

What it unlocks

"Show me what this agent would do." Pre-flight checks before committing. A/B prompts compared on their projected change set. Cost projections without spending money. The agent's plan becomes diffable in the same way code is — review the change, then approve the commit.

How it traverses the journal

Forward fold, against a snapshot, with the result discarded. Mathematically: diff = state(fold(journal, snapshot)) − snapshot. The work was already done by Slice 1's snapshot and Slice 5's fold mechanism; Slice 7 just wires them up differently.

Slice · 08gateway

The MCP gateway front-end

Same engine, new mouth.

What it adds

A second front-end: instead of intercepting in-process Python function calls, the gateway speaks MCP (Model Context Protocol — the wire protocol agents use to talk to tool servers). Agents in any language — TypeScript, Go, Rust, a hosted SaaS — route their tool calls through the gateway. The gateway translates each MCP message into an Effect and feeds the same core/.

What it unlocks

Language and deployment independence. You no longer need to pip install pherix into the agent process — the agent can be anywhere, in any language, in any framework, as long as it speaks MCP. Pherix moves from "a Python library" to "an infrastructure component you point your agent at".

How it traverses the journal

Identical traversals to every previous slice. The gateway is a different interception point, not a different engine. The core does not know, and does not need to know, whether the Effect on its desk came from an @tool-decorated Python function or a JSON-RPC message off a socket.

Section 06 · The shape that comes from Slice 8

Two front-ends, one core.

This is the architectural fact that licences the whole roadmap. The core/ directory of Pherix — journal, runtime, adapters, policy, isolation — is driven by a front-end: a thin layer that turns "something the agent did" into "an Effect on the journal". Today that something is a Python function call. Tomorrow it is an MCP wire message. The core does not change.

frontends/library.py · today

Interception point: the @tool decorator. A ContextVar (Python's async-safe thread-local) holds the active transaction; tools check it on call and route through the runtime if one is active.

  • In-process Python only
  • Agent code stays transaction-unaware
  • pip install pherix, wrap your loop, done
  • Zero network hops
  • Same language as the agent

frontends/proxy/ · Slice 8

Interception point: the wire. The gateway terminates the MCP connection, translates each tool-call message into an Effect, runs it through the same runtime, and returns the result over MCP.

  • Language-agnostic: any MCP-speaking agent
  • Out-of-process: the agent need not trust Pherix's runtime, only its wire
  • One Pherix instance can mediate many agents (Wedge 3 lives here)
  • Suits SaaS / managed deployments
  • Same engine, same journal, same audit
why this matters

The discipline pays off here.

Because core/ was kept ignorant of how it is driven — the CLAUDE.md rule "frontends/ is thin; core/ knows nothing about how it is driven" — bolting on the gateway is additive work, not a rewrite. Both front-ends produce the same Effect objects and feed the same journal. An audit row from a Python-library txn and an audit row from an MCP-gateway txn are indistinguishable. That is what "front-end-agnostic" buys.

Pillar II · first beyond Slice 8

Memory substrate — indexing the journal

The journal is the row. Embeddings are views over it. Agent memory finally stops being a guess because the truth path is singular.

The hurdle, stated plainly

Most production agent memory today is built on vector similarity search over embeddings. The pattern is honest about its compromises but rarely about their consequences:

  • Non-deterministic. Same query, slightly different recall. A similarity function returns approximately-similar chunks; cosine distance between query and stored vector ranks results, but rank-order shifts with embedding model version, index parameters, even floating-point order.
  • Lossy. An embedding is a projection Rtext → Rd from arbitrary text into a fixed d-dimensional space (typically d = 768 to 3072). Projections discard. A summary discards differently. Either way, the stored object is not the fact — it is a sketch of the fact.
  • Causally amnesic. Ask "why does the agent believe X?" and you get a plausible reconstruction, not a defensible answer. The system can point at which embedding chunks ranked highest — not at which events caused the belief.

The field treats this as a foundational unsolved problem. Pherix's journal is the structural opposite of the lossy embedding store, and that opposition is the whole opportunity.

The reframe: journal as ground truth, embeddings as views over it

This is the load-bearing claim. Borrow the shape of a relational database with full-text search: full-text indexes find candidate rows quickly and approximately; you read the row for truth. The full-text index is never the source of truth — it is a probabilistic accelerator over rows that are themselves exact.

Apply the same shape to memory. The journal is the row. Every fact in it — every Effect — happened at a specific time, in a specific transaction, with specific arguments, under a specific policy, with a specific outcome. Provenance is built in by construction; it is not an afterthought retrieved by guessing at chunks. The embedding store does not go away. It stops being the truth and becomes one retrieval index among several.

The diagram cashes out three structural advantages an embedding-only stack cannot have:

Multiple indices, one truth

Semantic embedding for "similar memories". Temporal for "what happened on day X". Causal for "what led to fact Y" — literally walk parent effects in the journal. Graph for "what's related to entity E". The agent picks the retrieval strategy; whichever it picks, the answer resolves to the same underlying journal. The query path is plural; the truth path is singular.

Embedding drift stops corrupting truth

Swap embedding model versions — and you will — re-index, facts don't change. The journal is bytes. A re-embedded store yields different rankings; the rows themselves are byte-identical. Today, when teams upgrade their embedding model, they are quietly upgrading their memory; under Pherix, they are upgrading their index.

Provenance becomes queryable

"Why does the agent believe X?" no longer means "which chunk happened to be near in embedding space". It means: walk the journal backwards from the assertion of X. Find the effects that produced or read it. Show the transactions they were part of. Show the policy under which they were permitted. That is the same audit walk Slice 1 already gives you — just queried in service of a different question.

Substrate vs. use — the most important distinction

If you take away one thing: the AI community's "memory is broken" debate conflates two layers. Most complaints labelled as use failures are actually substrate failures. Pherix attacks substrate hard. It does not touch use.

Substrate problem

Pherix's job
  • "The agent forgot what we talked about." storage layer dropped or summarised it — substrate
  • "The agent hallucinated a fact about me." no row to resolve to — the index was the truth — substrate
  • "The agent said something it should have forgotten." no decay / redaction policy on stored effects — substrate
  • "Two agents disagree about the user." no isolation, no shared journal of truth — substrate
  • "I can't prove why the agent believes X." no provenance trail — substrate

Use problem

Model's job · not Pherix
  • "The agent retrieved the wrong memory for the current question." choosing what to recall — use
  • "The agent fixated on an irrelevant past detail." attention weighting — use
  • "The agent didn't generalise across past episodes." model reasoning capability — use
  • "The relevant memory didn't fit in the context window." architectural limit of the model — use

Pherix raises the ceiling of what can be remembered reliably. It does not replace the model's job of choosing what to remember and when. A perfect Pherix-backed memory store can still be used badly by a model that can't reason over it well — and conversely, no amount of clever prompting can compensate for a substrate that quietly loses, drifts, or fabricates.

Earned by end of Slice 8? Architecturally yes. Product-ready no.

By Slice 8 the engine supports memory as a use case: snapshot/rollback, isolation between concurrent agents reading and writing memory, deterministic replay of how a memory state was reached, policy enforcement at commit, speculative dry-run of memory mutations, and a language-agnostic gateway. Everything a memory system needs from a transactional substrate is already there.

What is not there is the memory-specific layer on top. That is Slice 9 and beyond — the same pattern of additive growth that took Pherix from Slice 1 to Slice 8, applied to a new resource family.

Three things that arrive in Slice 9+, each a thin layer on the existing engine:

A memory-adapter family

A VectorAdapter (similarity, HNSW recompute), KV stores with TTL/decay, summarisation-aware stores. A VectorAdapter knows that "write" is more than insertion (it triggers index recomputation) and "read" is similarity-bounded — which changes what snapshot/restore even mean. The protocol does not change; the implementations encode the new resource semantics.

A memory-specific policy vocabulary

PII redaction predicates. Decay rules ("decay PII after 30 days; never decay user-stated preferences"). Multi-agent visibility ("agent A can read this memory; agent B cannot"). Consensus requirements ("a memory mutation requires N agents to agree"). These are new predicates consumed by the existing Slice 6 policy engine — not a new engine.

An agent-memory frontend

Same way Slice 8's MCP gateway is a frontend for tool calls, Slice 9+ has a frontend for memory ops. Agent frameworks — Claude Code, LangChain, Letta, LangMem — plug in directly and inherit transactional memory operations without rewriting their stack. write_memory(key, value) becomes a journalled effect; recall(query) becomes a two-stage retrieval that resolves to journal rows; their existing code paths stay shaped the way they already are.

Bootstrap angle: Claude Code's own memory is already Pherix-transactable

free dogfood · available today

The first honest memory test runs through your own filesystem.

Claude Code stores user / project / feedback memory as files in ~/.claude/projects/<project>/memory/. With Slice 2 merged (c926f97), those memory files are already Pherix-transactable resources — no new code, no new adapter, no new protocol.

Saving a memory inside an agent_txn and rolling back when the conversation goes wrong is the first honest dogfood of the memory thesis. It is available now. The memory-adapter family of the first pillar beyond Slice 8 generalises the idea; the filesystem case is the existence proof, already in tree.

Honest scope note

The memory hurdle is two hurdles entangled: the persistence/structure layer (substrate) and the model's own limits (context window, generalisation across episodes, retrieving the right memory at the right time). Pherix attacks the first. It does not attack the second.

A perfect Pherix-backed memory store can still be used badly by a model that can't reason over it well. Pherix raises the ceiling of what can be remembered reliably; it does not replace the model's job of choosing what to remember and when. The wedge here is the same as the wedge for the other seven slices: nobody else makes the substrate this honest.

The memory thesis is the same trick one more time. Take a hard problem — agent memory, here — and notice that the structural half of it (storage, provenance, isolation, replay, decay-as-policy, dry-run a memory write) is exactly the set of things a journal-plus-adapters substrate already gives you. Add the resource-specific adapters and predicates on top. That is the first pillar beyond Slice 8. The engine, again, does not need to change.

Pillar III · second beyond Slice 8

Observability & audit — querying the journal

The capability table already names this: audit / observability = walk the journal. Slice 1 ships the walk for free. The audit pillar exposes that walk — as a query language, as compliance projections, as replay-on-demand — for humans and regulators rather than only for the runtime.

The shape, in physics

Treat the journal as a time series of observables. Each Effect is a sample point: a tuple (effect_id, ts, txn_id, idx, tool, resource, args, read_keys, write_keys, status, snapshot, result). The audit layer is then a query language over that time series. Replay is a fold over a contiguous sub-sequence. Compliance views are projections — dropping columns or filtering rows. Concurrent agents impose a partial order under happens-before via shared read_keys/write_keys; auditing causal precedence between them falls out of that ordering for free.

No new substrate. No new engine. Just the journal — consumed through different lenses.

Lenses on a single journal

Same visual rhetoric as the memory pillar: the journal at the centre, multiple consumers reading from it. Memory's lenses are indices (lossy, ranked, approximate). Audit's lenses are queries (filters, projections, folds — exact). The substrate is shared.

What this pillar adds on top of the engine

Slice 1 already persists the journal to SQLite; the audit pillar exposes it for human and regulator consumption. Four lenses, each a thin operation:

A journal query interface

Filter the time series by predicate: ts ∈ [t0, t1], tool = "charge_card", resource = "db:orders", status = GATED, txn_id = …. Compose freely. Project to whichever columns matter. Same shape as SQL SELECT … WHERE, because that's all it is — the journal is already in a relational store.

Transaction replay & inspection

Given txn_id, walk its effects in idx order. See what fired, what compensated, what gated, when, by whom, against what policy. Replay (fold over a sub-sequence against fresh state — Slice 5's machinery, reused) reproduces the run for incident response or debugging without touching production.

Compliance projections

"Show me every effect this agent fired against this resource in the last 90 days, with the policy decision that permitted each one." Filter, project, export. The journal's structure is already regulator-friendly because the regulator's question is a fold/projection — the same shape as everything else.

Observability dashboards

Aggregates over time: gate rate, compensator firing rate, conflict rate from Slice 4, throughput per tool, anomaly highlights. Each metric is a fold of the journal under a different reducing function. No separate metrics pipeline; the same time series feeds them all.

The other observability story — and why this is a different category

Most "agent observability" today — LangSmith, Langfuse, Arize, Helicone — traces what the model said: prompts, completions, token counts, latencies, model versions, tool-call intents. Useful. Not the same data category. Pherix's audit row is what happened in the world: the resource-level effect, the transaction it belonged to, the isolation outcome, the gate event, the compensator firing.

Different category of data; complementary, not redundant. Trace observability tells you why the agent decided to act. Pherix's audit tells you what actually landed, in what order, under what isolation, with what permission. The same regulator-friendly determinism story the database industry has shipped for forty years, applied to agent side-effects. Where the LLM trace ends, the Pherix audit begins.

Replay as fold over a sub-sequence

The mathematical shape worth making explicit. Pick any txn_id; its effects form a contiguous sub-sequence ei, ei+1, …, ej of the global journal, ordered by idx. Replay folds that sub-sequence forward against a fresh substrate of each resource type. Same shape as Slice 5; this pillar just exposes it as a tool for humans rather than a determinism check for the runtime.

Same machinery as Slice 5's determinism check — that's the point. The runtime built it once. The audit pillar gives the operator a button.

Honest scope note for this pillar

The audit pillar does not attempt to be a model-trace observability product. LangSmith, Langfuse, Arize already exist and do that job well. The point of this pillar is that nobody is doing resource-level transactional audit for agents — because nobody else has a journal of resource-level effects to query.

The two layers compose: most teams running serious agent infra will want both a trace product (why the model decided) and Pherix's audit (what actually happened, in what order, under what isolation and policy). The two answer different questions; they don't compete.

Two pillars beyond Slice 8, one substrate. Memory indexes the journal — lossy lenses with the truth path resolving to exact rows. Audit queries the journal — exact lenses for humans and regulators. Both rest on the same fact: the journal is the time series of resource-level observables. Every face of Pherix — the seven deterministic verbs, the memory indices, the audit queries — is one operation on that time series. Six verbs, three faces, one substrate.

Section 08 · Closing

The shape of what Pherix becomes.

A versioned effect journal. A protocol for resources. A pair of folds, a couple of diffs, a walk. Every other capability is one of those, applied to a different argument — whether the consumer is the runtime, a memory index, or a regulator.

The slices in this document are not seven independent products glued together. They are seven views of the same object — the journal — under seven different traversals. The cheapness of the later slices is not an accident of scope; it is the dividend of having built the engine carefully the first time, in Slice 1. And the two pillars beyond Slice 8 — memory, audit — are the same dividend paid one more time: index the time series, or query it, but do not rebuild it.

If you find yourself, mid-slice, designing something that does not reduce to "fold forward, fold backward, fold against fresh state, fold against a snapshot then discard, diff two sets, or walk" — the slice has either drifted out of scope, or the abstraction has sprung a leak. Both are recoverable; both are worth catching early.

Five hard problems. One engine. Seven slices, three faces — each the same idea wearing a different costume.
