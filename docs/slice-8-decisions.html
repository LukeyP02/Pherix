Pherix · Slice 8 decisions

The same engine, driven by a protocol instead of a Python import.

Slice 8 is the test of the whole build's central discipline: core/ knows nothing about how it's driven, so a second front-end can bolt on with no rewrite. If that discipline held across Slices 1–7, this slice is wiring — a process boundary and a wire format. If it didn't, this is where it shows. Six decisions below (one more than usual — Slice 8 has the most open architectural surface of any slice), one open call, plus the absorbed Slice-7.5 state-diff work folded into the result-shape decision.

The premise — the discipline on trial

Since Slice 1, CLAUDE.md has insisted: "frontends/ is thin; core/ knows nothing about how it is driven. That is what lets the MCP proxy bolt on later with no rewrite." Slice 8 is where that claim gets tested. The library front-end (agent_txn, run_txn, replay, dry_run) is one driver. The MCP gateway is a second driver for the same engine. If building it requires changes inside core/, the discipline failed somewhere earlier — and we'd rather find out now than at v1.0.

MCP — the Model Context Protocol — is Anthropic's open standard for how an agent (Claude Code, Aider, Cursor's agentic mode) talks to external tools. An MCP client sends "call tool X with args Y"; an MCP server runs the tool and returns the result. Slice 8 makes Pherix an MCP server that sits between the agent and its tools — every tool call the agent makes is wrapped in a Pherix transaction before it touches the world.

In scope

A minimum-viable gateway that proves the discipline.

  • An MCP server speaking enough of the protocol that a real client (Claude Code) can connect and have its tool calls wrapped.
  • Tool-call dispatch through agent_txn / dry_run.
  • Per-client identity + policy at connection time.
  • The absorbed Slice-7.5 StateDiffable work (the wire consumer needs pre-computed diffs).
  • Cross-process audit + arbitration via Slice 4's meta-connection machinery.

Out of scope

Anything beyond proving the second driver works.

  • Full MCP feature set (resources, prompts, sampling, completions) — tool-call only.
  • Multi-machine / cluster gateways — same-machine, separate-process only.
  • A web UI, dashboards, or hosted control plane.
  • Cryptographic client authentication — identity is a declared string for v1.
  • Concurrency-aware dry-run (dropped with Slice 7 Strand B).

D1 — What does "MCP gateway" actually deliver?

The question

"MCP is a sizeable protocol — tools, resources, prompts, sampling. Does Slice 8 implement all of it, or just enough to prove an agent's tool calls can be wrapped?"

Option A — full MCP spec compliance

Tools, resources, prompts, sampling, completions — a feature-complete MCP server.

Most of the MCP surface has nothing to do with Pherix's wedge. Resources/prompts/sampling are about feeding context to the model; Pherix is about transacting the model's effects. Building them is effort spent away from the thing being tested.

Turns a discipline-test into a protocol-implementation project. Scope explosion.

Option B — tool-call interception subset

Implement exactly the MCP surface needed to wrap an agent's tool calls: tools/list, tools/call, plus the handshake.

An MCP client connects, discovers the tools Pherix exposes (the operator's registered @tool functions), and calls them. Each tools/call is dispatched through a Pherix transaction. That's the whole wedge — the rest of MCP is orthogonal to what Pherix does.

Keeps the slice a discipline-test, not a protocol project.

The tool-call flow is exactly the surface where Pherix's guarantees matter — wrapping it is the honest proof.

Other MCP features can be added incrementally later if a real consumer needs them, without touching core/.

Locked — Option B

Slice 8 implements the MCP tool-call subset: initialize handshake, tools/list, tools/call. Resources, prompts, sampling, and completions are out of scope — they're about feeding the model, not transacting its effects. The gateway exposes the operator's registered Pherix tools and wraps each call in a transaction.

D2 — Where does the gateway run relative to Pherix?

The question

"The MCP client (Claude Code) is its own process. The Pherix engine is Python. Does the gateway live in the same process as the engine, a separate one, or a separate machine?"

Option A — same process

The gateway is a Python module inside the same runtime as core/; the MCP client talks to it over stdio/socket but the engine is in-process.

Simplest — no cross-process state to coordinate.

Doesn't actually test the front-end-agnostic discipline. The whole point is "a different driver in a different process can use the same core." Same-process is just the library with extra steps.

Option B — separate process, same machine

The gateway is a Python process the MCP client spawns (or connects to over a local socket). The engine runs inside the gateway process; the agent is fully external.

This is the real test. The MCP client (Claude Code) is one OS process; the Pherix gateway is another. They speak MCP over stdio or a local socket. The gateway holds the engine; the agent never imports Pherix. If the engine needed agent-side state, this is where it'd break — and the discipline says it won't.

Genuinely exercises the front-end-agnostic claim — the agent is in a different process with no Python coupling.

Matches how real coding-CLIs consume MCP servers (spawn a subprocess, talk stdio).

Cross-process audit + arbitration (D4, D6) become real and testable — which is the point of the interception-axis story.

Option C — separate machine (RPC / network)

Gateway is a network service; clients connect over TCP/HTTP from other hosts.

Multi-machine arbitration, network auth, TLS, connection management — a distributed-systems project, not a discipline-test. Overscoped for Slice 8. A later slice can lift the same gateway to a network service if a real deployment needs it.

Locked — Option B

Separate process, same machine. The gateway is a Python process speaking MCP over stdio (the standard coding-CLI transport); the engine runs inside it; the agent is fully external with no Pherix import. Cross-process concerns (audit identity, arbitration) are in scope precisely because this boundary makes them real.

D3 — How does the gateway know what an agent is allowed to do?

The question

"A library caller constructs a Policy and passes it to agent_txn. But the agent is now across a process boundary and doesn't know about Pherix. Where does the policy for this agent come from?"

Option A — policy travels in each MCP request

Every tools/call message carries the policy to apply.

The agent doesn't know about Pherix policy — it can't author one. And if it could, an agent self-declaring its own permissions defeats the entire point of an enforcement layer.

Option B — identity at handshake, policy from gateway config

The MCP client declares an identity at initialize; the gateway maps identity → policy bundle from operator-supplied config.

# operator configures the gateway (Python or a config dict)
gateway = PherixGateway(
    adapters={"sql": SQLiteAdapter(conn), "fs": FilesystemAdapter(root)},
    policies={
        "claude-code": Policy.with_rules(rules=[...], caps=[Cap.sum(...)]),
        "aider":       Policy(deny={"delete_database"}),
    },
)
# at connect time, the client's declared identity selects its policy.

The operator — not the agent — owns the policy. The agent declares who it is; the gateway decides what that identity may do. This is the capability-grant model CLAUDE.md named as the competitive wedge, now distributed across the process boundary.

Enforcement integrity — the agent can't grant itself permissions.

Per-agent policy (Claude Code gets one budget; Aider another) falls straight out.

The Slice 6 Policy object is reused verbatim — the gateway just selects which one per connection. No core/ change.

A declared identity string is spoofable. Acceptable for v1 (see the out-of-scope note on crypto auth); the gateway trusts its local clients the way a Unix socket trusts local processes.

Locked — Option B

Identity declared at the MCP initialize handshake; policy selected from operator-supplied per-identity config. The agent declares who it is; the operator's gateway config decides what that identity may do. Slice 6's Policy is reused unchanged — the gateway is just a policy selector, not a new policy engine.

D4 — How does the audit row record who made the call?

The question

"The audit journal logs every effect. With a library caller there's one implicit actor. With the gateway, calls come from named agents across the boundary. How does the journal answer 'which agent did this'?"

Option A — a client_id column on the audit row

The identity from D3's handshake flows into TxnContext and lands as a client_id column on the transactions table.

Mirrors exactly how Slice 5 added replayed_from and Slice 7 added dry_run — a single additive column, keyword-only param on record_transaction, default None for library callers. An operator querying the audit can answer "show me everything aider did today" with WHERE client_id = 'aider'.

Tiny, additive, matches the established audit-column pattern (third time — replayed_from, dry_run, now client_id).

Library callers are unaffected (client_id defaults to NULL).

Directly serves the audit pillar (beyond Slice 8) — per-agent attribution is the compliance question that pillar answers.

Option B — implicit identity (connection / process id)

Use the OS connection or process id as the actor.

A process id is meaningless in a compliance view three months later. The operator wants "aider", not "pid 48213". Semantic identity beats incidental identity.

Option C — cryptographic actor attestation

Each call signed by a verified client key.

Overscoped for v1. The local-process trust model (D3) doesn't need it. A later slice can add signing if a deployment requires non-repudiation.

Locked — Option A

A client_id TEXT column on the transactions audit table, populated from the D3 handshake identity, default NULL for library callers. Same additive pattern as replayed_from (Slice 5) and dry_run (Slice 7). Per-agent attribution for free in the audit pillar.

D5 — What does a transaction result look like over the wire? (absorbs Slice 7.5)

The question

"A library caller gets a Python DryRunResult or transaction object and can introspect the journal directly. An MCP client gets a JSON message. What does the result contain over the wire — and this is where the deferred state-diff finally becomes necessary."

A library caller can walk the in-memory journal and query the adapters directly. A wire consumer can't — it gets whatever the gateway serialises. So the gateway is the consumer that finally forces the structured state diff Slice 7 deferred: the operator-facing answer to "what would my transaction change" has to be pre-computed before it crosses the wire, because the client can't reach back into the adapters to compute it itself.

The absorbed Slice-7.5 work — StateDiffable sub-protocol

Adapters opt into computing a structured before/after diff; the gateway serialises it into the wire result.

class StateDiffable(Protocol):
    def diff(self, snapshot: SnapshotHandle, current: Any) -> dict:
        """Adapter-specific before/after diff."""

# SQLiteAdapter, FilesystemAdapter implement it. HTTPAdapter doesn't
# (its diff is the would_have_fired list). Result shape, per Slice 7.5 D3:
result.state_diff = {
    "sql": {"rows_added": [...], "rows_modified": [...], "rows_deleted": [...]},
    "fs":  {"files_added": [...], "files_modified": [...], "files_deleted": [...]},
}

This is exactly the design the (now-retired) Slice 7.5 aid locked: a StateDiffable sub-protocol mirroring TransactionalResourceAdapter, a standardised outer shape (per-resource dict) with adapter-specific inner payload. It lands here because here is where it's load-bearing — the wire consumer has no other way to learn what changed.

Wire serialisation — dataclass → JSON

The result objects (transaction summary, DryRunResult) serialise to JSON via the existing strict_json_default machinery.

Slice 1's strict_json_default already handles bytes (base64), datetime (ISO), dataclasses (asdict). The wire result reuses it — the gateway adds no new serialisation vocabulary, it leans on what the audit journal already uses. Effects, verdicts, conflicts, and state-diffs all serialise through the one function.

No new serialisation layer — the wire format is the audit format.

Round-trip honesty: the same encoder that persists the journal encodes the wire result.

Locked — StateDiffable sub-protocol + strict_json_default wire encoding

The absorbed Slice-7.5 state-diff lands as a StateDiffable sub-protocol on adapters/base.py; SQLiteAdapter and FilesystemAdapter implement it; HTTPAdapter does not. Results cross the wire as JSON via the existing strict_json_default encoder. The gateway computes the state-diff before serialising because the wire consumer cannot reach back into the adapters — which is the honest reason the Slice-7 deferral resolves here and not earlier.

D6 — Two gateway connections both want to commit. Who wins?

The question

"Slice 4's JournalRegistry arbitrates concurrent transactions in one process. The gateway is a separate process — and could host several agent connections at once, or there could be several gateway processes. How do concurrent commits arbitrate now?"

Option A — in-process registry only

Keep Slice 4's JournalRegistry; concurrent connections inside one gateway process arbitrate; cross-process is unhandled.

No new work.

If the operator runs two gateway processes (or a gateway plus a library script) against the same database, the in-process registry can't see across the boundary — exactly the lost-update Slice 4 set out to prevent, reintroduced at the gateway layer.

Option B — reuse Slice 4's filesystem-shared meta-connection

Slice 4 already ships cross-process arbitration for SQL via the autocommit meta-connection on the shared DB file. The gateway leans on it; no new mechanism.

Slice 4's SQLiteAdapter meta-connection was built precisely so multiple processes against the same on-disk SQLite file see each other's committed version bumps. The gateway is just another process hitting the same file — the commit-time check_conflicts diff already reads through the meta-connection, so cross-gateway lost-updates are caught by the machinery that already exists.

No new arbitration mechanism — Slice 4's design anticipated this exact case ("multiple Python processes against the same SQLite file").

The cross-process story the interception-axis discussion wanted lands using infrastructure already merged and tested.

Filesystem adapter cross-process arbitration is weaker than SQL's (no meta-connection equivalent). Mitigation: document the boundary — SQL is cross-process-safe; FS cross-process concurrent writes are a known gap, flagged for a follow-up.

Locked — Option B

Reuse Slice 4's filesystem-shared SQLite meta-connection for cross-process arbitration of SQL transactions. The gateway is "just another process against the same file" — the machinery Slice 4 built for that case carries the gateway. Filesystem cross-process concurrency is documented as a known gap (a follow-up, not a Slice 8 blocker — single-gateway deployments don't hit it).

Open — the rest of the MCP surface

The question

"MCP has resources, prompts, sampling, completions. Slice 8 ships tool-call only. When (if ever) do the others matter for Pherix?"

The honest answer is "probably never, in core/ terms" — but worth naming the boundary:

  • Resources (an MCP server exposing readable data to the model) — Pherix could expose the audit journal as an MCP resource, so an agent could query its own transaction history. That's an audit-pillar feature, not a gateway-core one. Defer to the audit pillar.
  • Prompts / sampling / completions — these are about feeding and steering the model. Pherix transacts effects; it does not run or steer models (CLAUDE.md: "Pherix does not call LLMs"). These are out of scope permanently, by design — not deferred, declined.

Recommendation — tool-call only; audit-as-resource is an audit-pillar idea

Slice 8 ships the tool-call subset. Exposing the journal as an MCP resource is a genuinely interesting audit-pillar feature (let an agent read its own provenance) — parked there, not here. The model-steering MCP features are declined permanently, not deferred: they're outside Pherix's wedge by definition.

Closing — the discipline, paid back or exposed

Every prior slice added a capability by traversing the journal a new way. Slice 8 adds nothing to the journal — it adds a second mouth that speaks to the same engine. The measure of success is a diff stat: if Slice 8 lands with core/ essentially untouched (one additive client_id audit column, the absorbed StateDiffable sub-protocol, nothing else), the front-end-agnostic discipline held for eight slices straight. If it needs to reach into core/ and reshape the engine, the discipline was a story we told ourselves.

The bet from Slice 1 — "build the library first, the gateway bolts on later with no rewrite" — gets settled here. The absorbed state-diff is the one genuinely new capability, and it lands in Slice 8 for an honest reason: the wire consumer is the first consumer that can't compute the diff itself.

One engine. Two front-ends. One journal. If this slice is small, the architecture was right.
