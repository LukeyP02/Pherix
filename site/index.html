The problem it kills

An agent calls a tool and the effect just happens — irreversibly, immediately, unaudited. A wiped table, a payment sent to the wrong account, a file clobbered. Pherix sits at the tool-call layer and makes that mistake contained instead of catastrophic: reversible effects roll back, irreversible ones are gated behind explicit approval, and the whole run is on the record.

What you get, in one breath

Undo the reversible

DB and filesystem writes journal live and roll back via the backend's own savepoints and copy-on-write. The backend does the undo — exact, not best-effort.

Gate the irreversible

An un-undoable effect — a charge, an email — is staged, not fired. It only happens at commit, and one without a compensator blocks until a human approves it.

Audit everything

Every side-effecting tool call is an append-only journal entry. The journal is the audit log — every action, its arguments, and its outcome, on the record.

Where it earns its place

Ops / SRE agent

An agent operating on a production database — let it act, and roll the whole step back the moment something looks wrong. Nothing persists until you commit.

Payments agent

A charge is staged behind the gate; a refund is its registered compensator. A failed run unwinds cleanly — the money moves only when the whole step is sound.

Data agent

Bulk writes across SQL, object storage, and search indexes journal as one transaction — commit them together, or roll them all back as one.

How it fits

Pherix is a library. You pip install it, wrap your agent's tool-call layer, and keep your existing agent loop and model provider. It does not run agents or call LLMs — it makes their actions behave like database transactions. Pre-release and self-hosted; no console, no hosted service.
