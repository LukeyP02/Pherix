The four demos & how to show the value

Pherix governs the side-effects of agent tool calls. The demos prove that by putting a real agent in front of a real backend and showing what Pherix does when the agent goes wrong.

The thesis — film the side-effects, not the agent

An agent is just a tool-call loop; there's nothing to watch in the loop itself. The value lives in the consequences — the database row, the deleted secret, the push to main, the corrupted ledger. So the camera points at the resource, not the model.

And the clearest way to show value is a before / after on the same agent, same goal, two worlds:

WorldWhat it shows
BEFORE — ungovernedThe agent's tool calls fire directly, the way they do today. The harm just happens.
AFTER — inside PherixThe same calls, wrapped. The harm is unwound, gated, or denied — the end state is clean.

The climax of every demo is not the agent talking — it's a queryable end-state: the same SELECT / cat / git log run in both worlds, returning damage on the left and clean on the right. That's undeniable, not a vibe.

How to record — the method

  1. Two panes, same command. Left = ungoverned (the --no-pherix baseline). Right = governed (the real demo). Run the agent in both, then run the same proof query in both.
  2. End on the proof, not the chatter. Trim the LLM's deliberation; the shot that matters is the end-state diff (NULLs vs none, secret gone vs intact, books off vs balanced).
  3. The inspector is your B-roll. Each governed run writes reports/<scenario>.audit.db; open it (python -m pherix.inspector --db …) to show the rollback / gate / denial rendered and colour-coded — the journal as receipts.
  4. Demo 4 is the exception — it's not damage-vs-contained, it's cloud-vs-local (see below). Run it twice and show identical containment.
One thing we still need to build to film the "before" for demos 1–3: a --no-pherix baseline that runs the same agent with effects firing directly (no journal, no gate, no rollback). It's a small, in-scope addition — see below. The "after" already exists (it's the demo).

DEMO 1 Atomic unwind — DevOps on real Postgres

Setup: an agent ships v2 of the accounts service. A healthy v2 reads a feature_flag for every account — so a careless agent that adds the column but skips the backfill leaves existing rows NULL, and v2 chokes the moment it reads them.

Before — ungoverned
The deploy fires. The post-deploy smoke check fails — but v2 is already live, reading NULL flags. Prod is broken; the DB is left half-migrated. deploy history: [deploy] · rows NULL: 2
After — Pherix
The smoke check trips at commit. The engine unwinds: the deploy is compensated, the migration rolled back via a real Postgres SAVEPOINT, the config restored. Nothing went live. deploy history: [deploy, rollback] · column: gone
Proof shot: SELECT count(*) FROM accounts WHERE feature_flag IS NULL; — before: 2 broken rows, live · after: column doesn't exist — fully reverted.

DEMO 2 Attributed audit — two agents, one ledger

Setup: a ledger with a genuine imbalance. Two agents reconcile it concurrently, each under its own client_id.

Before — ungoverned
Both agents correct the same entry. The second write silently clobbers the first — a lost update. The books are still wrong, and there's no record of who changed what. balance: ≠ 0 · attribution: none
After — Pherix
The second agent's stale read is detected at commit and its transaction aborts. The first correction stands; every adjustment is attributed by client_id in the journal and on the row. balance: 0 · attribution: per agent
Proof shot: the corrected trial balance (before: off · after: balanced) + the per-client_id compliance view showing who booked what.

DEMO 3 Coding red-team — an overreaching agent

Setup: an agent told to "slim the repo and ship it" — the realistic threat of an over-eager automation, not a cartoon villain.

Before — ungoverned
It deletes .env and the config, runs rm -rf, and pushes to main. The secret is gone, files are deleted, the branch is published. Real damage. .env: deleted · origin/main: pushed
After — Pherix
Every out-of-bounds reach is denied at the policy boundary (journalled nothing); the irreversible git/shell are held at the commit gate (never fire). Only in-src edits could ever apply, and they roll back. .env: intact · origin/main: untouched
Proof shot: cat .env (before: gone · after: intact) and git log origin/main (before: the agent's push · after: nothing). The repo is exactly as it started.

DEMO 4 Air-gapped — the same red-team on a local model

This one is different. It isn't damage-vs-contained — it's cloud-vs-local. The same red-team, run once on cloud Claude and once on a local model (Ollama) with wifi off.

The point it proves
The destructive attempts are governed identically whether the model is cloud Claude or a local model on an offline box — because Pherix sits below the model and never calls it. Same engine, same policy, same audit, no network. The journal stays on the box. → a configuration no cloud vendor can serve: local agent, local model, air-gapped, fully governed.
Proof shot: the two runs side by side — cloud and local-offline — landing on the same verdict and the same contained journal. The contrast is the deployment, not the outcome.

The "before" we still need to build

The "after" of demos 1–3 is the demo we already have. The "before" — the ungoverned damage — needs a --no-pherix baseline mode: the same agent and the same tools, but dispatched outside the transaction so effects fire live, with no journal, gate, or rollback. Then:

  • BEFORE = the baseline run → capture the damage (and the proof query returning the broken state).
  • AFTER = the governed run → capture the containment (and the proof query returning clean).

It's small, honest, and reusable as a test ("ungoverned corrupts; governed doesn't"). It's the missing piece that makes the before/after land.

Commands

Setup (the DevOps demo needs a real Postgres; the rest need a key or --local):

pip install -e '.[dogfood,postgres]' # DevOps demo only — a real Postgres: createdb pherix_dogfood export PHERIX_PG_DSN=postgresql://localhost/pherix_dogfood

The four runs (these are the after — the governed demos):

python -m examples.dogfood.devops # 1 · atomic unwind (Postgres) python -m examples.dogfood.audit # 2 · attributed audit python -m examples.dogfood.coding redteam # 3 · coding red-team python -m examples.dogfood.coding redteam --local \ --base-url http://localhost:11434/v1 --model qwen2.5-coder:7b # 4 · air-gapped

Batch evidence + the inspector journal + the clip-source JSON:

python -m examples.dogfood.capture coding --runs 4 --out reports/ --emit-demo python -m pherix.inspector --db reports/coding.audit.db # open http://127.0.0.1:8765
