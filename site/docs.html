Docs.

Pherix records every side-effecting tool call, in order, as your agent runs. Commit, rollback, and audit are all just reading that record — forward, backward, or as-is. The whole model fits on one page.

Overview

Pherix is a library. You wrap your agent's tool-call layer and keep your existing agent loop and model provider — the loop never knows it's inside a transaction. Reversible effects run live and roll back through the backend's own snapshots; irreversible ones are held back and only fire at commit, gated for approval when they have no safe undo.

Install

Pre-release 0.1.0:

pip install pherix

Or from source: git clone https://github.com/LukeyP02/Pherix && cd Pherix && pip install -e .

Quickstart

Declare each side-effecting tool with the resource it touches, then wrap the run. Leaving the block cleanly commits; calling rollback() undoes everything.

import sqlite3
from pherix import AuditJournal, SQLiteAdapter, agent_txn, tool

@tool(resource="sql")
def insert_user(conn, name, role):
    conn.execute("INSERT INTO users (name, role) VALUES (?, ?)", (name, role))
    return name

conn = sqlite3.connect("app.db", isolation_level=None)
adapters = {"sql": SQLiteAdapter(conn)}
audit = AuditJournal.in_memory()

with agent_txn(adapters, audit=audit) as txn:
    insert_user(name="ada", role="engineer")
    # caught a problem? txn.rollback() — nothing was saved.
# left the block cleanly → commit. The writes are now durable.

The full end-to-end walk-through — including irreversible effects and the approval gate — is on Get started.

The record

Every side-effecting tool call is recorded as one entry, in the order it happened. Commit reads the record forward and applies each effect; rollback reads it backward and undoes each one; replay re-runs it against fresh state; dry-run runs it against a snapshot, then discards. Audit needs nothing extra — the record is the audit log.

The two lanes

Each effect goes down one of two lanes, decided by whether its resource can be snapshotted. These are two different ways to undo, not one.

Reversible → snapshot, then restore

Runs live. The backend snapshots before the effect and restores on rollback — a database SAVEPOINT, a filesystem copy. The backend does the undo: exact, and it needn't know what the action meant.

Irreversible → hold back, then approve

Can't be snapshotted, so it's held back and only fires at commit. With a registered opposite action (a refund for a charge) it auto-commits and that opposite runs on rollback; without one it waits for a human to approve.

Adapters

An adapter knows how to snapshot, run, and restore one kind of resource. Tools say which resource they touch; Pherix routes each effect to the right adapter. Backend drivers are optional and loaded only when used, so the core has no dependencies.

Honest about what it can't undo: an HTTP call can't be snapshotted, so Pherix holds it back for approval rather than pretending it can be rolled back.

Policy

Allow and deny rules, plus caps — total spend, count limits — and rules that read live external state before deciding. Policy is a clean slate you write your own rules against. State can change while an effect waits, so the same rule is checked twice — once when the effect is staged, once again at commit — and the effect can't slip through in the gap.

Public API

The handful you actually touch:

tooldeclare a side-effecting tool
SQLiteAdapter …the per-resource adapters

AuditJournalthe recorded run / audit log
approve_irreversibleclear a held effect to fire
replayre-run the record against fresh state
dry_runrun against a snapshot, then discard
