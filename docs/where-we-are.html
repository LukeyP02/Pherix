Premise — you have not built junk

Over-vibing is the natural cost of build-pace momentum. It is not that the code is sloppy; it is that six slices have landed faster than the mental model has had time to consolidate around them. The shape of the system has been hidden by the speed of laying it down. What follows re-exposes that shape.

Pherix in one sentence. A Python library that makes the side-effects an AI agent has on the outside world — SQL writes, filesystem writes, HTTP calls — behave like database transactions. Atomicity, isolation, capability enforcement, durability, over external resources. The operator wraps their agent's tool-call layer with agent_txn(...) and keeps the rest of their stack unchanged. Pherix is not an agent framework; it does not call LLMs; it does not orchestrate prompts. It sits at the tool-call boundary and enforces.

The reassurance

Six slices compose into a coherent thing. The architecture has not drifted — if anything, the discipline has tightened with each slice, because each new capability has had to fit through the same engine. The next three pages walk that engine.

What shipped — six slices, one engine

Each slice adds one capability to the same engine. The engine itself — the journal of effects and the adapter protocol — was built in Slice 1; everything since is a different way of traversing it.

Slice 1 — the engine itself

ad4e9c0

The failure mode: an agent's tool call fires immediately, irreversibly, unaudited — no transaction boundary exists.

with agent_txn(adapters): ... intercepts every registered tool call and records it as an Effect in an ordered, append-only journal held by the Transaction. SQL writes route through a SAVEPOINT; rollback() issues ROLLBACK TO SAVEPOINT and the row is gone; commit() finalises and the row stays. The journal is the spine of everything that follows.

Forward fold — adapter.apply per effect

Slice 2 — the engine is not SQL-specific

c926f97

The failure mode: agents touch the filesystem too, and a SQL-only transaction boundary cannot cover them.

A FilesystemAdapter implements the same (snapshot, apply, restore) triple via copy-on-write of touched paths. One transaction can now carry SQL writes and filesystem writes together; on rollback, both unwind; on commit, both land. This is the slice that proves ResourceAdapter is a real abstraction and not a SQL-shaped hole pretending to be general.

Same forward fold — second adapter under the same protocol

Slice 3 — irreversible things

a13018b

The failure mode: a credit-card charge cannot be rolled back — it has already happened.

Irreversible tools are staged: the effect is recorded in the journal with status=STAGED and the agent receives a placeholder StagedResult rather than a real return value. The actual call only fires at commit() time. If the txn rolls back, it simply never fires. If a tool registers a compensator — a semantic left-inverse like refund after charge — it auto-commits; rollback runs the compensator. With no compensator, the txn lands in STUCK and waits for human review. Pherix is honest about what it cannot undo.

Forward fold with deferred terminator — staged-fire loop runs at commit

Slice 4 — concurrent agents on the same resource

0fab344

The failure mode: two agents read the same row, both write back, and one of them silently clobbers the other.

Every effect records the keys it read and wrote, with versions. At commit time the runtime diffs this txn's read/write set against what other live transactions have done in the meantime; if anyone moved a key this txn depended on, IsolationConflict fires. Three resolution policies are available: Abort (give up), Retry(N) (re-run the agent body), and Serialize (block until the conflict clears). This is MVCC-style isolation — the database concept — lifted to the level of agent tool calls.

Diff of read/write key-sets across journals

Slice 5 — replay

9ff69a6

The failure mode: yesterday's transaction did something interesting; today there is no way to ask "what would running it again, against fresh state, actually do?"

The journal becomes executable. pherix.replay(txn_id, adapters) re-fires the recorded effects against a fresh adapter target. Verify mode asserts the new results match the old — a determinism check on the agent's tool layer. Reconstruct mode rebuilds world state from scratch by re-applying every reversible effect in order — disaster recovery from the audit log alone. Irreversibles are not re-fired; their original results are replayed-as-data.

Forward fold against fresh state — with equality check

Slice 6 — real policy

c2318d5

The failure mode: an allow-list cannot express "this agent may spend at most £100 today" or "this agent may not publish to enterprise tenants."

Policy is now a predicate fold over (effect, ctx). The operator registers Python callables with @policy.rule that return Allow or Deny(reason). Cap.count and Cap.sum fold the journal-so-far to enforce spend and frequency caps. Rules fire at stage-time (fail fast) and again at commit-time (because the world may have moved between the two evaluations — the TOCTOU problem, where the truth of a predicate depends on when you ask). Crucially, the commit-time bracket runs before the staged-fire loop, so a denial at commit pre-empts every irreversible. The strongest containment the architecture can offer.

Predicate fold over the journal — evaluated twice

The axiom — one engine, six operations

The unifying claim

Every capability Pherix offers is a different traversal of the same journal.

The journal is an ordered, append-only sequence of Effect records. The adapter protocol exposes a (snapshot, apply, restore) triple per resource. Once those two primitives exist, every capability the system needs is just a different walk over the sequence, possibly under a different terminator. This is not retrofitted phrasing — it is the design axiom that made the build coherent.

Four walks over one sequence. Two terminators (commit, rollback). Two predicate folds (isolation diff, policy predicate). The whole architecture.

Spelled out as a table, the capability-to-operation map is exact:

Capability Operation on the journal Slice
Commit forward fold → adapter.apply per effect 1
Rollback backward fold → adapter.restore or compensator 1, 3
Multi-resource txn forward fold across N adapters under one bracket 2
Irreversible staging forward fold with terminator deferred to commit 3
Isolation diff of read/write key-sets across live journals 4
Replay forward fold against fresh state · equality check 5
Policy predicate fold over (effect, ctx), evaluated twice 6
Dry-run forward fold then discard (rollback terminator) 7 · in worktree

Read down the middle column. Every entry is a fold or a diff over the same sequence. The system is not a pile of subsystems — it is one engine with many uses, now reachable through two front-ends (the Python library and the MCP gateway). That is what makes Pherix coherent rather than a pile of features, and that is the mental model the over-vibing has obscured.

Closing

Eight slices and a real-agent dogfood suite compose into a coherent thing. The architecture has held under every capability asked of it, because each capability has had to fit through the same engine — and when a real agent finally drove it against a real backend, the bug it found was a missing test, not a broken design.

The journal is the asset that compounds — every slice writes to the same shape, and every new walk over it is cheaper than the last. The over-vibing did not corrupt the build; it just outran the moment to look back at it. This page is that moment.

If a future feature does not reduce to a fold or a diff over the journal, it is the wrong feature. So far, everything has.
