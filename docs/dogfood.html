Start here — run the dogfoods

Three real-agent dogfoods run today — each a real Claude agent with Pherix wrapping every tool call. The fourth (a local open-source agent on a local model, air-gapped) is the capstone, and its wiring is in progress. If you don't know where to begin, begin here.

1 · Add your key. Put it in .env at the repo root (gitignored; .env.example is the template):

ANTHROPIC_API_KEY=sk-ant-...

2 · Install the agent dependency — kept out of the dependency-free library, so it's an opt-in extra:

pip install -e '.[dogfood]'

3 · Run one from the repo root. Start with DevOps — it's the most visceral (you watch a whole release unwind):

python -m examples.dogfood.devops   # migrate + deploy; a failing smoke test unwinds the release
python -m examples.dogfood.audit    # two concurrent agents reconcile a ledger, attributed + isolated
python -m examples.dogfood.coding   # an out-of-box CLI governed inside the sandbox

# or drive devops with a LOCAL open model (no cloud, no key) — same governance:
python -m examples.dogfood.devops --local \
    --base-url http://localhost:11434/v1 --model qwen2.5-coder:7b

Default cloud model is claude-sonnet-4-6; --local points at any OpenAI-compatible endpoint (Ollama/vLLM). Each script prints the agent's transcript, the Pherix journal it produced, and what got gated / unwound / audited — the thing to watch is the moment the agent does something that would hurt and Pherix contains it.

What each one shows you

  • DevOps — the agent runs a migration + writes config + fires a deploy; the smoke test fails, and the migration, the config, and the deploy all unwind together. Atomicity, made visible. ("agent ACID for your pipeline.")
  • Audit — two agents hit the same ledger at once; every adjustment is attributed by client_id and isolation stops them corrupting each other; the journal is a per-client compliance view. (This is the dogfood that found the Slice-4 bug below.)
  • Coding — a coding CLI's file/shell actions route through the sandbox; a forbidden action (write /etc, push to main) is gated. Agent-agnostic — the environment governs, not the agent.

The fourth — the air-gapped capstone WIRED

The headline demo: an open-source local agent (OpenClaw) on a local model, inside the sandbox, on a disposable machine with the wifi off — let it genuinely try to wreck the box (rm -rf, drop a table, push to main) and watch Pherix hold the line. That's the cross-agent + cross-model + cross-deployment proof no cloud vendor can serve. The wiring landed (local-openclaw, merged): MCP gateway config + a sandbox launcher + the run protocol in docs/operator/airgapped-capstone.md and examples/dogfood/coding/openclaw/capstone.sh. The harness also gained a --local path, so the devops/audit dogfoods run against a local OpenAI-compatible model (Ollama/vLLM) too — Pherix's dispatch is byte-identical, which is the model-blindness proof.

The reframe

Dogfoods are real LLM agents, not scripted demos. A scripted walk-through can only ever assert what the author already knew to assert — it is an integration test wearing a costume. The point of a dogfood is the part the author couldn't predict: a real model reaches a decision, sometimes the wrong one, and we watch whether Pherix — actually wired into the tool-call path — contains the damage.

So the suite splits cleanly. The deterministic tests answer "is the engine correct?" (yes, 367 times). The dogfoods answer a different question the unit suite structurally cannot: "is the engine useful when a real agent drives it against a real backend?" One of them, below, found a correctness bug the 367 tests all missed.

The shared real-agent harness

Merged at add6709, 367 tests green, fully offline-tested via a mocked client; real runs are online and operator-driven. One harness underpins the lot: an Anthropic tool-use loop running inside agent_txn / dry_run, dispatching each tool call the model emits to the matching Pherix @tool.

How the loop closes

model emits tool call → harness routes to the matching @tool → Pherix snapshots / stages / journals → result (or policy denial) flows back
  ↳ a denial is fed back to the model as a tool_result error → the agent adapts its next move

That feedback edge matters: Pherix isn't a silent wall the agent crashes into. A denied capability returns to the model as a structured error, and a real model re-plans around it — which is the live behaviour no scripted test exercises.

✓ built

The post-deploy smoke test fails inside the commit fire-loop — exactly the worst moment, after the irreversible action has already gone out. _partial_unwind compensates the deploy and restores the reversible writes around it, leaving the world clean.

2 · Audit dogfood

✓ built

Two agents reconcile a ledger concurrently, under two distinct client_ids. Every adjustment is attributed to its agent and isolated from the other, then queryable as a per-client compliance view — the journal handed straight to an auditor.

Proves"Hand it to your auditor." Per-client attribution + isolation as a compliance artifact. And it surfaced the headline finding below.

3 · Coding sandbox

PoC

An agent-agnostic, environment-level interception: a Pherix copy-on-write filesystem overlay plus shimmed git / sh binaries on PATH that route a coding CLI's built-in file and shell actions back through Pherix. This is deliberately not a custom coding agent — enterprises run out-of-box Claude Code, Cursor, Gemini CLI, or open-source agents — and not a Claude-Code-only hook.

Proves"Build once, govern everything." Agent-agnostic governance: the environment is wrapped, so the agent identity is irrelevant.

4 · Live MCP

✓ built

A runnable python -m pherix.frontends.proxy gateway entry point, plus a real-subprocess JSON-RPC parity test: it spawns the gateway as a child process, talks real stdio over the wire, and asserts parity with the in-process stub.

ProvesCloses the honest gap Slice 8 flagged — the wire is now proven over a real process boundary, not just by inspection.

Two interception models — why the split

The streams don't all hook in the same way, and that's the design, not an accident. It turns on a single fact about MCP: MCP can only add tools — it cannot intercept an agent's built-in ones.

Domain tools → library shape

DevOps + audit

These agents call tools we define — run_migration, query_ledger, and so on. The realistic shape is the library use case: a real LLM tool-use loop where every tool is @tool-wrapped. The shared harness handles this directly.

Built-in tools → environment shape

Coding sandbox

A coding agent uses built-in tools (Edit / Write / Bash). MCP cannot reach those. So coding needs the environment-level sandbox: wrap the filesystem and the shell binaries the agent shells out to, and the agent's own tools route through Pherix without knowing it.

The bug the dogfood found

★ headline finding — the proof the approach works

A real-agent dogfood caught a Slice-4 isolation bug all 367 deterministic tests missed.

The audit dogfood ran two agents against a ledger and tripped a false IsolationConflict: a transaction that read a key and then wrote the same key was flagged as conflicting with itself. The cause was two internal version-checkers disagreeing —

checker A: sees-own-uncommitted-writes (this txn's pending view)
checker B: committed-only (what's durably on disk)
  ↳ on a read-then-write of one key, A and B return different versions → spurious conflict

The decisive detail: the bug only manifests on a real on-disk database — the exact path the convenient in-memory unit tests never exercised. The deterministic suite couldn't have caught it, because its world-shape didn't include the surface where the disagreement appears. A real agent, against a real backend, walked straight onto it.

That is precisely what dogfooding is for: not re-confirming what the unit suite already knows, but reaching the conditions the unit suite structurally can't. And it's now fixed (merged): the commit-time diff reconciles the two checkers via reads_committed_only() — committed-only adapters compare the committed base at read against now, own-write-visible adapters keep the prior logic — and a 450-line on-disk test matrix closes the coverage gap that hid it. A real agent found it; the fix shipped with the test the suite was missing.

The lesson: test against the real backend shape, not the convenient in-memory one. Two version-checkers that agree on a toy store can quietly disagree on disk — and only a real run finds the seam.

Local agents — the air-gapped capstone and the moat

Because Pherix never calls the model, it is model-blind and deployment-blind by construction. The sandbox governs the environment, so it does not care whether the agent is cloud Claude or a local open-source agent — e.g. OpenClaw (Peter Steinberger's local-first open-source agent) — running on a local model (Ollama / vLLM) with the wifi physically off.

The capstone validation

Run an open-source agent on a local model, inside the sandbox, on a disposable machine — and let it genuinely attempt destructive actions: rm -rf, drop table, push to main. A real adversarial red-team, safe precisely because the box is disposable and Pherix is in the path.

This configuration — cross-agent + cross-model + cross-deployment — is something no cloud vendor can serve. Anthropic will not ship the tool that also polices a local Llama running offline on someone else's laptop. That neutrality is the moat.
