The four axes

To make an action transactional you must answer four questions. Pherix is exactly those four answers, built on one shared journal — and there is no fifth.

AxisThe questionWhat fills it
InterceptionHow do actions reach us?The Python library, the TypeScript SDK, the MCP gateway.
PolicyWhat do we allow?Allow/deny, caps, the human gate, world-state rules.
AdaptersHow do we undo what can be undone?Per-resource snapshot / apply / restore.
CompensatorsHow do we undo what can't?The staged/gated lane + vetted semantic inverses.

1. Interception

Pherix wraps your tool-call layer so effects flow through the journal. You decorate tools with @tool and run your agent inside agent_txn; the agent code is unchanged and unaware. The same core is driven by three front-ends — the Python library, the TypeScript SDK, and an MCP gateway for other-language agents.

2. Policy

A policy decides what an agent is allowed to do. Rules can allow or deny by tool and arguments, cap aggregates (e.g. "no more than $100 charged per transaction"), and require a human to approve sensitive effects. Crucially, policy is evaluated twice:

  • at stage-time, to fail fast; and
  • again at commit-time, because the world may have changed between the two — the safety property that closes time-of-check/time-of-use gaps.
from pherix import Policy, Allow, Deny, Cap

policy = Policy(rules=[
    Deny(tool="charge_card", when=lambda a: a["cents"] > 50_000),
    Cap.sum(tool="charge_card", arg="cents", limit=100_000),
])
with agent_txn(adapters, policy=policy) as txn:
    ...

See the Python API for the exact rule surface.

3. Adapters — undoing the reversible

An adapter is a triple (snapshot, apply, restore) over a class of resource. The backend does the undo: the SQL adapter issues a real SAVEPOINT and rolls back to it; the filesystem adapter keeps a content-hash backup of touched paths; object stores capture the before-image of each touched key. Restore is exact and needs no knowledge of what the action meant. Browse the adapters →

4. Compensators — undoing the irreversible

Some effects can't be snapshotted — you can't un-send an email or un-charge a card by reverting state. Pherix is honest about this. An irreversible effect is staged, not run live, and fires only at commit:

  • If a compensator is registered (a semantic inverse — charge→refund, invite→revoke), the effect auto-commits and the compensator is used on rollback.
  • If none is registered, commit gates: it blocks until an explicit approve_irreversible() — a human, or a higher-trust policy.

A rollback before commit means the irreversible effect simply never happened — that is the whole point. See the compensator catalog →

Adapters and compensators are two different undo mechanisms. Adapter restore is state rollback (snapshot, then revert — exact). A compensator is a semantic inverse (run an opposite action — approximate, needs the meaning). The boundary is whether the resource can be snapshotted.

Everything else is a consequence

The four axes ride on one append-only effect journal — and the rest falls out of traversing it:

CapabilityJournal operation
Commitfold forward, apply each effect
Rollbackfold backward, restore / compensate each
Dry-runfold forward against a snapshot, then discard
Isolationdiff this transaction's read/write keys against the journal
Auditread the journal — it is the log
from pherix import dry_run

with dry_run(adapters) as ctx:        # fold forward against a snapshot…
    plan_some_changes()
# …then the snapshot is discarded — nothing was committed. Inspect ctx for the plan.
