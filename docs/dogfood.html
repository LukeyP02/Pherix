Pherix · dogfood

Four verticals to prove Pherix in production.

The reframe: these aren't Pherix's verticals. They are test surfaces for Pherix-as-infrastructure — each prototype evaluates Pherix at a different point in capability-space. Build them as a solo founder, learn where the engine carries us through and where it hits a wall, and feed every failure mode back into the roadmap as a customer-driven feature request.

The reframe

The vertical-AI play is an integration business: compliance, distribution-via-existing-network, accumulated industry knowledge. The moat is non-portable; the ceiling is set by how much industry context the founder brings. That isn't Pherix's shape and it isn't the solo-founder leverage curve we're optimising for.

But a prototype inside a vertical, where the agent uses Pherix internally, is a different proposition entirely. The thing built is not the vertical product; it is the evidence that Pherix carries the agent through real production-shaped conditions. The dogfood signal is «where does Pherix carry us through, and where does it hit a wall we have to fix or document?»

Two implications.

  1. Pick prototypes that stress different parts of Pherix's roadmap. Three SQL-shaped variations of the same agent teach Pherix nothing new; three different adapter mixes plus audit profiles teach Pherix four times.
  2. Each prototype's failure mode is a customer-driven Pherix follow-up. When the SRE bot needs result-piped compensators because kubectl scale returned a count the compensator needs, that's no longer a speculative cleanup — it's a feature request from the dogfood itself. The dogfood matrix is product discovery.

Reading the matrix: the four columns cover the Pherix capability-space with low overlap. DevOps and reconciliation are the two heaviest stress tests overall — DevOps on the irreversible / isolation / audit axes, reconciliation on the SQL / isolation / audit axes. Migration is the only one that stresses the filesystem adapter heavily, which is why it earns a slot even though it overlaps lightly elsewhere. Filing is the smallest dogfood by Pherix-surface, but it's the only one where audit-grade artifacts are the user-facing product, not a side effect.

The four picks

1. DevOps / SRE incident-response agent

strongest pick if you build only one — build this

An agent that runs runbooks against prod infrastructure — kubectl actions, database failovers, log queries, monitoring acknowledgements, scale operations. Triggered by alerts or invoked by an on-call human; produces a clear audit trail of every action it took. Co-exists with human operators (the on-call rotation) so multi-agent concurrency is the norm, not the exception.

Why this is the strongest pick. Every action is irreversible (Slice 3's lane is the entire game). Most actions need pre-approval gates — «who killed that pod» is exactly the question a regulator or post-incident review asks. Compensators are real: a failover has an undo, a scale-down has a scale-up. Multi-agent concurrency is the natural shape: the on-call rotation plus automated alerts interleave on the same infra, exercising Slice 4 isolation. And the buyer for this prototype (SRE leads, platform teams) is also Pherix's eventual buyer — meaning success in this dogfood doubles as customer development.

Solo-founder buildable. kubectl + Datadog / Grafana / PagerDuty APIs + an LLM with tool-use + Pherix wrapping the writes. The agent half is straightforward; the load-bearing piece is the audit trail and the gate UX, which is precisely what Pherix already does.

Pherix surfaces exercised: Slice 3 (every action), Slice 4 isolation (concurrent human + automation), Audit pillar (the product), Cross-resource txn (SQL state alongside HTTP infra calls).
Failure modes to expect that become roadmap items: result-piped compensators (the kubectl scale output needs to flow into the compensator — Slice-3 FU#1); compensator execution as a separate audit row (Slice-3 FU#2 — «when did the scale-back actually fire» is a real question here); Serialize across hosts (the gateway dependency from Slice-4 FU#4).

2. Devtools migration / dependency-upgrade fleet bot

build second covers FS + cross-resource that DevOps doesn't

"Upgrade Stripe SDK across 47 repos", or "migrate Mongoose v6 to v7 across the monorepos", or "apply this codemod across the org's package manifests and open PRs". The agent reads the codebase, plans the diff, writes the changes, opens the PR, monitors CI, and on failure rolls everything back to a clean state.

Why this earns a slot. It's the only one of the four that stresses the filesystem adapter heavily — copy-on-write earns its keep precisely when a migration step on file 32 of 47 needs to roll back. The cross-resource transaction (D3 from Slice 2) is the point: a half-migrated repo plus a closed PR plus a deleted branch is the worst-of-all-worlds state Pherix exists to prevent. The agent writes to filesystem (Slice 2), opens PRs (Slice 3 HTTP), and the success criterion is that those two commit-or-rollback together.

Solo-founder feasible. The GitHub API is well-documented; codemod tooling exists per ecosystem; the agent's job is plan-then-apply-then-watch-CI. Visible win is concrete and shareable: «our upgrade bot never leaves a repo in a broken half-state.»

Pherix surfaces exercised: Slice 2 (FS — primary write surface), Slice 3 (HTTP — PR opens, branch pushes), Cross-resource txn (the FS+HTTP commit-together story is the whole point).
Failure modes to expect: needing to capture the PR number from the HTTP fire as a journalled result so a compensator can close it (this maps directly to Slice-3 FU#1 result-piped compensators again — different vertical, same feature request).

3. Back-office finance / reconciliation agent

build third covers the boring-load-bearing SQL axis

Match transactions across Stripe / accounting ledger / bank feed; flag and resolve discrepancies; post journal entries; produce the monthly close artifact. Boring on purpose — that's where the audit-pillar value lives, and where the competitors aren't paying attention.

Why this earns a slot. SQL-heavy (Slice 1 is the workhorse), strict idempotency requirements (Slice 3's effect_id earns its weight when daily reconciliation re-runs the same matches against the same source data), audit is a regulatory artifact (SOX-adjacent for any company with finance compliance — a real budget line item, unlike "agent safety" which is still seeking one). The pitch — «every journal entry has a deterministic trail back to its source events» — sells the Slice-8++ audit pillar to a real buyer before Pherix ships it.

Solo-founder feasible. Stripe / Mercury / QuickBooks / Xero APIs are well-documented; the agent's job is rule-driven matching plus exception handling, which LLMs handle adequately and Pherix journals deterministically. The reconciliation logic doesn't need industry credentials to prototype against fake data; the production sale would need a design partner, but the dogfood doesn't.

Pherix surfaces exercised: Slice 1 (SQL — daily ledger writes), Slice 4 isolation (multiple reconciliation agents on the same ledger), Audit pillar (the actual product).
Failure modes to expect: read/write keys not in the audit row (Slice-4 FU#2 — «show me which rows this reconciliation touched» is a basic compliance ask); durable-by-default audit journal (Slice-1 FU#4 — the audit can't be in-memory for a finance use case).

4. Narrow regulatory filing automation

build only if you have access smallest market, sharpest pitch

Pick one filing type in one jurisdiction — UK VAT, US 1099, a specific FCA submission, the Companies House confirmation statement. The agent prepares the filing, runs it through validation, submits to the regulator's API or portal, returns the receipt. Audit pillar isn't a feature; audit pillar is the deliverable.

Why this earns a slot. A submitted filing is gone (HMRC will not accept «actually we'd like to roll that back»). The gate via approve_irreversible is the operational pattern, not the exception — every filing goes through a human or higher-trust-policy sign-off. Audit must be regulator-grade. Smallest market of the four, but the narrowest scope is its strength: a solo founder can be the canonical name for "this specific filing" within weeks.

Why "only if you have access". Regulator portals and APIs vary wildly; some require live test environments only granted to incorporated entities; some require professional accreditations to submit. The market isn't competitive because the access barrier is real. If you have the access (you've filed this type yourself; you know an accountant who'd be a design partner), it's the cleanest Pherix-pitch in market. If you don't, the activation energy outweighs the dogfood gain.

Pherix surfaces exercised: Slice 3 (the submission itself), Slice 6 policy (filings have rule-based pre-checks), Audit pillar (regulator-grade), Slice 2 (prepared filings on disk before submission).
Failure modes to expect: Slice-6 policy engine arriving in time (Slice 1's allow-list is insufficient — filings need content-aware rules); audit-row schema bumps to capture the filing's full content (Slice-4 FU#2 plus a schema decision).

What NOT to build — and honest reasons

For completeness. None of these are wrong businesses; they are wrong dogfood for Pherix. The reasoning matters: each one fails a different Pherix-shaped test, and seeing the failure modes sharpens the picture of which dogfoods earn their slot.

Sales / outreach agents — the obvious AI-vertical play.reject · wrong wedge

An agent that drafts cold emails, sequences them, books meetings, updates the CRM. Hot vertical (Apollo, Outreach, Clay, hundreds more); GTM-heavy (you compete on distribution, not on technical wedge); and almost every action is recoverable: an email can be unsent in spirit by a follow-up, a calendar invite can be cancelled, a CRM row can be edited. The agent's failure modes are quality ("did the AI write something stupid") rather than containment ("did the AI cause irreversible harm"). AI-guardrails — content classifiers, tone detectors — are the right tool layer; Pherix's wedge is genuinely weak here.

When it could be right: if you already have the distribution network, sales agents are a real business — but as Pherix dogfood it stresses the wrong surfaces. Build it as a business; not as a Pherix test.

Customer support agents — the other obvious AI-vertical play.reject · wrong wedge

Decagon, Sierra, Ada and the long tail of Intercom-clones-with-AI. The agent reads tickets, drafts replies, escalates to humans, occasionally takes an action against a billing system or knowledge base. Most actions are reads; the writes that exist (email replies, CRM updates) are reversible at the social rather than transactional level. The buyer's pain is «will the AI say something embarrassing», not «will the AI cause an irreversible outcome». Same wedge mismatch as sales: detection > containment for this category. Pherix would be invisible.

When it could be right: if the support agent has authority to refund, cancel subscriptions, escalate to compliance review, then a narrow band of it becomes Pherix-shaped. But the headline use case (reply to tickets) isn't.

Healthcare / medical-record automation — Abridge / Suki / OpenEvidence territory.reject · regulatory tail unmanageable

The wedge is actually a near-perfect match — every action is irreversible-adjacent, audit is regulator-required, multi-agent concurrency exists across clinicians and AI scribes. The problem is HIPAA, BAAs, the cost of an information-security audit, and the requirement for live test environments only available to entities with healthcare insurance. As a solo-founder dogfood, the regulator-tail activation energy dwarfs the engineering work. The right way to enter healthcare is with a clinician co-founder or via a partnered design firm; neither is the solo-founder dogfood loop.

When it could be right: with a healthcare-side co-founder, this becomes the strongest Pherix vertical of all. Park it.

Legal automation — Harvey / Spellbook / EvenUp territory.reject · industry-knowledge moat

The wedge alignment is genuine (contract execution, filing, settlement disbursement are all irreversible) but the moat in legal AI is «which firm trusts you with their workflows», which is bought with industry credibility, not technical superiority. A solo founder without legal-industry trust can build the best legal-AI agent technically and lose to the worst one with three former Big Law partners on the cap table. Pherix as substrate-for-legal-AI is a real positioning long-term, but as solo-founder dogfood it stresses the GTM muscle Pherix is deliberately not building.

When it could be right: Pherix-as-platform sold to existing legal-AI vendors (Harvey buys your engine, doesn't replace it). Sales motion not Pherix-shaped today.

Marketing / copywriting agents — Jasper / Copy.ai etc.reject · no irreversibility

The agent's outputs are text: drafts, social posts, blog articles. Every "side effect" is reversible at the click-of-delete level. There's no «wire transfer»-shaped action, no «regulatory filing»-shaped action, no «production deploy»-shaped action. Pherix's wedge requires irreversibility somewhere in the agent's action set; marketing agents structurally don't have it. The right tool layer here is content quality (detection: brand-voice classifiers, hallucination filters) and editorial review, not transactional containment.

When it could be right: if the marketing agent fires the content (auto-posts to social, auto-sends emails to a list), the fire step is Pherix-shaped. The drafting step isn't.

Coding agents — Cursor / Copilot / Devin-style.reject · git makes reversibility cheap

Hot space, real money, but every action a coding agent takes is reversible via git — branches, reverts, stashes, undo-buffers. The cost of an undo is approximately zero (a commit). The AI-guardrails tool layer (linters, type-checks, CI) already does most of what could go wrong. Pherix's wedge — making irreversible side effects survivable — adds little when the underlying medium is git-reversible by default. The interesting Pherix-shape inside coding agents is the fleet version (migration bots, pick #2 above): fan out work across many repos and need cross-repo transactional semantics. The single-repo coding agent isn't it.

When it could be right: production-deploy actions inside the coding loop (auto-deploy on green CI) — but that's just the DevOps pick (#1) wearing a different hat.

Voice agents — phone agents, real-time conversational.reject · continuous channel

We covered this earlier in the conversation. The speech itself is the irreversible side effect; it's continuous and entangled with an un-pausable real-time channel; there's no clean stage-then-commit boundary because the agent is narrating in real time; and the gate is human-speed, which kills the call. The backend resource layer beneath a phone agent (the actual booking, the actual charge) could be Pherix-transacted — but that's the booking-system / payment-system pick, not the voice agent itself.

When it could be right: as the resource-layer infrastructure beneath voice products, never as the voice product itself.

The pattern in the rejections. Every one fails on one of three axes: wrong wedge (the agent's actions are reversible by default, so Pherix's containment story is invisible), wrong moat (the buyer's decision is bought with industry credibility, not technical alignment), or wrong channel (real-time / streaming kills staging). The four picks fail none of these — and between them, they cover Pherix's capability matrix with low overlap.

The structural advantage worth naming

Each prototype's failure modes feed Pherix's roadmap directly. The SRE bot needs result-piped compensators because kubectl scale returns a count the compensator needs — that's Slice-3 FU#1 becoming a customer-driven feature request rather than a speculative cleanup. The reconciliation agent needs the audit row to persist read / write keys so «which rows did this match touch» is answerable — that's Slice-4 FU#2 turning into a real ask. The migration bot needs the audit journal to survive a process crash so it can resume across a CI restart — that's Slice-1 FU#4 (durable-default audit) becoming load-bearing for a real use case.

The dogfood matrix is product discovery. Three small prototypes will surface more roadmap signal than three months of speculative spec work, and each one is also evidence we can hand to a buyer or a YC partner: «here's the agent running against real conditions; here's the journal; here's the rollback in action.» The verticals don't need to succeed as businesses for the dogfood to deliver its value. They need to survive their failure modes through Pherix.
