§ 01

What got shipped

Slice 1 is the smallest version of Pherix that is still Pherix. It is the reversible path, end‑to‑end, with one resource class: a SQLite database. Wrap an agent in agent_txn(...), call decorated tools, watch writes journal live, call rollback() — the rows are gone. Call commit() — they persist. The audit journal contains the entire story.

Nine modules. The interfaces are deliberately small. The discipline is that core/ never imports frontends/; every transactional capability is reached by traversing one ordered list of Effect records held by a Transaction. The SQLite adapter is correct by construction — SAVEPOINT and ROLLBACK TO SAVEPOINT do the heavy lifting at the database, not in Python.

Slices 2–8 do not extend this engine. They traverse it differently. That is the bet.

Jargon
Fold: collapse a sequence by repeatedly combining its elements with an accumulator. fold f init [a,b,c]  =  f(f(f(init, a), b), c). Forward folds left‑to‑right; backward folds right‑to‑left.

Savepoint
A nested marker inside a database transaction; you can roll back to it without aborting the outer transaction.
§ 02

The mental model

A Transaction is, formally, a pair: a state s ∈ TxnState and an ordered sequence J = [e₀, e₁, …, eₙ] of Effect records. The sequence is append‑only while the transaction is OPEN. We call J the journal.

For each effect e the runtime resolves a ResourceAdapter — a triple (snapshot, apply, restore). Each piece has a job:

commitforward fold

fold apply forward over J

commit(J)  =  for e in J : adapter[e].apply(e)

Effects were already journalled and applied live as they ran. Commit just finalises the underlying resource transactions in arrival order — the database COMMIT, the filesystem fsync, the HTTP send (in later slices). Idempotent on the journal: re-folding does nothing new.

rollbackbackward fold

fold restore backward over J

rollback(J)  =  for e in reversed(J) : adapter[e].restore(e.snapshot)

Reverse the list, ask each adapter to restore the before‑state it captured. Newest‑first matters: a later effect's snapshot is only meaningful relative to the state left by the earlier ones.

If the feature does not reduce to fold the journal forward, fold it backward, fold it against a snapshot, or diff it — it is not the right feature.

The point of having one structure is that every later capability is a small variant of these two folds:

CapabilityJournal operationSlice
Commitfold apply forward1
Rollbackfold restore backward1
Auditread the journal1
Replayfold apply against fresh state, assert ≡5
Dry‑runfold apply against a snapshot, discard7
Isolationdiff read/write keys against the journal4

That table is the entire shape of the system. Slice 1 builds the rows in bold; the rest are mechanical extensions of the same iteration.

§ 03

The architecture

Nine modules. The arrows show import direction; no cycles. The boundary between core/ and frontends/ is the discipline that keeps the MCP gateway (Slice 8) a thin re‑skin rather than a rewrite.

fig 1. Module dependency graph 21 files · one direction
hard import uses type but no direct dep cycle runtime drives Arrow head → X means “depends on X”.

How a tool call flows through it

The agent calls a decorated function. tools.py sees an active TxnContext in its ContextVar — Python’s async‑safe analogue of thread‑local state — and routes the call to ctx.record_tool_call(...) instead of executing the raw function. runtime.py then resolves the adapter from spec.resource, builds the Effect, appends it to the journal, asks the adapter to snapshot(), then to apply(), and finally writes the result to the audit journal. Outside agent_txn() the @tool wrapper is a transparent passthrough — tools work un‑journalled, no transaction awareness needed at the call‑site.

§ 03 · b

Why these splits? — the orthogonality argument

a deepening, not a restating

Nine modules to support a transaction that, in the end, decides whether some writes happen or unhappen, is a fair thing to push back on. The defensible test is orthogonality: two things belong in different modules iff they need to vary independently. Borrowing the notation from linear algebra — A ⊥ B reads “A is independent of B” — the claim is that each module corresponds to exactly one axis of variation, and no two modules share an axis. That is why the splits are the splits.

The pitch — one engine handling five hard problems as different traversals of a single journal — forces a specific set of axes. Read across the matrix: each row owns a column nothing else owns.

Not every split is defensible on the same grounds. Three categories, plus an honest concession:

A · logically necessary

splits that must exist

Future capabilities (replay, isolation, dry‑run) demand these axes vary independently. Collapsing them would produce a category error, not just a clutter.

  • audit state ⊥ resource state
B · categorically different work

  • policy.py — predicate logic
  • runtime.py — orchestration
C · the architectural bet

the one split the whole roadmap depends on

  • frontends/ ⊥ core/ — library frontend (in-process decorator) and MCP gateway (Slice 8) share the same engine; the engine must depend on neither. Collapse this split and Slice 8 becomes a rewrite, not a re-skin. This is why frontends/ exists as a top-level package on day one — it is not decoration.
D · honest concession

not every split earns it on logic alone

  • pherix/__init__.py separate from library.py — Python convention (re-exports). Not necessity.
  • policy.py as its own ~50‑line module for Slice 1 — defensible by trajectory (Slice 6 grows it into capability grants, spend caps, content‑aware rules) but honestly it could have lived inside runtime.py today. The one forward‑leaning case.

The three that logically must be split

Each of these is a real orthogonality — future capabilities demand the axes vary independently. The third is the sharpest.

01journal ⊥ state
effects.py ⊥ transaction.py   (journal ⊥ state machine)

The journal is data that future capabilities — replay, dry‑run, audit — must walk without going through the state machine. Coupling them forces replay to depend on transitions it doesn’t care about: replay isn’t governed by OPEN → COMMITTED, it is a pure forward fold of apply over the effects. If the only way to walk the list ran through a state machine, you would be re‑running a state machine just to read data.

02protocol ⊥ impl
adapters/base.py ⊥ adapters/sql.py   (protocol ⊥ concrete adapter)

The same journal must drive multiple adapters — SQL today, filesystem in Slice 2, HTTP in Slice 3. The protocol — who can be an adapter — must live separate from any one concrete implementation. Putting the protocol inside the SQL file would couple the abstraction to its first implementation. That is exactly the bug abstractions are meant to prevent: the interface should outlive the implementation that motivated it.

03audit ⊥ state
audit.py ⊥ any adapter’s store   (audit persistence ⊥ resource state)

The sharpest one. audit.py writes to its own SQLite database, separate from any adapter’s. If audit lived in the same database the transaction is transacting, then rollback() would roll back the audit too — and you would lose the record of what happened. Audit must live in separate persistence to be honest about “rolled back” vs. “never happened”. That is logical necessity, not convention. A failed transaction whose audit trail also vanished is indistinguishable from a transaction that was never attempted — which is the whole thing Pherix is built to make distinguishable.

Map the four categories back onto the dependency graph in fig. 1 — the same nine modules, now coloured by which kind of split they are:

The bet (frontends/ ⊥ core/) is the only edge that runs out of the core box. Everything in core is colour‑coded by which category earns its split. audit.py gets the extra weight because of the writes-its-own-DB property — it is the only module whose persistence layer is independent of the system it observes.
honest concession

Two of the nine splits do not survive the orthogonality test on logic alone:

  • pherix/__init__.py separate from library.py — Python convention (the package re‑exports its frontend). Not necessity. A flatter project layout could collapse them and lose nothing.
  • policy.py as its own module for Slice 1 — ~50 lines, one allow/deny check. Defensible by trajectory (Slice 6 grows it into capability grants, spend caps, content‑aware rules, commit‑time re‑evaluation) but in Slice 1 alone it could honestly have lived inside runtime.py. That is the one forward‑leaning case — and naming it as such is what keeps the rest of the argument credible.

The framework is defensible to the degree splits correspond to real orthogonalities — most do. It is over‑engineered to the degree splits anticipate growth that hasn’t happened yet — and policy.py is the one honest case of that. No others.

Why an idempotency key on day one

Slice 3 introduces staging — an effect is recorded but not yet applied, and may be applied later (potentially after a crash, definitely after a network retry). At that point you need a way to say “this is the same effect, do not double‑apply it”. The natural key is hash(txn_id, index, tool, args) — deterministic, stable across runs, and computable without coordination because the txn already owns a monotonic index.

Slice 1 doesn’t need this key, but it is cheap to compute and impossible to retrofit safely once the journal schema is set. So we compute it now.

§ 05

The state machine

Six states are defined. Two transitions are exercised. The rest are scaffolding for later slices — the type lives in code so consumers never need to be patched.

Slice 1 only allows OPEN → COMMITTED (clean exit, or explicit .commit()) and OPEN → ROLLED_BACK (exception inside the with block, or explicit .rollback()). Every other transition raises TransactionStateError.

fig 2. TxnState transition graph 2 of 30 edges live
Filter:
live transition dormant (later slice) illegal (any) → OPEN, or COMMITTED → anything, etc.

The transition table itself lives in core/transaction.py as one literal dict — _ALLOWED_TRANSITIONS = { OPEN: {COMMITTED, ROLLED_BACK} }. Adding a transition is one line. The state machine knowing it’s incomplete is the point.

§ 06

The effect lifecycle

Each effect has its own state machine, smaller and orthogonal to the transaction’s. The runtime walks an effect through it deterministically: built → policy.check → snapshot → apply → APPLIED. On rollback it walks the journal backward, turning APPLIED into COMPENSATED. A failing apply() short-circuits to FAILED.

fig 3. EffectStatus transitions per-effect state, mutated in place
happy path failure path rollback path reserved (Slice 3)

Important subtlety: STAGED here is an instantaneous state — an effect is born STAGED for a microsecond before the runtime applies it. It is not the same as Slice 3’s irreversible staging where an effect sits in STAGED awaiting approve_irreversible(). Same name, same enum value, different residency time.

§ 08

The six locked decisions

These are the design choices that propagate into every later slice. Locked means: changing them retroactively is expensive, so we picked deliberately, wrote them down, and pinned tests to them.

D1interception

Transparent interception via a ContextVar

The @tool wrapper checks active_txn.get(). Inside agent_txn() calls are journalled and routed; outside, they passthrough untouched. No txn.call(...) at the agent's call-site — the agent loop stays transaction-unaware.

tools.py:21 · tools.py:90
D2handle injection

Adapter injects the resource handle

The SQL conn is supplied by the adapter at apply‑time as the tool's first positional arg. The agent never sees, names, or passes it — bind_args strips it before the journal records the call.

adapters/sql.py:64 · tools.py:33
D3lifecycle

Runtime drives adapter txn-scope

The adapter exposes begin / commit / rollback for transaction‑wide lifecycle, separate from the per‑effect snapshot / apply / restore triple. The runtime calls them at the boundaries of agent_txn().

adapters/sql.py:34 · runtime.py:170
D4rollback order

Rollback is a backward fold, newest-first

Not one outer ROLLBACK — the runtime walks reversed(txn.effects) and calls adapter.restore per effect. This is the universal engine Slices 2–3 need (a filesystem revert is per-path; an HTTP compensator is per-call).

runtime.py:107
D5audit shape

In-place status update, no event log yet

One row per effect in audit.db, mutated to track status. Slice 5 (replay) needs finer event grain — deferred deliberately. For Slice 1, the in‑place row tells the whole story.

audit.py:1 · audit.py:125
D6policy timing

Stage-time policy only (for now)

The runtime calls policy.check(tool) once, at stage‑time, before any resource is touched. Slice 6 adds commit‑time re‑evaluation (TOCTOU safety). Allow‑list / deny‑list is enough for Slice 1.

policy.py:34 · runtime.py:63

Where the next slice picks up

Slice 2 adds FilesystemAdapter — copy‑on‑write or content‑hash backups of touched paths. It is the first real test of the adapter protocol: if the runtime can drop in a second adapter and the journal still folds correctly, the abstraction is load‑bearing rather than decorative. Everything in this document — the journal, the folds, the lifecycle, the six decisions — stays unchanged.
