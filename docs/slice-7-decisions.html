Pherix · Slice 7 decisions

"Show me what this transaction would do — but don't actually do it."

Slice 7 is the smallest slice in the build. Mechanically, it is agent_txn that rolls back at the end instead of committing — the same fold over the same journal, against the same adapters, with the existing BEGIN/ROLLBACK bracket already giving us "discard the world" for free. So unlike Slice 5 (new walker) or Slice 6 (new engine), Slice 7 mostly composes. The interesting decisions are about what gets handed back to the agent — the shape and contents of DryRunResult — and where the "nearly free" framing stops being honest.

The premise — what's free, what's actually new

An agent wants to ask "what would happen if I committed this transaction?" — to inspect the full state diff, the policy verdict, and the list of irreversible effects that would have fired, all before any of it is real. The classic pre-flight check.

The mechanical move is small: open the txn normally, run the agent's body, evaluate policy as usual — but at the end, call adapter.rollback() instead of adapter.commit(). Reversibles (SQL, FS) unwind via the existing bracket; irreversibles (HTTP) never fired in the first place because the staged-fire loop runs after the bracket terminates. The world is unchanged. What the agent gets back is the journal — the complete record of what would have happened.

What comes free from Slices 1–6

Almost everything.

  • Snapshot/rollback bracket — Slice 1 ships BEGIN/ROLLBACK per adapter; Slice 2 ships filesystem copy-on-write. Both restore on rollback.
  • The journal — Slice 1 records every effect. The journal IS the diff at the per-effect level.
  • Irreversible staging without firing — Slice 3's gate already means staged irreversibles don't fire until commit. Dry-run never commits. Done.
  • Policy evaluation — Slice 6's evaluate_journal walks the journal at commit-time. Dry-run calls the same method.
  • Per-effect snapshot detail — Slice 1 records the snapshot handle on every effect. Adapters with structured snapshots (SQL savepoints, FS file-list) carry "before" state at no extra cost.

What Slice 7 actually has to build

The result surface.

  • A dry_run(...) entry point that wraps the existing runtime and routes commit→rollback at the end.
  • A DryRunResult dataclass packaging the journal, the would-have-fired irreversibles, and the policy verdicts.
  • Capture-mode policy evaluation (so a denial becomes data in the result rather than an exception that aborts the dry-run).
  • A demo + tests proving the world is unchanged after dry-run.
  • (Open call) Per-adapter structured state diff — the bit where "nearly free" stops being honest.

The five decisions below cover the shape of that surface. The open call (concurrency-aware dry-run) is the natural follow-up; the deferred per-adapter state diff lives there too.

D1 — What's the literal Python the operator types?

The question

"I have an agent body that today runs inside with agent_txn(adapters): .... What's the smallest change I make so the same body becomes a dry-run?"

Option A — flag on agent_txn

with agent_txn(adapters, dry_run=True) as ctx: ...

Mechanically smallest. agent_txn gains one kwarg; if true, the commit-path branches to adapter.rollback() and the result lands on ctx.dry_run_result for the caller to inspect after the with exits.

Tiny diff for the operator — flip a flag.

Couples two distinct semantic operations into one entry point. agent_txn is "commit-or-rollback-on-failure"; dry-run is "always-rollback-and-return-diff." Conflating them grows agent_txn's contract every time dry-run gains a knob.

Doesn't compose well with the existing replay()-as-separate-entry-point pattern, which Slice 5 explicitly locked.

Option B — top-level dry_run() context manager

with pherix.dry_run(adapters) as ctx: ...; result = ctx.result

from pherix import dry_run

with dry_run({"sql": SQLiteAdapter(conn), "fs": FilesystemAdapter(root)}) as ctx:
    publish_doc(doc_id="d1", title="Intro")
    notify(doc_id="d1")

# Adapters unchanged. The result is the diff.
result = ctx.result
print(result.would_have_fired)        # []
print(result.policy_verdicts)         # [allow, allow]
print(result.journal)                 # the full per-effect record

Mirrors the agent_txn / run_txn / replay shape — each entry point is its own operation. Internally it shares 90% of agent_txn's machinery; externally it has its own contract.

Clean separation: agent_txn stays focused on commit semantics, dry_run on observation semantics.

Matches replay()'s register — separate entry point per distinct semantic operation. Slice-5 lock holds.

Future dry-run knobs (per-adapter diff opts, concurrency-aware mode) attach to dry_run without growing agent_txn.

Slightly larger diff for the operator — they import a different symbol.

Option C — separate function, not a context manager

result = pherix.dry_run(adapters, fn=lambda: ...)

The agent body is naturally written inside a with block — passing it as a callable forces a refactor. The context-manager shape is what's idiomatic.

Locked — Option B

Top-level pherix.dry_run(adapters) context manager, yielding a DryRunContext whose .result attribute holds the DryRunResult after the with exits. Internally shares the runtime with agent_txn; externally is its own entry point.

D2 — What happens to irreversible effects inside a dry-run?

The question

"My agent body calls charge_card(amount=50). Charging the card is irreversible — that's exactly what makes dry-run useful. But what does the staged effect actually do during the dry-run?"

Option A — refuse to dry-run any txn containing irreversibles

Raise at the first irreversible @tool call inside dry_run.

The whole point of dry-run is to preview transactions that include irreversibles. A pre-flight check that refuses on the most important case is no pre-flight check at all.

Option B — stage as normal, never fire, list in the result

The Slice 3 staging path already does this. Dry-run just never commits, so the staged-fire loop never runs.

Concretely: the agent's charge_card(amount=50) call returns a StagedResult (Slice 3's pattern) carrying a deterministic effect_id. The Effect lands in the journal with status=STAGED. At the end of the dry-run, the journal contains a STAGED irreversible that never fired and never will. The runtime extracts these into the result's would_have_fired list.

Zero new logic — Slice 3 already does exactly this for the "staged but unapproved" case.

The agent gets the most useful answer: "your txn would have called charge_card with {amount: 50}", with the args inspectable.

A tool that returns a StagedResult instead of a real value during dry-run will surprise agent code that immediately dereferences the result. Mitigation: StagedResult already has this property in Slice 3; agents written to it work in both modes.

Option C — pre-approve and still don't fire

Auto-call ctx.approve_irreversible(...) for every staged effect, then route commit→rollback.

Semantically incoherent. Pre-approval exists so the agent can sanction firing at commit; if there's never a commit, approval is meaningless theatre. Worse, it might mislead a future contributor into thinking dry-run carries any commit-time semantics it doesn't.

Locked — Option B

Irreversible tools stage normally during dry-run: the Effect lands in the journal with status=STAGED, the agent receives a StagedResult, and the runtime never reaches the staged-fire loop because dry-run rolls back instead of committing. The result's would_have_fired list is just the journal filtered by (reversible=False, status=STAGED).

D3 — What does DryRunResult actually contain?

The question

"I'm the agent. I called dry_run(...) and the with just exited. What can I now inspect to answer 'what would my txn have done'?"

Three honest layers of detail, each one more expensive than the last:

Layer 1 — the journal itself (free)

Every effect that staged during the dry-run, in order.

Slice 1 already records this for every txn. Dry-run hands it back. Each effect carries its tool name, args, resource, reversible flag, status, and (for adapters that snapshot structurally) the snapshot handle.

Zero new work. The journal already exists.

Per-effect granularity — agent can iterate and inspect each call individually.

Layer 2 — would_have_fired list (free)

The subset of the journal where reversible=False.

Filter the journal. Zero new logic. Useful because irreversibles are the operator's primary concern — a dry-run that says "this txn would have called charge_card({amount: 50})" is the headline answer.

Trivial to compute.

Maps to the operator's mental model — "what would have actually happened in the world?"

Layer 3 — policy verdicts (free, see D4)

Per-effect Allow / Deny from Slice 6's evaluate_journal.

Run the policy bracket as usual; capture every verdict into a list instead of raising on first Deny. The agent gets a complete map of which rules would have allowed each effect, which would have denied, and (under D4) where each denial would have fired.

Layer 4 — per-adapter structured state diff (real work)

{"sql": {rows_added: [...], rows_modified: [...]}, "fs": {files_added: [...], files_modified: [...]}}

The operator-most-useful layer. "What rows would have been inserted? What files would have been written? What bytes would have been changed?"

Genuinely new work: each adapter needs a diff(snapshot, current) → dict method. SQLiteAdapter can compute it via the savepoint plus a query before rollback; FilesystemAdapter via the copy-on-write tree. HTTPAdapter has no state to diff (it's already in the would-have-fired list).

This is where "nearly free" stops being honest. Per-adapter diff is at least a slice of its own work — savepoint queries, diff serialisation, large-table strategies.

Deferred to Slice 7.5 or per-adapter opt-in via a StateDiffable sub-protocol.

Locked — Layers 1, 2, 3 ship; Layer 4 defers

DryRunResult carries the journal (per-effect record), the would_have_fired list (filtered irreversibles), and the policy_verdicts map (one verdict per evaluation). Per-adapter structured state diff (rows added, files written) defers to Slice 7.5 via an opt-in adapter sub-protocol. The "nearly free" framing stays honest.

D4 — What does policy do during a dry-run?

The question

"My txn would have been denied by a Slice 6 rule. The dry-run is just observation — should the agent get a PolicyViolation exception, or should the denial come back as data in the result?"

Slice 6 has the right hook: evaluate_journal walks the journal at commit-time and raises PolicyViolation on the first Deny. Two ways dry-run can use that:

Option A — raise like a real txn

Dry-run runs evaluate_journal; first Deny raises PolicyViolation; the with block aborts with the exception.

Preserves the exact contract of a real txn — operator's except PolicyViolation: code handles both modes identically.

The agent stops at the first denial and can't see what would have happened past that point. Dry-run's job is to be observation, not enforcement.

Partial results — the DryRunResult is unreachable because the exception bypassed it. Operator gets one denial and no journal context.

Option B — capture verdicts, never raise

Dry-run runs every rule against every effect; collects all verdicts (Allow + Deny) into result.policy_verdicts; the txn proceeds to rollback regardless.

A Deny is data, not an error, in dry-run mode. The agent learns "rule no_enterprise_updates would have fired on effect #2 with reason 'enterprise tier off-limits'" — and also learns what effects #3, #4, #5 would have done. Complete observation.

The whole journal is inspectable, not just the first denial.

Slice 6's PolicyViolation attributes (where, rule, reason, effect_index) become structured data on the verdict list — no information lost.

Operator can branch on the result's shape: result.is_clean (no denials), result.is_denied (at least one Deny). Cleaner than try/except.

Different contract from a real txn — the operator has to know dry-run captures instead of raising. Mitigation: this is the only sensible answer for an observation mode; document it loudly.

Locked — Option B

Dry-run runs evaluate_journal in capture mode: every rule evaluates against every effect, every verdict lands in result.policy_verdicts, no exception is raised. The agent reads the result and decides what to do. Genuine errors (adapter failures, malformed effects) still raise; policy denial is observation.

D5 — What's the state-preservation guarantee?

The question

"After dry_run(...) returns, can I trust that nothing changed in my SQL database, my filesystem, my HTTP world, my audit log? Where's the line?"

The three places state could plausibly leak:

Reversible adapters (SQL, FS)

Full restore via the existing bracket. No leaks.

Slice 1 ships adapter.begin() / adapter.commit() / adapter.rollback() as the per-txn bracket. Dry-run uses the same bracket but calls rollback() instead of commit() at the end. SQLite issues ROLLBACK; the SQL state is identical to its pre-dry_run state. Filesystem's copy-on-write tree gets discarded; the on-disk filesystem is unchanged.

The bracket already exists. Zero new code; the guarantee is mechanical.

Irreversible adapters (HTTP)

Never fired, so nothing to roll back.

Slice 3's staging pattern: the irreversible tool's apply method runs only after explicit approval and only inside the staged-fire loop at commit-time. Dry-run never reaches that loop. The agent's call to charge_card(...) returned a StagedResult; the real charge_card function was never invoked.

The strongest possible guarantee: the irreversible function literally did not execute.

Audit journal

Operator-supplied target_audit, defaults to in-memory.

Mirroring Slice 5's replay() pattern: dry_run takes an optional target_audit kwarg. Default is AuditJournal.in_memory() — the dry-run's journal lands in memory only and disappears when the result goes out of scope. Operators who want a persistent record of dry-runs (e.g. "audit every preview I ran today") can pass a real AuditJournal(path).

The audit row for a dry-run txn carries a dry_run=True column (or equivalent — a single boolean on the transactions table). An operator querying the audit can filter dry-runs out of compliance views cleanly.

Honest: dry-run can leave a trace, but only one the operator opted into.

Composes with Slice 5's replayed_from column shape — both are "this txn was something other than a normal commit" flags.

Locked — strict on adapters, opt-in on audit

Reversible adapters: full restore via existing bracket. Irreversible adapters: function never invoked. Audit: operator-supplied target_audit defaulting to in-memory, with a dry_run=True column on the audit row so traces can be filtered. The world is exactly as it was before dry_run ran — except for the result handed back to the caller and the audit row if the caller asked for one.

Open — concurrency-aware dry-run

The question

"My dry-run is running while another agent is writing to the same database. Should the dry-run see that other agent's in-flight writes? Should it surface 'your txn would conflict with txn-foo'?"

Slice 4 ships JournalRegistry + check_conflicts for exactly this question in the real-txn case. Could be extended to dry-run:

  • Dry-run registers with JournalRegistry at start;
  • Runs check_conflicts at end against the live registry;
  • Reports detected conflicts in result.isolation_conflicts;
  • De-registers on exit.

Useful, but real complexity:

  • Should dry-run block other txns? If a dry-run is "in flight" in the registry, does a real txn waiting on a key the dry-run claims to write get held up? Probably no (dry-run never commits), but the registry contract has to be widened to express "soft claim."
  • Phantom reads. The dry-run's read-set sees a snapshot. If a concurrent committed write changes the value between snapshot and dry-run end, the dry-run's "what would my txn do" answer is stale by the time the operator reads it.
  • Cross-process arbitration. Slice 4's filesystem-shared SQLite meta-connection works for real txns. Dry-runs cross-process would need either the same machinery (and the same complexity) or an explicit "in-process only" caveat.

Recommendation — defer to Slice 7.5

Slice 7 ships isolated dry-run: the dry-run sees a snapshot of state as it was when begin() ran, and does not participate in JournalRegistry. Concurrency-aware dry-run becomes a natural Slice 7.5 along with per-adapter structured state diff — both are "make dry-run answer more questions about the real production world." Slice 7 stays small; the engine surface for both extensions already exists.

Closing — fold then discard

Three sentences of formalism, the only ones in this document. Commit is a forward fold over the journal followed by adapter.commit() per resource. Rollback is the same forward fold followed by adapter.rollback(). Dry-run is the same forward fold followed by adapter.rollback() plus a returned diff. The engine doesn't change; the terminator changes.

That is what "nearly free" actually means here. The journal is the fold's record; the bracket is the discard mechanism; Slice 6's evaluate_journal is the verdict layer; Slice 3's staging keeps irreversibles unfired by default. Slice 7's contribution is to compose these four pieces into a new entry point and package the result. No new abstractions in the core; one new dataclass for the result; the rest is wiring.

The journal records what would have happened; the bracket ensures it didn't. Slice 7 is the moment Pherix's structural choice — separating effects from their application — earns its keep across the build for the third time, after rollback (Slice 1) and replay (Slice 5).
