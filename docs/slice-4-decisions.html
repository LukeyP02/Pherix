Isolation: where Pherix earns the "I" in ACID.

Slices 1–3 give each agent atomicity, durability, and (via staging plus compensators) a credible consistency story. None of that yields isolation. Isolation is a relational property — it lives between transactions, not inside one. You cannot sum it from per-agent guarantees. Two agents that are individually flawless can lose updates against each other the moment their reads and writes interleave on shared state. Slice 4 closes that gap. Four decisions are locked. D5 is the open call — the question is how wide a circle of arbitration to draw around the journal at this slice.

Read this first if any term is new — isolation, conflict, serializability, MVCC, TOCTOU.

Isolation is the I in ACID. It is the only one of the four that is not a property of a single transaction in isolation — it is the property that concurrent transactions interleave without observable interference. The textbook formal target is serializability: every interleaving of committed transactions must be equivalent to some serial ordering of those same transactions.

Conflict in this setting has a precise definition. Two operations conflict iff (i) they touch the same key, (ii) at least one of them is a write, and (iii) they are in different transactions. Conflicts are the non-commutativity events of the system — the points where reordering changes the result.

Version is a monotonically increasing tag attached to a key's state. When a write happens, the key's version increments. A read records the version it saw. At commit, "has the version moved?" is the conflict-detect question.

MVCC — multi-version concurrency control — is the family of techniques where each key keeps a history of versions and readers see a snapshot rather than locking. Postgres SSI is a production example. Pherix's mechanism is MVCC-flavoured at the journal level, but the journal is the witness, not a per-key version chain.

TOCTOU — Time-Of-Check vs Time-Of-Use, the class of bug where a property is checked, then acted on, with intervening state change invalidating the check. In Slice 3 it motivated the second policy check at commit. Here it is the entire reason conflict detection exists.

§0

Per-agent transactionality does not aggregate into isolation.

Each agent gets A · C · D from Slices 1–3. The "I" is a property of the relation between agents — it has to be installed at the layer that sees all of them.

Two agents A and B are running in parallel. Both wrap their tool calls in Pherix. Each is, by construction, atomic and durable. Each runs against a snapshot of the world that it can roll back from. Yet the world they share can still go wrong: A reads users:42.balance = £100, decides to deduct £30, and writes £70; concurrently B reads the same row at the same version, decides to deduct £40, and writes £60. Both commit. One update is lost. Neither agent did anything wrong internally. Slice 4 is what stops this.

Two operations on the same key commute iff at least one is a read.
Two operations conflict iff they touch the same key and at least one is a write and they are in different transactions.
Isolation = the system preserves serializability — every interleaving of conflicting transactions is equivalent to some serial ordering.

That last line is the target. Pherix's mechanism for hitting it is journal-based and structurally consistent with everything before it. Every Effect has carried read_keys and write_keys slots since Slice 1 — tuples (resource, key, version) for reads, (resource, key) for writes. Currently inert. Slice 4 is the slice that finally reads them: at commit, diff this transaction's read_keys versions against the journal of what else has committed since the transaction opened, and act.

The herd-immunity argument restated under load: per-agent transactionality is a vaccine; isolation is the herd. The vaccine works fine on the individual — but it does not, by itself, stop the pathogen at the population level. The herd-level mechanism is the journal-as-arbiter pattern that Slice 4 brings online.

D1

How does every effect populate read_keys and write_keys?

The slots exist. The question is who fills them — the handle that mediates the I/O, the tool-author's metadata, or static analysis.

Conflict detection needs to know which keys this transaction read and which keys it intends to write. The tuples must be accurate by construction — a single missed key silently breaks the isolation guarantee. Why (a) wins

Capture-by-handle has the same epistemic shape as Slice 2's lazy backup: the operation is the source of truth, not a description of the operation. The handle is the only object that actually executes the I/O, so it is the only object that knows the truth. FsHandle.touched is already this pattern proved out — Slice 4 extends it from "we touched these paths, snapshot them" to "we touched these keys at these versions, journal them."

Both (b) and (c) ask a different party to predict the handle's behaviour. Predictions drift, and isolation guarantees that depend on accurate predictions are not guarantees. (a) is the only option where the data is correct by construction.

The single-site change in the handle and the runtime. 
# pherix/core/adapters/filesystem.py — FsHandle extension
class FsHandle:
    def read_bytes(self, path):
        v = self._adapter.read_version(path)         # D2 — adapter exposes
        self._effect.read_keys.append(("fs", str(path), v))
        return path.read_bytes()

    def write_bytes(self, path, b):
        self._effect.write_keys.append(("fs", str(path)))
        self.touched.add(path)                       # Slice 2 unchanged
        path.write_bytes(b)

Two lines added per primitive. The runtime hands the active Effect to the handle when the tool body runs — the same mechanism Slice 2 already uses to thread the per-effect backup directory through.

D2

What is a version, and who computes it?

A read records (resource, key, version). The version is a tag in a totally ordered set, per key. Where does it come from?

For each resource we need a function v : Key → ℕ that assigns a monotonically increasing tag to a key's state. When a write happens, v(k) increments. A read records the value of v(k) at the moment of the read. Two reads at the same version are guaranteed to have observed the same state; if the version has moved between read and commit, the state has moved.

Different resources have radically different mechanisms for computing such a function. The adapter is the place where that knowledge lives — it is the only object in Pherix that knows the resource's actual semantics. The adapter interface gains two methods, read_version(key) and write_version(key); each adapter implements them in whatever way is cheapest and most accurate for its world.

The three options on the table

Each adapter implements read_version(key) → v.

The journal stores whatever the adapter returned. SQL, FS, future Vector, future Queue — each picks its own scheme. The adapter abstraction is already Pherix's seat for "this resource's actual semantics," so the version function naturally lives there.

Why (a) wins

Pherix is already built around the claim that the adapter is the natural home for resource-specific semantics. Versioning is the most resource-specific thing imaginable — what "the same state" means for a SQL row, a file, an embedding index, and an outbound HTTP call are four entirely different questions. (a) is the only option that lets each adapter answer in its own language and hands the journal a uniform answer back.

(b) doesn't cross-resource — and the Pherix wedge is that it crosses resources. (c) creates a global counter that must be incremented under a lock on every write everywhere, which is the bottleneck that motivates MVCC's existence in the first place. (a) generalises both: each adapter is free to use its native MVCC where it has one (Postgres) and a logical counter where it doesn't (SQLite).

D3

When does conflict detection actually run?

Commit is the natural single chokepoint. The question is whether to also check inside the transaction, the way D4 of Slice 3 re-evaluated policy.

In Slice 3 we made the case for re-checking policy at commit-time: state moves between stage and commit, and a single check at stage-time has a TOCTOU window. The same instinct asks here: should conflict detection also run at every read or write, not just at commit? The answer this time is no — and the reason is structural, not just pragmatic.

Why the policy analogy breaks: a txn's reads are invisible until commit

Slice 3's policy re-check existed because the world could move between stage and commit — other transactions could push the spend counter up, and our staged effect would silently become non-compliant. Conflict detection has no equivalent window. The journal is append-only; a transaction's writes are not visible to anyone else until that transaction commits. So inside our transaction, our own reads see a consistent snapshot — the world cannot have moved between two of our own reads in a way that we should care about, because we're reading our own pre-commit view. The only moment where "the world has moved under us" becomes observable is the moment we try to make our changes visible. That moment is commit().

Why (a) wins

Conflict detection has exactly one place where it has both the information and the standing to fire: the moment this transaction asks to become visible. Before that moment our reads are inside our own bubble; the world's writes are not yet committed against us. Per-effect checking is the database-equivalent of holding a row-lock for the entire transaction — a recipe for deadlocks under contention, and unnecessary because the journal's own append-only semantics give us the snapshot for free.

Slice 3's two-phase policy check is the right pattern for policy because policy depends on world-state that genuinely moves. Conflict detection depends on a relation between our read-set and the journal — and that relation is meaningful only at commit. The pattern doesn't generalise; the question's shape is different.

The commit-time diff in pseudocode. 
# core/runtime.py — commit() additions
def commit(self) -> None:
    self._guard_open()

    # D3 — diff our read-set against the journal of what has committed since we opened.
    conflicts = []
    for effect in self.txn.effects:
        for (resource, key, seen_v) in effect.read_keys:
            now_v = self._adapters[resource].read_version(key)
            if now_v != seen_v:
                conflicts.append((resource, key, seen_v, now_v))

    if conflicts:
        return self._resolution.resolve(conflicts, self)   # D4

    # ... existing commit logic (adapter.commit, transition, finalise) ...

One pass, O(|read_keys|), executed exactly once per transaction. The diff hits the adapter — which already knows the current version — so it does not require holding a coordination structure beyond the journal itself.

D4

Conflict found — now what?

Three classical answers. They are not interchangeable; each suits a different operator goal. Pherix's job is to provide the mechanism, not pick the policy.

A conflict has been detected at commit. There are exactly three classical responses: abort, retry, or serialise (wait). Each is correct for some use case and wrong for others. A financial reconciliation pipeline wants abort and a human in the loop. An idempotent data pipeline can retry safely. A critical-section operation between two agents that must both eventually succeed wants serialise. Pherix exposes all three; the operator picks per-transaction; the default is the safest one.

Resolution policy: f : Conflict → {Abort, Retry(n), Serialize}
Choice of f is the operator's, per agent_txn(); Pherix records the choice and applies it deterministically.

Forcing one is forcing the project ethos onto Pherix's users, who know more about their workload than Pherix does. The IsolationPolicy protocol — three implementations. 
# pherix/core/isolation.py

class IsolationPolicy(Protocol):
    def resolve(self, conflicts: list[Conflict], ctx: TxnContext) -> None: ...

@dataclass
class Abort:
    def resolve(self, conflicts, ctx):
        ctx.rollback()
        raise IsolationConflict(conflicts)

@dataclass
class Retry:
    n: int = 3
    def resolve(self, conflicts, ctx):
        if ctx.attempt >= self.n:
            raise IsolationConflict(conflicts)
        ctx.rollback()
        ctx.reopen()                # new snapshot, attempt += 1

@dataclass
class Serialize:
    timeout: float = 30.0
    def resolve(self, conflicts, ctx):
        ctx.await_settle(conflicts, timeout=self.timeout)
        ctx.recheck()               # may raise IsolationConflict if still conflicting

Three small dataclasses, one protocol. The runtime calls self._resolution.resolve(conflicts, self) exactly once per detected conflict-set; the policy decides the rest.

∴

Isolation = serializability under interleaving — and the journal is the witness.

Pherix doesn't enforce isolation by locking; it enforces it by certifying that the interleaving could have been a serial ordering.

The formal target is precise. A set of concurrent transactions {T1, …, Tn} is serializable iff their interleaved execution produces a final world-state and a set of return-values equivalent to some serial execution of the same transactions. The witness for that equivalence is a transcript that orders the conflicting operations in a way that no transaction sees a version of a key that has subsequently been overwritten.

An execution E of transactions {Ti} is serializable iff there exists a permutation σ of {Ti} such that the serial execution Tσ(1); Tσ(2); …; Tσ(n) has the same observable result as E.
The journal — every Effect row, ordered, with read_keys and write_keys and version tags — is the transcript that proves the permutation exists.

Slice 4's mechanism is the assertion that σ exists. When a transaction commits cleanly, the journal can be read top-to-bottom as a serial order — every read saw a version that was not subsequently overwritten before that transaction's commit. When the conflict detector fires, it has found a pair of operations for which no such σ exists — and the resolution policy f chooses how to recover. Abort throws the conflicting execution away and leaves the journal witnessing only the surviving serial-equivalent prefix. Retry replays the aborted transaction against a fresh snapshot, after which it becomes a new entry in the serial order. Serialize stalls until the conflicting prefix has settled, after which σ exists by construction.

That is what isolation is, mechanised. Not "two agents can't interact" — they can, freely, and Pherix journals every read and every write they make. The guarantee is that the interleaving Pherix lets stand has a serial equivalent. The journal is the witness. Conflict detection is the predicate that fires when no equivalent exists. Resolution is the operator-chosen rule that recovers a serializable execution from the conflicting one.

ethos · restated Pherix provides the mechanism, not the policy. f is chosen by the operator. Nothing in Slice 4 is AI-powered; the resolution function is a deterministic rule on a conflict-set. The same line that separated Pherix from LLM-approver vendors in Slice 3 separates it from auto-resolution heuristics here. Pherix records, detects, and applies f; the operator picks f.

Same engine, more rows in the journal. Reversible effects still fold backward on rollback; staged effects still fire at commit. The new thing in Slice 4 is what the runtime asks the journal at commit — "has anything I read moved?" — and the policy f the operator hands it for the moments the answer is yes.
