Pherix · Slice 6 decisions

From "is this tool name allowed?" to a real predicate over the journal and the world.

Today's policy is a name-check — "is this tool's name in the allow list?" That covers a tiny corner of what an operator actually wants when they say "policy". Slice 6 turns it into a real predicate over (effect, world_state), evaluated at both stage-time and commit-time so a rule that passed before side-effects fired still has to pass after the journal folded. Five decisions below, one open call, one explicit deferral.

The premise

Policy.allow_all() and Policy(deny={...}) — that is the entire vocabulary today. The runtime calls policy.check(tool_name) at stage-time and raises PolicyViolation if the name is in the deny set. It is a one-line predicate: P(effect) = effect.tool ∉ deny.

That is enough for "agent can call index_doc, agent cannot call delete_database". It is not enough for any of the things below — which are, between them, what "real policy" actually means in production:

Args-aware

"Agent can update users where tier='basic' but not where tier='enterprise'."

Same tool name. Different args. Different verdict.

Needs: rules that inspect effect.args, not just effect.tool.

Spend caps

"Agent can call charge_card as many times as it wants, but not for more than $50 cumulative across this txn."

No single call breaks the rule. The journal taken together does.

Needs: rules that fold over the journal so far, not the current effect alone.

TOCTOU safety

"Agent stages a refund. Between stage and commit, the operator revokes the agent's spending grant. Commit must fail."

The check that passed at stage-time must run again at commit-time.

Needs: rules evaluated twice — once before journalling, once before the journal lands.

World-state-aware

"Agent can refund order #42 only if order.status='paid' right now."

The args don't change. The world changes.

Needs: rules that read live adapter state at evaluation time. (Deferred to 6.5 — see open.)

Slice 6 ships the first three. The fourth is the open call — every other architectural choice in this slice is shaped to leave room for it.

D1 — What kinds of rules does Slice 6 actually support?

The question

"We've named four failure modes. Which of them does Slice 6 ship, and which defer?"

Option A — bare minimum: args-aware only

Allow/deny stays, args-aware rules join, spend caps and TOCTOU bracket defer.

Smallest slice, fastest to ship.

Spend caps are the operator-facing demand most loudly heard ("limit my agent's blast radius"). Punting them makes Slice 6 feel like a half-step.

Without stage-and-commit bracket the architecture grows two evaluation points later, which is exactly the kind of retrofit Pherix has been disciplined to avoid.

Option B — args-aware + spend caps + stage/commit bracket

Three of the four. World-state-aware rules defer to Slice 6.5.

Args-aware predicates inspect effect.args. Spend caps fold over the journal so far. Both fire at stage-time (cheap, fail-fast) and again at commit-time (TOCTOU re-eval). World-state reads — "is this row's tier still 'basic'?" — defer.

Covers the three operator-facing demands that don't require new adapter primitives.

Stage/commit bracket lands as architecture even though args-only rules give identical results at both points — when world-state-aware rules join in 6.5 the engine doesn't need refactoring.

Honest about the limit: until 6.5, commit-time re-eval is structural, not behavioural.

Option C — full set including world-state reads

Ship all four failure modes in one slice.

World-state reads need a per-adapter "read at policy time" primitive — what version do we read at? what happens on read failure? how does it interact with isolation? Doubles the slice's complexity for the failure mode whose shape is least clear.

"Build it once" is a discipline, not an obligation to build it all at once. Slice 6.5 is the natural home for this.

Locked — Option B

Args-aware predicates + spend caps + stage/commit bracket land in Slice 6. World-state-aware reads defer to 6.5 with an explicit seam for them to slot in later.

D2 — How does the operator actually WRITE a policy rule?

The question

"Someone has to author 'no enterprise updates' or '$50 cap'. What does that look like — a Python function, a YAML file, or some kind of rules DSL?"

The three honest options operators reach for in real systems:

Option A — Python callables

A rule is a function. Register it with a decorator.

# A rule takes the effect about to fire and a context
# object (carrying the journal so far). Returns Allow,
# Deny(reason), or Defer.
@policy.rule
def no_enterprise_updates(effect, ctx):
    if effect.tool == "update_user" and effect.args.get("tier") == "enterprise":
        return Deny("enterprise tier off-limits")
    return Allow()

Matches Pherix's existing shape — tools are functions, compensators are functions, comparators are functions, rules are functions. Fully expressive, code-reviewed, importable from tests, no new file formats to maintain.

Operators already know how to test Python; rules become ordinary unit tests.

Expressivity is unbounded — anything the operator can write in Python, they can express as a rule.

Rules live in the same process as Pherix; rules-as-config files (option B) are easier to externalise to a different team. Slice 8 gateway can reopen that conversation if it bites.

Option B — declarative config

YAML/JSON file parsed at startup.

# policies.yaml
rules:
  - deny:
      tool: update_user
      args:
        tier: enterprise
  - cap:
      tool: charge_card
      sum: "args.amount"
      max: 50

Externalisable — security team can author rules without committing Python.

Expressivity ceiling — every new predicate shape needs a config-schema change. The first time someone needs "deny if (tool == X and args.role == 'admin' and time < 09:00)" you reinvent a DSL.

Doesn't compose with Pherix's existing decorator-and-callable register.

Option C — DSL (Rego / OPA-style)

Industry-standard authZ language, separate process or library.

Decades of policy engineering baked in; battle-tested for cloud AuthZ.

Heavy import — a whole new language for the operator to learn and the project to depend on.

Pherix's current ergonomic story is "Python all the way down". A Rego import breaks that without operator-asked-for value at this scale.

Locked — Option A

Rules are Python callables decorated with @policy.rule. They take (effect, ctx) and return Allow / Deny(reason). If externalisation becomes a requirement for the gateway (Slice 8), config-loading becomes a Slice 6.5 conversation — but the engine evaluates callables either way.

D3 — When does each rule actually fire?

The question

"Should the rule run when the agent calls the tool, or when the txn commits, or both?"

Same effect, same rule, two possible evaluation points. They are not interchangeable:

  • Stage-time — before the effect is journalled and (for reversibles) before the adapter applies it. Cheap. Fails fast. The agent sees the denial immediately.
  • Commit-time — after every effect in the txn has been folded forward into the journal, just before the adapter brackets commit. Expensive (the whole txn unwinds on a denial). Catches drift in world_state between the two points.

Option A — stage only

What we have today. No TOCTOU re-eval.

A rule that passed at stage-time can be silently violated by the time the txn commits — exactly the failure mode CLAUDE.md has been promising commit-time re-eval would catch since Slice 1.

Option C — commit only

Defer all evaluation to the end of the txn.

One evaluation point, simpler runtime.

Fails late. The agent stages 50 effects, blows resources doing it, and only finds out at commit that rule #3 would have denied effect #1. Pherix's own runtime calls policy.check at stage-time for a reason — stop early when you can.

Option B — both, same predicates, two evaluations

Each rule fires at stage-time AND again at commit-time. Same callable, evaluated twice.

Stage-time: agent calls a tool → runtime evaluates every applicable rule with (effect, ctx) where ctx carries the journal so far. Deny → raise immediately, fail fast.

Commit-time: every effect folded, journal complete → runtime walks the journal one more time and re-evaluates every rule against every effect. Deny here → reversibles roll back, irreversibles compensate, the txn unwinds.

For Slice 6's args-only rules, both evaluations give identical results (the args don't change between stage and commit). That is the honest position: the bracket lands as architecture, not as new behaviour, until 6.5 brings world-state-aware rules in.

CLAUDE.md has had stage-AND-commit as an invariant since Slice 1; Slice 6 is where we deliver it.

Engine is ready for 6.5's world-state rules without refactoring — they slot into the existing commit-time hook.

Locked — Option B

Every rule fires at stage-time and again at commit-time. Same predicate, same callable, two evaluation points. Args-only rules produce identical verdicts at both; the bracket is the architecture that lets world-state-aware rules join later without surgery.

D4 — Spend caps — what shape does the API take?

The question

"The operator wants to write '$50 cumulative cap on charge_card this txn'. What's the literal Python they type?"

A spend cap is a fold over the journal so far. Two questions in one: how do we say what to fold (the per-effect contribution), and how do we say how much we'll tolerate?

Option A — count caps only

Cap.count(tool='charge_card', max=10) — "no more than N fires."

Simplest possible primitive. Easy to implement, easy to explain.

Doesn't express the $-cap, which is the most-asked-for shape.

Option B — count + sum-via-fn

Two primitives covering count-based and arg-derived caps.

# "No more than 10 charge_card fires"
Cap.count(tool="charge_card", max=10)

# "Sum of args['amount'] across charges ≤ 50"
Cap.sum(
    tool="charge_card",
    via=lambda args: args["amount"],
    max=50,
)

Caps are themselves rules — they live in the same registry as custom @policy.rule functions. The runtime carries per-cap running totals on the context object and the cap's check is: "would this effect push the running total above max?" If yes, Deny.

Two shapes cover virtually every operator-facing use case (count, monetary, byte-budget, token-budget) without forcing tool authors to annotate cost ahead of time.

via=fn is composable with the comparator/compensator pattern — Pherix's "function-shaped extension points" register stays consistent.

No automatic discovery — the operator has to know which arg carries the cost. That's correct: tool authors shouldn't be guessing what "cost" means; the policy author should declare it.

Option C — cost-decorated tools

@tool(cost=lambda args: args['amount']) — tool declares its own cost.

Cost lives next to the tool definition, single source of truth.

Couples cost semantics (a policy concern) to tool definition (a behaviour concern). Different policies may want different cost interpretations of the same tool.

Forces a tool-author change every time a new spend axis emerges.

Locked — Option B

Two primitives: Cap.count(tool, max) and Cap.sum(tool, via=fn, max). They register alongside @policy.rule callables; the engine treats them identically. Cost-decorated tools (option C) can be a thin sugar layer in a future slice if it's asked for; the underlying engine doesn't change.

D5 — What does the operator catch when a rule fires?

The question

"The rule fired and denied the effect. Operator code is wrapping this in try/except. What goes in the except clause?"

This matters more than it sounds, because the consequence of a denial differs depending on when it fired:

  • Stage-time denial — no side-effects have happened yet (reversibles weren't applied, irreversibles weren't gated). The txn just doesn't start the effect. Cheap, clean.
  • Commit-time denial — reversibles have applied to the adapters; if they roll back cleanly, fine. Irreversibles already approved have fired against the external world — they need compensation, and if they have no compensator the txn lands in STUCK (the Slice 3 partial-unwind path).

The operator's recovery code is genuinely different in those two cases. So the exception they catch has to make the distinction visible.

Option A — keep one PolicyViolation

Existing exception, no new fields. Operator inspects txn.state to figure out where it fired.

Forces the operator to correlate exception with transaction state — easy to get wrong. The exception itself should carry enough context to make the operator's response correct without further lookup.

Option B — one PolicyViolation with where + rule attributes

Single exception type, but instances carry where they fired and which rule fired them.

class PolicyViolation(Exception):
    where: Literal["stage", "commit"]
    rule: "PolicyRule | Cap"
    reason: str
    effect_index: int | None  # which effect tripped it

One catch site for callers who just want "policy said no". The where attribute for callers who care about pre- vs post-side-effect handling. The rule attribute for operators building higher-level workflows ("if this specific rule fired, apply this specific recovery").

Backwards-compatible with the current PolicyViolation exception — existing code keeps working.

Operator's response code stays at one indentation level instead of branching on txn state.

Option C — split into two exception types

StagePolicyViolation and CommitPolicyViolation.

Type-driven dispatch (operators can except StagePolicyViolation: separately).

Doubles the API surface for what's really one concept ("policy denied"). Most callers want one catch site.

Breaking change for existing except PolicyViolation: code.

Locked — Option B

Single PolicyViolation exception, enriched with where ('stage' | 'commit'), rule, reason, and (for commit-time denials) effect_index. Existing except PolicyViolation: code continues to work; new code can introspect the attributes for richer recovery.

Open — World-state-aware rules, the deferred fourth failure mode

The question

"The operator says 'refund order 42 only if its status is paid right now'. The rule needs to read the live database at evaluation time. How?"

This is the failure mode D1 set aside as Slice 6.5. The shape isn't clear enough yet, and the deeper architectural questions it opens are real:

  • Read consistency. When the rule reads "order 42's status", does it read at the txn's snapshot version (consistent with isolation) or the latest committed version (consistent with the real world)? Different answers for different rules.
  • Read failures. The adapter is unreachable; the rule can't evaluate. Allow or Deny by default?
  • Interaction with isolation. A policy read introduces an additional read into the journal — does it participate in commit-time conflict detection, or is it a "policy-only" read that doesn't count?
  • Performance. Stage-time rules that touch the adapter are an order of magnitude slower than args-only ones. Does the runtime want a "this rule needs adapter access" hint so it can short-circuit cheap rules first?

Three candidate shapes if 6.5 picks this up:

Shape α — adapter.read_for_policy(key)

Adapter gains a dedicated read method for policy queries.

Clean separation but doubles the adapter protocol's surface area.

Shape β — rules take adapters parameter

Rule signature becomes (effect, ctx, adapters); rule calls adapters['sql'].read(...) directly.

Maximally flexible. Maximally easy to get wrong (rules accidentally writing instead of reading).

Shape γ — context exposes a read-only handle

ctx.read(resource, key) — runtime gates the read.

The cleanest middle ground; the runtime can short-circuit reads that aren't needed.

Recommendation — defer, but design Slice 6 with γ in mind

Slice 6 ships args-only rules and a context object that already carries a placeholder ctx.read attribute (raises NotImplementedError until 6.5). That way 6.5 is a content-only slice — fill in the read implementation, write the tests, no architectural surgery on the engine.

Closing — a policy is a predicate over the journal × the world

Three sentences of formalism, the only ones in this document. A policy is a predicate P : (effect, world_state) → {Allow, Deny}. Stage-time evaluation calls P(effect, world_before_apply); commit-time evaluation calls P(effect, world_after_journal_fold) for every effect in the journal. The world_state at evaluation time is where Slice 6 and Slice 6.5 part ways: 6 treats world as "the journal so far" (a pure fold, no adapter reads); 6.5 lets world include live adapter state.

Args-aware rules, spend caps, and the stage/commit bracket are all special cases of that one predicate. Spend caps are a fold over the journal's contribution to one running counter; args-aware rules ignore world_state entirely; stage/commit bracket is the same P evaluated at two different times. One engine, one rule abstraction, three operator-facing capabilities.

The journal is a witness; policy is a judge. Slice 6 hires the judge for the first time and gives them the journal to read from. Slice 6.5 will give them the right to subpoena the adapters too.
