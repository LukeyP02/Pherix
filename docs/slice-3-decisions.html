Read this first if any term is new — staging, gate, compensator, TOCTOU.

Reversible vs irreversible. A reversible effect is one whose before-state can be reconstructed: SQL row insertion (DB savepoint), file write (copy-on-write backup). An irreversible effect is one whose before-state cannot be recovered: an email recipient remembers the email; a credit-card charge moves money. The adapter's supports_rollback() is the honesty flag.

Staging means: record the effect in the journal as intent, do not execute it yet. The act is deferred to commit(). Mathematically, staged effects live in a partial order with respect to commit time — they are temporally posterior to it, by construction. Reversible effects, by contrast, are temporally prior to commit (they fired live, with a snapshot, ready to be undone).

Compensator. A second tool that semantically undoes the first. refund_charge compensates charge_card; the money came in, then went out; net effect (modulo bank fees) approximates the identity. Pherix does not verify the inverse relationship — the developer asserts it. Pherix records the pairing and provides the firing mechanism.

The gate. A predicate at commit time: every staged irreversible effect must be EITHER compensator-backed OR explicitly pre-approved. If neither holds, commit blocks. The gate is what makes "stage then commit" qualitatively different from "execute immediately."

TOCTOU. Time-Of-Check vs Time-Of-Use — a class of bugs where a property is checked, then acted upon, with intervening state change invalidating the check. The classic example: os.access(f); open(f) — a symlink swap between the two lines defeats the check. In policy terms: a spend cap looks fine at stage-time and is exceeded by commit-time. Two-phase checking closes that window.

§0

The two lanes the execution model has always had.

Slice 1's EffectStatus.STAGED, GATED, and TxnState.STAGED / PARTIAL / STUCK aren't new — they were defined as dormant. Slice 3 brings them online.

The execution model in CLAUDE.md §"Execution model" is bilateral: every tool either has a way back, or it doesn't. The reversible lane (Slice 1, 2) is live; the irreversible lane is dormant code waiting on the orchestration around it.

Reversible effect ≼ commit ·   (temporally prior — fired live, snapshot held, rollback restores)
Irreversible effect ≽ commit ·   (temporally posterior — staged, fires at commit, gate or compensator demanded)

Read that as a partial order. Commit is the cut. Reversible effects live on one side, irreversible on the other. Rollback before commit is in the "never instantiated" regime for the irreversible lane — which is the entire reason staging exists. The agent can ask for an email; if the transaction aborts before commit, no email was sent.

What "dormant since Slice 1" means in concrete code.

In pherix/core/effects.py, the EffectStatus enum already lists STAGED and GATED; the dataclass already has a compensator: str | None field. In pherix/core/transaction.py, the TxnState enum already lists STAGED, PARTIAL, and STUCK; the _ALLOWED_TRANSITIONS table currently only opens the reversible paths. Slice 3's job is to wire them up — not to add new types.

That's the project ethos working: the data model encodes the full execution model from day one; later slices fill in the orchestration. No retrofits.

D1

What does a staged tool call return to the agent?

The tool hasn't run yet. The agent needs something back — what shape, and what can it do with it?

In the reversible lane, the runtime returns the real result — the row id, the bytes that were written. Now the tool hasn't actually fired. The contract has to be honest about that, but it also can't break the agent's synchronous control flow.

Why (a) wins

A staged result is, semantically, a promise the runtime makes. The token is the artifact of that promise — it has a stable identity (effect_id) and no content. That is exactly the right ergonomic: the agent can branch on "have I asked for this yet?" but not on "what will it return?" — because asking the second question is the bug the staging model exists to prevent.

Mathematically, StagedResult is a token with one field (the identifier) and no semantic content. It's a name for a future point in the journal; it carries no information about what will live there. The agent that tries to do arithmetic on its value is asking a question outside the model.

The dataclass and the runtime branch. 
# core/effects.py — addition
@dataclass(frozen=True)
class StagedResult:
    effect_id: str
    _kind: str = "StagedResult"   # wire-tag for JSON consumers

# core/runtime.py — staging branch
if not adapter.supports_rollback():
    effect.status = EffectStatus.STAGED
    self.audit.update_effect(effect)
    return StagedResult(effect_id=effect.effect_id)
# ...else reversible path (unchanged)

Three lines. The contract — "if the resource can't be rolled back, the tool doesn't fire and you get a token" — is enforced at one site, not negotiated per tool.

D2

How does a tool declare its compensator?

A compensator is itself a tool. The question is how the original tool refers to it.

A compensator is the semantic left-inverse of an irreversible tool. The composition refund(charge(x)) approximates the identity in the resource's state-space — money out, money back. Pherix doesn't verify the inverse property (that's developer-asserted, often modulo fees and clock skew) but it does record the pairing.

For a tool f: Args → World, a compensator g: Args → World such that g ∘ f ≈ id.
Pherix records the pair (f, g); the developer asserts the approximation; the world is the rest of the universe outside Pherix.

What lands in the journal — the deciding axis

Why (a) wins

Every tool already has a string name — the registry is keyed by it. Compensators reusing that vocabulary means no new abstraction is introduced; the journal entry stays a plain JSON object; replay (Slice 5) can resolve compensator: "refund_charge" against a fresh registry at the moment of replay, not the moment of decoration.

A compensator is structurally another tool, so it gets the entire @tool apparatus for free: policy checks when it fires, journal entries when it runs, audit when it completes. (c) adds vocabulary without earning it; (b) is incompatible with the audit journal Pherix already has.

The runtime detail — what fires when, and what the journal shows.

When an irreversible effect with a compensator commits successfully, Pherix records the pairing — the compensator is registered as "available," not invoked. The actual firing of the compensator happens only on rollback after commit, or (D5) on partial-commit recovery. So:

# at commit:
charge_card  → APPLIED   # real HTTP call fires
                         # compensator "refund_charge" recorded but NOT invoked

# at rollback-after-commit (or partial-commit unwind):
refund_charge → APPLIED  # compensator fires as its own journalled effect
charge_card  → COMPENSATED   # original status updated

The compensator firing is itself a journal row. The pairing is visible. Audit reads as a story: "we charged, then we refunded, here are both rows."

D3

How does an irreversible effect with no compensator get approved?

The gate is the predicate at commit-time. The approver lives outside Pherix. The wiring between them is the design question.

Some irreversible tools genuinely have no compensator — there's no "un-send" for an email to noreply@regulator.gov. For those, the gate refuses to commit unless an explicit, per-effect approval has been recorded. Pherix's job is to record the verdict; the approver — a human pressing a button, a deterministic policy, an AI guardrail vendor — runs outside Pherix's walls. Pherix doesn't generate approvals; it accepts them.

Per-effect pre-approval treats approval as a first-class event in the journal, not a parameter to commit. D4

When is policy actually checked?

Slice 1 checks at stage-time. The data model promises a re-check at commit-time. Wire it now, even though the current policy is stateless.

Slice 1's allow-list policy is a pure predicate of the tool name: its verdict doesn't depend on world state. Re-checking it at commit-time gives the same answer as checking at stage-time. So why bother?

Because Slice 6 introduces state-dependent policy — spend caps, content rules, rate limits — and at that point the answers diverge. State changes between stage and commit. Building the second check now means it costs ~1 function call today, ~0 lines tomorrow.

The two options on the table

Rejected

(a) Stage-time only

What Slice 1 does today. Policy is evaluated once when the effect is added to the journal. If state changes before commit, the runtime acts on the stale check.

Failure mode: by Slice 6, a TOCTOU window opens for every staged effect. The longer the txn, the wider the window. Closing it later is a retrofit through every adapter — bad.

Picked

(b) Stage-time and commit-time

Every staged effect's policy.check() runs again at commit start. For Slice 1's stateless policy, the second call is a constant-time pure function returning the same verdict. The cost is negligible; the structural correctness is permanent.

Why: the hook is in the right place for Slice 6 with no retrofit. Policy becomes a function of (effect, world-state-at-commit) from day one.

Why two checks

Policy is, formally, a predicate P(effect, world_state). Slice 1's allow-list is the degenerate case P(effect) — no world_state dependence. As soon as Slice 6's spend-cap, content, or rate-limit rules arrive, P genuinely takes both arguments. A single check at S₀ only suffices when P is stage-invariant; in general it isn't.

The structural cost today is ~5 lines in runtime.commit(). The structural cost of retrofitting it later is "audit every adapter and every test that relied on the one-check invariant." Build the hook now.

The commit-time loop, with the new check. 
# core/runtime.py — commit()
def commit(self) -> None:
    self._guard_open()

    # D4 — re-evaluate policy on every staged effect at commit time.
    # For Slice 1's stateless policy this is a no-op; for Slice 6 it's load-bearing.
    for effect in self.txn.effects:
        if effect.status is EffectStatus.STAGED:
            self._policy.check(effect.tool, effect.args)   # raises on denial

    # ... then proceed to gate check, fire staged effects, transactional commit ...

The policy interface gains an optional args parameter (already part of Effect). Slice 1's allow_all() / allow-list ignore it; Slice 6's content rules will read it.

D5

What happens when commit fires effect 1, fires effect 2, and effect 3 fails?locked · option (b)

A 3-effect transaction enters commit. Two side effects land in the world. The third raises. Now what?

This is the genuine open call. The scenario is concrete: a transaction with three irreversible effects — charge_card (compensator refund_charge), send_email (no compensator, pre-approved), write_audit_record (compensator delete_audit_record). The first two land. The third fails with a 5xx. The transaction is now in TxnState.PARTIAL — a state Slice 1 defined as dormant.

Recommendation — (b)

Auto-fire compensators backwards over the fired effects. Saga semantics are what CLAUDE.md describes; (b) delivers them, closes the correctness gap, and produces a fully readable audit trail (charge, refund, both visible). Effects without compensators remain in the world and the txn ends in STUCK — Pherix is honest about what it cannot undo, which is exactly the project ethos.

(a) is faster to ship but leaves a real money-on-the-table gap between Slice 3 and Slice 5. (c) is the cleanest scope but punts work that CLAUDE.md explicitly puts in Slice 3 ("compensation registry"). (b) is the right amount of work to actually deliver the slice's promise.

The compensator-unwind loop, as it would look. 
# core/runtime.py — commit() — recommended (b) path
fired: list[Effect] = []
for effect in staged_effects:
    try:
        adapter.apply(effect, spec.fn)
        effect.status = EffectStatus.APPLIED
        fired.append(effect)
    except Exception:
        effect.status = EffectStatus.FAILED
        # unwind: fire compensators newest-first over already-fired effects
        for prior in reversed(fired):
            if prior.compensator is None:
                self.txn.transition(TxnState.STUCK)   # honest about what we can't undo
                raise PartialCommitStuck(prior.effect_id)
            comp_spec = REGISTRY.get(prior.compensator)
            adapter.apply(prior_comp_effect, comp_spec.fn)
            prior.status = EffectStatus.COMPENSATED
        self.txn.transition(TxnState.ROLLED_BACK)   # or PARTIAL if some failed
        raise

The loop walks the fired list backward, firing each compensator. When a compensator is missing, the txn enters STUCK and the operator gets a precise error naming the un-undoable effect. When every compensator succeeds, the txn ends in ROLLED_BACK — the same end state as a clean rollback.

where this is now TxnState.STUCK is already defined in pherix/core/transaction.py. The _ALLOWED_TRANSITIONS table needs three new edges (OPEN→PARTIAL, PARTIAL→ROLLED_BACK, PARTIAL→STUCK) and the commit loop needs the unwind block above. No new types; pure orchestration.
§

Everything still lives in the journal.

A staged irreversible effect's full life — stage, approval, fire, compensate — is six rows. Six readable rows.

The closing claim is the project's central claim restated under Slice 3's load: the engine doesn't grow. The journal grows. commit is still the fold-forward; rollback is still the fold-backward; partial-commit recovery (D5·b) is still just a fold-backward over the already-fired prefix.

∴

Why these four — plus D5 — are the slice.

Slice 3 isn't "add HTTP." It's "make irreversibility containable."

D1 fixes the agent contract: a staged tool returns a promise, not a result. D2 picks the cheapest, most replayable shape for compensator wiring: strings, in the registry, journalled like any other call. D3 keeps Pherix on the right side of its own scope: we record the approval verdict; somebody else generates it. D4 closes the TOCTOU door before it's ever opened: policy is a function of effect and world-state, even when the world-state argument is currently unused.

D5 is what decides whether Slice 3 ships with saga semantics or without. (b) is the recommendation: do the unwind loop now, while the engine is already being touched, and the project can credibly say "Pherix delivers the correctness story it claims." (a) and (c) are real options, but each leaves a gap the operator (you) will end up explaining to a user.

Either way: same engine, more rows in the journal. The irreversible lane lights up; nothing else changes shape.
