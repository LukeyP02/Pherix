Features

What Pherix gives you.

The guardrails Pherix puts around your agent's tool calls. Everything below is built and tested today.

These ship as a library plus a read-only inspector — not a SaaS dashboard or hosted service. You pip install it, wrap your tool-call layer, and self-host. Pre-release 0.0.0.

Control

policy

Allow and deny rules, plus caps — total and count limits — and rules that read live external state before deciding.

Checked at commit, and again when the effect is staged, so a bad call fails early rather than at the end of a run.

Containment

undo

Reversible effects roll back through real backend semantics — database savepoints, filesystem copy. The backend does the undo, exactly.

For effects with no exact undo, a registered opposite action runs instead — a refund for a charge, a deprovision for a provision.

Oversight

the gate

An irreversible effect with no safe undo is held back, not fired. It blocks at commit until someone approves it with approve_irreversible.

The human — or a higher-trust policy — is the only thing that lets it through.

Audit

the record

Every side-effecting tool call is recorded — the record is the audit log, no separate logging to trust.

Read it back with the inspector (python -m pherix.inspector), or re-run it to check it still does the same thing.

Isolation

concurrency

When two agents run at once, Pherix tracks what each one reads and writes and catches conflicts before they corrupt shared state.

Dry-run

speculative

Run against a snapshot, then throw it away. See exactly what a run would do before letting it touch anything real.

Governed memory

memory

The same guardrails pointed at the agent's memory — so what the agent remembers is contained and on the record too.
