The one mental model — everything is a traversal of the journal

A Transaction is an ordered, append-only list of Effects — the journal. A ResourceAdapter makes each entry executable and reversible against a real backend (SQL savepoint, filesystem copy, HTTP call). Every capability the roadmap delivers is then just a different way to walk that journal: fold forward, fold backward, fold against a snapshot, or diff it. The eight slices below are not eight subsystems — they are progressively richer traversals of one engine.

Where am I? — between Slice 1 and Slice 2

What's behind you Slice 1, merged

The reversible-path skeleton: Transaction as an ordered list of Effects, the ResourceAdapter protocol with one real implementation (SQLite via SAVEPOINT / ROLLBACK TO SAVEPOINT), @tool interception through a ContextVar, an agent_txn() context manager, a SQLite append-only audit journal, and a stage-time allow-list policy. The whole thing is offline-testable; 69 tests green.

What's in flight Slice 2, in worktree

FilesystemAdapter with copy-on-write. The point isn't filesystem-in-isolation — it's the protocol stress test: a structurally different backend (file copies, not savepoints) must satisfy the same snapshot → apply → restore interface, and the engine must route a mixed SQL+FS transaction with no special-case code. Three locked decisions: D1 add a TransactionalResourceAdapter sub-protocol, D2 rooted-and-lazy FsHandle, D3 the mixed-resource demo is the acceptance bar.

01 SQL adapter + engine

done · merged ad4e9c0 69 tests

The reversible path end-to-end. A SQLite SAVEPOINT is what makes rollback correct-by-construction.

Scope & what shipped

Why this slice first. A transaction system without a working reversible adapter is a logging library. SQLite gives us a backend whose semantics already match what we need — SAVEPOINT / ROLLBACK TO SAVEPOINT is a real, atomic snapshot/restore pair. The slice exists to prove the engine on the easy case before the hard ones.

Shipped.

  • Effect dataclass with read_keys / write_keys slots already present (so Slice 4 isolation isn't a retrofit).
  • Transaction + TxnState state machine — OPEN → STAGED → COMMITTED / ROLLED_BACK.
  • ResourceAdapter protocol; SQLiteAdapter implements snapshot/apply/restore.
  • @tool decorator + registry; agent_txn() context manager intercepts via a ContextVar.
  • SQLite append-only audit journal — transaction + effect records.
  • Minimal allow-list policy at stage-time.

Outstanding from review. Four follow-ups (see section below) — none block Slice 1's correctness; they gate later slices.

02 Filesystem adapter (copy-on-write)

active · in worktree ../Pherix-slice-2 D1 · D2 · D3

A structurally different backend that must satisfy the same protocol. Acceptance: a mixed SQL+FS transaction works with no engine special-cases.

Scope, decisions, done-when

The point. Slice 2 is the first real test of whether ResourceAdapter is a genuine abstraction or an SQLite-shaped hole. File copies and database savepoints share no implementation; if they share a protocol, the engine survives. If the runtime needs special-case code for FS, the protocol is the wrong shape — and that's the more interesting bug.

Locked decisions.

  • D1. Introduce TransactionalResourceAdapter(ResourceAdapter, Protocol) — a sub-protocol that adds begin/commit/rollback. The runtime switches from hasattr-sniffing to isinstance dispatch. Also closes Slice-1 follow-up #2.
  • D2. FilesystemAdapter(root: Path); tools take an FsHandle first arg (same shape as SQL's conn injection). Path traversal rejected (raises ValueError). Snapshots are lazy — first touch of a path inside an effect backs it up; subsequent touches in the same effect don't.
  • D3. The acceptance bar is mixed-resource: one agent_txn with {"sql": SQLiteAdapter, "fs": FilesystemAdapter}, an agent calling both kinds of tools, rollback wiping both, commit persisting both.

Out of scope. Staging, gating, compensators, isolation, replay, dry-run, MCP. The read_keys / write_keys slots stay inert.

Done when. Round-trip restore semantics verified (including the file-didn't-exist and file-was-deleted edge cases), path traversal rejected, backup tempdir cleaned up on every path, the D3 mixed-resource test passes, and the 69 Slice-1 tests stay green.

03 HTTP / irreversible adapter

queueddepends on FU#1

Effects that cannot be undone. Pherix's answer: stage them, only fire at commit, and gate or compensate.

Why this changes the model

Slices 1–2 work because the underlying resources support snapshot+restore. HTTP doesn't — you can't un-send a webhook. The runtime stops applying live and starts staging: the agent gets a StagedResult placeholder, the effect's real fire-time is moved to commit(). Two sub-cases:

  • Has compensator. Auto-commits at commit-time; on rollback, the compensator runs (e.g. refund(charge_id) for charge(amount)).
  • No compensator. Commit blocks on explicit approve_irreversible() — a gate, a human-in-the-loop hook.

Idempotency keys (the effect_id hash) become load-bearing here — replays of staged effects must not double-fire. This is why Slice-1 follow-up #1 (silent str() coercion of non-serializable args) must be resolved before Slice 3 starts.

04 Isolation

queued

MVCC-style: every effect declares what it read (with version) and what it wrote. Conflict detection is a diff over the journal.

Why the slots already exist on Effect

Two concurrent transactions can both look fine in isolation and corrupt state when interleaved. The classical fix is multi-version concurrency control: each effect records (resource, key, version) for reads and (resource, key) for writes; at commit, a conflict is just a non-empty intersection of this transaction's read-set with the write-sets of transactions that committed after it began.

Slice 1 already populates these slots with empty lists. That's deliberate — adding them later would touch every existing Effect and journal row. The shape is set; only the populating and the resolution policy (abort / retry / serialize) remain.

05 Replay from the journal

queueddepends on FU#4

Once the journal exists, replay is "fold forward against fresh state, assert identical results." Nearly free.

Why this is the cheap slice

The whole journal-as-canonical-truth design pays off here. Replay isn't a new subsystem — it's the existing commit path, run against a fresh database, with an equality assertion at each step. The only real work is making sure the journal is durable enough to replay from — which is why Slice-1 follow-up #4 (the audit journal defaults to :memory:) gates this slice. A non-durable journal cannot be replayed across a crash, so the default has to change before replay is meaningful.

06 Real policy engine

future

Capability grants, spend caps, content-aware rules — evaluated at stage-time AND again at commit-time (TOCTOU safety).

Why evaluate twice

Between stage-time and commit-time, state can change. A budget that was within cap when an effect staged can exceed cap by the time the transaction commits (because another effect in the same transaction has since spent). Single-shot policy is a time-of-check / time-of-use bug. Pherix evaluates at both edges — stage to fail fast and give the agent useful feedback; commit to actually enforce.

Slice 1 ships an allow-list as a placeholder. Slice 6 makes it real: declarative rules over the journal at commit-time.

07 Speculative dry-run

future

Fold the journal forward against a snapshot, then discard the snapshot. Same engine; different terminator.

What an agent gets out of this

Agents that can preview the consequences of their tool calls — full state diff, policy result, irreversible-effect list — before any of it is real. The implementation is the commit fold, but pointed at an ephemeral copy of state, with the diff returned as the result. Like Slice 5, this is "nearly free once the journal exists."

08 MCP gateway front-end

future

The same core engine, driven by an MCP proxy instead of a Python import. Validates the frontends/ discipline.

Why it's the last slice, not the first

Building Pherix as a library first forces the core to be front-end-agnostic. The MCP proxy is a different driver for the same engine — if the architectural discipline in core/ vs frontends/ has held, this slice is small (a process boundary, a wire format) and adds no new semantics. If it requires changes in core/, the discipline failed somewhere earlier and we'd rather find out now than at v1.0.

Slice-1 review follow-ups — four items, each gated to the slice it must be resolved before

Captured in docs/ROADMAP.md. None blocked Slice 1's correctness — they were intentional simplifications. Each is now attached to the slice that promotes the simplification from "fine for now" to "actively wrong."

1

Non-serializable args silently str()-coerced

In effects.py compute_effect_id and audit.py _json_default, args that aren't JSON-serialisable get stringified rather than rejected. Two consequences: distinct non-serializable objects can collide on the same effect_id, and the audit row is lossy. The plan called for raising; the code is silent. Decide: enforce-and-raise at Effect construction, or consciously accept the coercion and document it.

gates Slice 3
2

Adapter lifecycle is duck-typed, not in the protocol

runtime.py currently does hasattr(adapter, "begin") to decide whether to call begin/commit/rollback. The ResourceAdapter protocol in adapters/base.py only declares snapshot/apply/restore/supports_rollback. A typo in an adapter's method name silently no-ops. Resolved by D1 in Slice 2: introduce TransactionalResourceAdapter sub-protocol and dispatch with isinstance.

gates Slice 2 · D1
3

_guard_thread comment oversells coverage

The guard catches explicit sharing of the TxnContext across threads (good, tested). It does not catch the silent case — a tool dispatched to a worker thread where active_txn is empty, so the wrapper runs raw. Mitigation is reasonable, the comment just needs to be honest. Doc-only fix.

nit · anytime
4

Default AuditJournal() is :memory:

Fine for tests and the demo. Less fine as a real default — durability is one of Pherix's four stated ACID-style guarantees, and a non-durable default journal silently violates it. Decide: on-disk default (with explicit opt-in for in-memory), or keep in-memory and force callers to opt into durability. Either is defensible; the current implicit choice isn't.

gates Slice 5

Reading the map

The slices look like a ladder of features. They aren't. They're a ladder of journal traversals applied to progressively harder backends. Slice 1 proved the engine on a backend that already does what we need. Slice 2 proves the protocol holds when the backend doesn't share an implementation. Slices 3–4 add the two structural complications real systems have (irreversibility, concurrency). Slices 5–7 are nearly-free dividends from owning the journal. Slice 8 validates the front-end-agnostic discipline.

Every follow-up dot on the timeline is a Slice-1 simplification whose interest-rate becomes due at a specific later slice. Resolving them on time is how Pherix stays the shape it was designed to be.
